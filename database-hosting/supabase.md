# Supabase — security & free-tier guide

> ⭐ The primary backend for this project. Supabase is hosted Postgres + auth + storage + edge functions. On the **free tier**, your data-layer defenses are essentially **RLS + SSL + the anon-vs-secret key split** — there are no downloadable backups and no PITR, and inactive projects pause after ~7 days. Get RLS right and you are most of the way there. Tiers: 🟢 every app · 🔵 users/data · 🟠 at scale · 🔴 regulated.

---

Supabase is Postgres with an auto-generated REST/Realtime/Storage API in front of it, so its entire security model collapses onto one mechanism: Postgres Row Level Security (RLS). The single most damaging vibe-coded mistake — popularized by CVE-2025-48757, the missing-RLS flaw in AI codegen platform Lovable (NVD: "insufficient Row-Level Security policy ... allows remote unauthenticated attackers to read or write to arbitrary database tables of generated sites"; note the CVE is marked Disputed because the vendor argues customers own their app data) — is shipping the client-side publishable/anon key against tables that have RLS disabled or have no policies. Independent security research found 170+ Lovable-generated apps exposed this exact way (leaking emails, payment records, and API keys); that 170+ figure comes from the disclosure/research, not the CVE text itself. Because the anon key is *meant* to be public, anyone can open the network tab, grab it, and run arbitrary `select * from <table>` REST calls. RLS-on-every-exposed-table with deny-by-default policies is therefore non-negotiable, and you verify it with the built-in Security Advisor / database linter. The second class of disaster is leaking the service_role/secret key (it has BYPASSRLS and ignores all policies) into a client bundle, mobile app, or public repo. Beyond those two, harden Auth (email confirmation, MFA/TOTP, leaked-password protection, password strength, custom-claims RBAC), Storage buckets (private by default + folder-scoped RLS + per-bucket size/MIME limits + signed URLs), Edge Functions (keep verify_jwt on, keep secrets server-side), Realtime (private channels gated by RLS on realtime.messages), and the connection layer (enforce SSL, network restrictions, Supavisor pooling). FREE-TIER REALITY: 0.5 GB DB (read-only past 500 MB), 5 GB egress, 1 GB storage, max 2 active projects, 50 MB global file cap, projects PAUSE after ~1 week of inactivity, and — critically — NO automated backups and NO PITR. On free tier, backups are 100% your responsibility (your own `pg_dump`/`supabase db dump`). All security toggles below (RLS, MFA, SSL, network restrictions, Security Advisor, custom SMTP) are available on free tier; the notable security feature gated behind Pro is HaveIBeenPwned leaked-password protection.

### 🟢 Row Level Security (RLS) on EVERY exposed table — the #1 vibe-coded mistake

**What it is.** RLS is a Postgres feature that filters rows per-request based on the requester's identity; Supabase treats it as the primary authorization layer for its auto-generated REST/Realtime APIs. Enable it on every table in an API-exposed schema (default: public). Once enabled, access is DENY-BY-DEFAULT: with RLS on and zero policies, the anon/publishable key sees nothing (Supabase docs: 'no data will be accessible via the API using the publishable key until you create policies'). You then add explicit policies (effectively a WHERE clause appended to every query).

**Why it matters / why vibe-coded apps get it wrong.** This is THE failure that defines insecure vibe-coded Supabase apps. The anon/publishable key is embedded in the client and is meant to be public; if a table has RLS disabled (or enabled but no policy on a public table), anyone who extracts that key from the network tab can run `select * from users` over the REST API and dump the whole table — and also insert/update/delete. CVE-2025-48757 (missing RLS in Lovable) is exactly this class; security researchers found 170+ Lovable-generated apps leaking emails, API keys, and payment records this way. AI codegen tools focus on schema/function and routinely skip RLS because writing correct policies requires understanding the app's authz model. Common subtle bugs: enabling RLS but forgetting a policy for one operation (silent denial), or `auth.uid() = user_id` evaluating against NULL when logged out (always-false, but masks intent — add `auth.uid() IS NOT NULL`).

**How.** Enable per table: `alter table public.profiles enable row level security;` (Dashboard: Authentication > Policies, or Database > Tables shows an 'Unrestricted'/RLS-disabled warning label). Tables made in the Table Editor get RLS on automatically; tables from SQL Editor / CLI migrations / Lovable migrations do NOT — you must enable it explicitly (confirmed in RLS docs). Deny-by-default policy set scoped to the `authenticated` role:\n  SELECT: create policy "own rows" on public.profiles for select to authenticated using ( (select auth.uid()) = user_id );\n  INSERT: create policy "insert own" on public.profiles for insert to authenticated with check ( (select auth.uid()) = user_id );\n  UPDATE: create policy "update own" on public.profiles for update to authenticated using ((select auth.uid())=user_id) with check ((select auth.uid())=user_id);\n  DELETE: create policy "delete own" on public.profiles for delete to authenticated using ( (select auth.uid()) = user_id );\nPERFORMANCE: always wrap auth functions as `(select auth.uid())` not bare `auth.uid()` — the bare form re-evaluates per row; the subquery form is hoisted into an initPlan and cached per-statement (Supabase's own benchmark: 179ms→9ms, ~95% faster). The Splinter lint `auth_rls_initplan` (0003) flags the un-wrapped form. To auto-enable RLS on any future table regardless of creation path, use a Postgres event trigger calling `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. Note: enabling RLS does NOT block the service_role/secret key (it has BYPASSRLS).

**When it applies.** Every app — and specifically required before you ship anything using the public anon key. Free tier: fully supported, no gating.

**Sources:** <https://supabase.com/docs/guides/database/postgres/row-level-security> · <https://supabase.com/docs/guides/api/securing-your-api> · <https://supabase.com/docs/guides/database/database-advisors> · <https://nvd.nist.gov/vuln/detail/CVE-2025-48757> · <https://www.superblocks.com/blog/lovable-vulnerabilities>

### 🟢 API keys: publishable/anon (client-safe with RLS) vs service_role/secret (server-only, BYPASSRLS)

**What it is.** Supabase issues two privilege classes. The publishable key (new format `sb_publishable_...`, replacing the legacy `anon` JWT) is low-privilege, maps to the Postgres `anon` role (or `authenticated` once a user logs in), and is SAFE to ship in browsers/mobile/CLIs/source — but ONLY because RLS constrains it (docs explicitly list 'web page, mobile or desktop app, GitHub actions, CLIs, source code' as safe). The secret key (new format `sb_secret_...`, replacing `service_role`) maps to the `service_role` Postgres role which carries the BYPASSRLS attribute: it provides full access to project data, ignoring every RLS policy. It must live only in server-side code.

**Why it matters / why vibe-coded apps get it wrong.** Two distinct catastrophes. (1) Treating the publishable/anon key as a 'secret' — it isn't; security comes from RLS, not from hiding the key. (2) Leaking the service_role/secret key by importing it into client code, a mobile binary, a Next.js component that ships to the browser, or committing it to a public repo. Because it bypasses RLS entirely, a leaked secret key is total database compromise regardless of how good your policies are. Vibe-coded apps frequently paste service_role into frontend `.env` files (e.g. `NEXT_PUBLIC_...`, `VITE_...` prefixes ship to the browser) to 'make queries work,' silently disabling all security.

**How.** Find/manage keys in Dashboard: Settings > API Keys (or the project 'Connect' dialog). Migrate to the new model: Settings > API Keys > 'Publishable and secret API keys' tab (both old and new keys work simultaneously, so migration is non-breaking); legacy anon/service_role JWTs keep working but are slated for deletion in 'Late 2026, TBC' (new projects already ship without legacy keys as of Nov 2025). RULES: publishable key → client OK (with RLS on). Secret key → only in Edge Functions, your own backend, or secured admin tools; NEVER in a browser, even on localhost — Supabase additionally BLOCKS `sb_secret_` keys from browser origins, returning HTTP 401 if a REST/Realtime call carries one from a browser context; never in source/executables/public docs. Advantages of the new secret keys: instantly revocable, multiple keys, auditable, and because the format is fingerprintable, GitHub Secret Scanning detects leaks and Supabase AUTO-REVOKES any `sb_secret_...` found in a public GitHub repo and notifies the owner. Never use a framework public-env prefix for the secret key. Rotate via Settings > API Keys (legacy rotation: Settings > API > rotate anon/service/JWT secret).

**When it applies.** Every app. Free tier: full support including the new publishable/secret key model and GitHub leak auto-revocation.

**Sources:** <https://supabase.com/docs/guides/api/api-keys> · <https://supabase.com/docs/guides/getting-started/migrating-to-new-api-keys> · <https://supabase.com/changelog/29260-upcoming-changes-to-supabase-api-keys> · <https://supabase.com/blog/supabase-security-2025-retro>

### 🟢 Supabase Auth — email confirmation, MFA/TOTP, leaked-password protection, password strength, sessions

**What it is.** Built-in auth: email/password, magic links, OAuth (Google, GitHub, etc.), phone. Security controls include email confirmation (verify address before activation), MFA via TOTP authenticator apps (the TOTP MFA API is free and enabled on all projects by default), leaked-password protection (rejects passwords found in HaveIBeenPwned's Pwned Passwords corpus), configurable password strength/min-length, and JWT-based sessions with refresh-token rotation. Issues short-lived access-token JWTs whose claims (sub, role, aal, email...) feed directly into RLS via auth.uid()/auth.jwt().

**Why it matters / why vibe-coded apps get it wrong.** Defaults can be permissive for fast prototyping. If email confirmation is OFF, anyone can sign up under any email (account takeover / spam / impersonation). Without MFA, a phished password is full account access. Without leaked-password protection or strength rules, users reuse breached passwords. Long OTP/session lifetimes widen the stolen-token window. Vibe-coded apps usually accept defaults and never visit the Auth settings, and they often forget that auth.uid() returns NULL for anon requests — so RLS must explicitly require authentication.

**How.** Email confirmation: Authentication > Providers (or Sign In/Providers) > Email > enable 'Confirm email'. MFA/TOTP: free and enabled on all projects by default; build enrollment with supabase.auth.mfa.enroll/challenge/verify, and check status via getAuthenticatorAssuranceLevel(); enforce in RLS by gating on the `aal2` claim, e.g. `using ( (select auth.jwt()->>'aal') = 'aal2' )`. Leaked-password protection: Authentication > Passwords toggle — REQUIRES Pro plan or above (docs: 'available on the Pro Plan and above'). Password strength/min length: Authentication > Passwords — docs say 'anything less than 8 characters is not recommended'; set min length >=8 and require lowercase+uppercase+digits+symbols. OTP expiry: production checklist recommends 3600s (1h) or lower. Custom SMTP: Authentication > Emails/SMTP — recommended in prod so confirmation emails send from your domain and aren't rate-limited by Supabase's shared sender. Read claims in policies with auth.jwt(); session/refresh handled by the client SDK (httpOnly cookies via @supabase/ssr recommended for web).

**When it applies.** Every app for email confirmation, MFA, password strength (all free). Leaked-password (HaveIBeenPwned) protection: Once you have users / Regulated — Pro plan+ only. Custom SMTP: before any real launch (free tier supported).

**Sources:** <https://supabase.com/docs/guides/auth/password-security> · <https://supabase.com/docs/guides/auth/auth-mfa/totp> · <https://supabase.com/docs/guides/deployment/going-into-prod> · <https://supabase.com/docs/guides/auth/jwts>

### 🟢 Custom claims & RBAC via the Custom Access Token auth hook

**What it is.** For role-based access control beyond per-row ownership, attach a Custom Access Token Auth Hook — a Postgres function that runs before each JWT is issued and injects custom claims (e.g. user_role, permissions) into the token. RLS policies then read those claims to authorize, instead of joining a roles table on every query.

**Why it matters / why vibe-coded apps get it wrong.** Vibe-coded apps tend to store an `is_admin` boolean in a user-editable profiles row and check it client-side, or write RLS that an attacker can satisfy by updating their own role. Putting role in the JWT (set server-side at token issuance, not user-writable) is the correct pattern. Getting this wrong yields privilege escalation — a normal user flipping themselves to admin.

**How.** Create a `public.custom_access_token_hook(event jsonb) returns jsonb` function that looks up the user's role (from a roles table the user cannot modify via RLS) and adds it to the token claims. Register it in Dashboard: Authentication > Hooks > Custom Access Token > select the function. Critically (per docs): `grant execute on function public.custom_access_token_hook to supabase_auth_admin;` and `revoke execute ... from authenticated, anon, public;`. In RLS, read the claim via auth.jwt(), e.g. `using ( (auth.jwt()->>'user_role') = 'admin' )`, or build an `authorize(permission)` helper that does `(auth.jwt()->>'user_role')::public.app_role`. Remember the hook only changes the JWT, not the auth response object — decode the access_token client-side to read claims. NOTE: claims are baked at token issuance, so role changes take effect on the next token refresh.

**When it applies.** Once you have users with differentiated roles (admin/staff/member). Free tier: supported.

**Sources:** <https://supabase.com/docs/guides/database/postgres/custom-claims-and-role-based-access-control-rbac> · <https://supabase.com/docs/guides/auth/auth-hooks/custom-access-token-hook>

### 🟢 Storage security — private buckets, folder-scoped RLS, signed URLs, size/MIME limits

**What it is.** Supabase Storage objects live in the `storage.objects` table, so the SAME RLS model governs them. Buckets are PRIVATE by default. Private bucket: every operation (including download) checks RLS. Public bucket: read/serve bypasses access control (anyone with the URL gets the file), though upload/delete/move/copy still enforce RLS. Signed URLs grant time-limited access to private objects. You can cap file size and restrict MIME types per bucket.

**Why it matters / why vibe-coded apps get it wrong.** Marking a bucket 'public' for convenience exposes every file by URL forever — fine for avatars, a breach for invoices/IDs/private media. Conversely, leaving a private bucket with no upload policy means uploads silently fail. Without size/MIME limits, users upload huge files (egress/storage cost) or malicious content types. Vibe-coded apps often make one public bucket and dump everything in it.

**How.** Keep sensitive buckets private. Scope writes to a per-user folder so users can't overwrite each other:\n  create policy "users upload to own folder" on storage.objects for insert to authenticated with check ( bucket_id = 'avatars' and (storage.foldername(name))[1] = (select auth.jwt()->>'sub') );\n  create policy "users read own files" on storage.objects for select to authenticated using ( bucket_id='avatars' and (select auth.jwt()->>'sub') = owner_id );\nServe private files via short-lived signed URLs: `supabase.storage.from('avatars').createSignedUrl(path, 60)` (60s). Signed URLs use a dedicated internal signing key (separate from Auth JWT) and stay valid until expiry. Per-bucket limits at creation: `supabase.storage.createBucket('avatars', { public:false, fileSizeLimit:'1MB', allowedMimeTypes:['image/png','image/jpeg'] })` (or via SQL / Dashboard bucket settings). GLOBAL file-size cap: Storage > Settings — on FREE tier this cannot exceed 50 MB (Pro: up to 500 GB); per-bucket limits cannot exceed the global cap (both confirmed in file-limits docs).

**When it applies.** Once you accept user uploads. Free tier: 1 GB total storage, 50 MB global file cap; private buckets, RLS, signed URLs all supported.

**Sources:** <https://supabase.com/docs/guides/storage/security/access-control> · <https://supabase.com/docs/guides/storage/buckets/fundamentals> · <https://supabase.com/docs/guides/storage/uploads/file-limits> · <https://supabase.com/docs/guides/storage/buckets/creating-buckets>

### 🟢 Edge Functions — JWT verification, secrets, CORS, keeping service_role server-side

**What it is.** Deno-based serverless functions for logic that must NOT run in the client (Stripe calls, privileged DB writes, third-party API calls). They are the correct home for the secret/service_role key. `verify_jwt` defaults to true: the platform validates the caller's Auth JWT before your handler runs. Secrets are injected as environment variables.

**Why it matters / why vibe-coded apps get it wrong.** Edge Functions exist largely so you DON'T embed the secret key client-side — but only if you actually use them. Two mistakes: (1) disabling verify_jwt on a function that does privileged work, leaving it open to anonymous callers; (2) creating a service_role client but still applying it to data that should be user-scoped (it bypasses RLS, so a bug becomes a full data leak). CORS misconfig (wildcard `*` on credentialed endpoints) is also common.

**How.** Keep `verify_jwt = true` (the default) so the platform validates the JWT before your handler runs; configure per function in supabase/config.toml: `[functions.myfn]\nverify_jwt = true`. To RESPECT the caller's RLS, build the in-function client with the caller's Authorization header (publishable/anon key + `{ global: { headers: { Authorization: req.headers.get('Authorization')! } } }`) — queries then run as that user under RLS. Use the secret/service_role key ONLY for genuinely admin operations, and validate authorization yourself first. Set secrets: `supabase secrets set STRIPE_KEY=sk_...` (or Dashboard: Edge Functions > Secrets); read via `Deno.env.get('STRIPE_KEY')`. SUPABASE_URL/SUPABASE_ANON_KEY/SUPABASE_SERVICE_ROLE_KEY are auto-provided. For public webhooks (Stripe/GitHub) set `verify_jwt = false` and verify the provider's signature manually. Handle CORS by returning explicit Access-Control-Allow-Origin (your domain, not `*` for credentialed requests) and answering OPTIONS preflight.

**When it applies.** Once you have any server-side logic, payments, or privileged writes. Free tier: 500,000 invocations/month included.

**Sources:** <https://supabase.com/docs/guides/functions/auth> · <https://supabase.com/docs/guides/functions/secrets> · <https://supabase.com/docs/guides/functions/function-configuration> · <https://supabase.com/docs/guides/functions/cors>

### 🟢 Realtime authorization — private channels gated by RLS on realtime.messages

**What it is.** Realtime (Broadcast, Presence, Postgres Changes) also flows through RLS. For Broadcast/Presence you control channel access by writing RLS policies on the `realtime.messages` table; the client must opt into a private channel with `private: true`. For Postgres Changes, the change stream respects the source table's own RLS for each subscriber.

**Why it matters / why vibe-coded apps get it wrong.** By default a public channel doesn't check authorization — anyone with the anon key can subscribe/broadcast on a topic and snoop or spam other users' live data. Vibe-coded realtime chat/presence features routinely ship without private channels, leaking messages and presence across rooms. Postgres Changes will also broadcast rows a user shouldn't see if the underlying table's RLS is missing.

**How.** Client: open a private channel — `supabase.channel('room-1', { config: { private: true } })`. Then authorize via RLS on realtime.messages using the `realtime.topic()` helper and the `extension` column. Receive (must have a SELECT policy to join):\n  create policy "can receive broadcast" on realtime.messages for select to authenticated using ( exists (select 1 from rooms_users where user_id=(select auth.uid()) and room_topic=(select realtime.topic()) and realtime.messages.extension in ('broadcast')) );\nSend (INSERT policy):\n  create policy "can send broadcast" on realtime.messages for insert to authenticated with check ( exists (select 1 from rooms_users where user_id=(select auth.uid()) and room_topic=(select realtime.topic()) and realtime.messages.extension in ('broadcast')) );\nMirror the same pattern with extension `'presence'` for presence. For Postgres Changes, simply ensure the source table has correct RLS — subscribers only receive rows their policies allow.

**When it applies.** Once you ship realtime/collaborative features. Free tier: 200 concurrent connections, 2M messages/month included.

**Sources:** <https://supabase.com/docs/guides/realtime/authorization> · <https://supabase.com/blog/supabase-realtime-broadcast-and-presence-authorization> · <https://supabase.com/docs/guides/realtime/concepts>

### 🟢 Postgres roles & SECURITY DEFINER / function search_path cautions

**What it is.** Supabase maps API access to Postgres roles: `anon` (unauthenticated), `authenticated` (logged-in user), and `service_role` (BYPASSRLS). RPC functions exposed to the API run with these roles. A SECURITY DEFINER function runs with the privileges of its OWNER (often a superuser-like role), not the caller — which can intentionally or accidentally bypass RLS. A mutable `search_path` on such functions is an injection/privilege-escalation vector.

**Why it matters / why vibe-coded apps get it wrong.** A SECURITY DEFINER function carelessly exposed via RPC can let any anon caller perform privileged operations, sidestepping all your RLS work. If its search_path isn't pinned, an attacker who can create objects in a schema on the path can shadow a function/table the definer calls and hijack execution. AI-generated 'helper' RPCs frequently use SECURITY DEFINER to 'just make it work,' creating silent RLS bypass holes. SECURITY DEFINER views are similarly flagged because they run with the view owner's rights.

**How.** Prefer SECURITY INVOKER (the default) so functions respect the caller's RLS. Use SECURITY DEFINER only when you must elevate, and then: (1) pin the search path — `alter function public.myfn() set search_path = ''` and fully-qualify every object (public.table), or `set search_path = public, pg_temp`; (2) restrict who can call it — `revoke execute on function public.myfn() from anon, authenticated; grant execute on function public.myfn() to <specific role>;`; (3) keep its body minimal and validate inputs. The Security Advisor flags `security_definer_view` (0010) and `function_search_path_mutable` (0011). Don't expose internal SECURITY DEFINER helpers via PostgREST unless intended.

**When it applies.** Once you write RPC functions / custom SQL (which AI tools do early). Free tier: supported; advisor flags it.

**Sources:** <https://supabase.com/docs/guides/database/postgres/row-level-security> · <https://supabase.com/docs/guides/database/database-advisors> · <https://supabase.com/docs/guides/api/securing-your-api>

### 🔴 Network & connection security — enforce SSL, network/IP restrictions, DB password, Supavisor pooling

**What it is.** Direct Postgres access (psql, ORMs, migrations) bypasses the RLS-protected REST API and connects as a powerful Postgres role using the database password. Supabase lets you enforce SSL on all DB/pooler connections, restrict which IP ranges can connect (network restrictions), and provides Supavisor connection pooling in transaction (port 6543) or session (port 5432) mode.

**Why it matters / why vibe-coded apps get it wrong.** The database password (and the `postgres`/pooler connection string) grants direct, RLS-bypassing access — if it leaks (committed to a repo, pasted in a client app), it's full compromise. Without SSL enforcement, credentials/data can traverse unencrypted. Without network restrictions, the DB endpoint is reachable from anywhere. Serverless/edge apps that open a new direct connection per invocation also exhaust Postgres's connection limit — the right fix is transaction-mode pooling, not raising max_connections.

**How.** Enforce SSL: Database > Settings > SSL Configuration > enable 'Enforce SSL on incoming connections' (applies to Postgres, Supavisor, and PgBouncer); connect with `sslmode=require`, or for strongest assurance `sslmode=verify-full` plus the project CA cert (downloadable from the same SSL Configuration section — docs recommend verify-full when enforcement is on). Network restrictions: Database > Settings > Network Restrictions — allowlist only your server/CIDR; note if your DB resolves to IPv6 you must add BOTH IPv4 and IPv6 CIDRs (unless you have the IPv4 add-on / IPv6 migration extension). Keep the DB password out of client code and rotate it in Database > Settings if exposed. Connection strings live in the project 'Connect' dialog. Pooling: use TRANSACTION mode (port 6543) for serverless/Edge/Lambda (many short-lived clients share connections; no session-level features like prepared statements/LISTEN). Use SESSION mode (port 5432) for long-lived servers needing a dedicated connection. Total connections across both ports can't exceed the configured pool size.

**When it applies.** Once you connect directly (ORM/migrations) or deploy serverless. SSL enforcement & network restrictions: before real launch / Regulated. Free tier: SSL enforcement, network restrictions, and Supavisor pooling are all available.

**Sources:** <https://supabase.com/docs/guides/platform/ssl-enforcement> · <https://supabase.com/docs/guides/platform/network-restrictions> · <https://supabase.com/docs/guides/database/connecting-to-postgres> · <https://supabase.com/docs/guides/deployment/going-into-prod>

### 🟢 Security Advisor / database Linter + CLI migrations to catch RLS-disabled tables

**What it is.** Supabase ships a built-in Security Advisor (powered by the open-source Splinter Postgres linter) that scans for RLS-disabled tables in exposed schemas, SECURITY DEFINER views, mutable function search paths, exposed auth.users, RLS-enabled-but-no-policy tables, and more. It's available in the dashboard and as a CLI lint, and pairs with version-controlled migrations.

**Why it matters / why vibe-coded apps get it wrong.** This is the single fastest way to catch the catastrophic 'forgot RLS' mistake before it leaks data — it would have flagged the entire Lovable CVE class. Vibe-coded projects rarely run it. Without migrations in version control, manual dashboard changes drift and RLS gets dropped silently on rebuilds; AI tools that apply ad-hoc migrations frequently create tables WITHOUT RLS.

**How.** Dashboard: Advisors > Security Advisor — review and clear every error, especially `rls_disabled_in_public` (lint 0013). Other key lints: `security_definer_view` (0010), `function_search_path_mutable` (0011 → remediate with `alter function ... set search_path = ''`), `auth_users_exposed`, `rls_enabled_no_policy`, and the performance lint `auth_rls_initplan` (0003, the bare-auth.uid() trap). CLI: run `supabase db lint` (lints the local DB for schema/security issues; local stack must be running) and `supabase db lint --level error`. Manage schema as migrations: `supabase migration new add_rls`, write the `alter table ... enable row level security` + policies in the migration file, then `supabase db push` — so RLS is reproducible and reviewable. Re-run the advisor after every schema change and before launch (it's on the official production checklist). The Table Editor also shows a visible warning label on any RLS-disabled table.

**When it applies.** Every app — run it before launch and after each schema change. Free tier: Security Advisor and `supabase db lint` are fully available.

**Sources:** <https://supabase.com/docs/guides/database/database-advisors> · <https://supabase.com/blog/security-performance-advisor> · <https://supabase.com/docs/guides/local-development/cli/testing-and-linting> · <https://supabase.com/docs/guides/deployment/going-into-prod>

### 🔵 FREE TIER specifics — limits, project PAUSING, and NO automated backups (backups are YOUR job)

**What it is.** The Supabase Free plan (verified on the pricing page): 0.5 GB (500 MB) database size per project, 5 GB egress + 5 GB cached egress, 1 GB file storage, 50,000 monthly active auth users, max 2 ACTIVE projects, 50 MB global file-size cap, 500k Edge Function invocations, 200 Realtime concurrent connections / 2M messages. Two behaviors dominate operationally: projects PAUSE after ~1 week of inactivity, and there are NO automated backups and NO Point-in-Time Recovery.

**Why it matters / why vibe-coded apps get it wrong.** BACKUPS: On free tier, if you drop a table, get hit, or corrupt data, Supabase has nothing to restore — automated daily backups (with retention) and PITR are Pro-and-above features. Vibe-coded apps assume 'the cloud backs it up'; it does not on free. PAUSING: a side-project DB that goes quiet for a week is paused — the API stops responding until you manually restore it from the dashboard; this looks like an outage and can interrupt low-traffic production apps. SIZE/READ-ONLY: crossing the 500 MB limit flips the database to READ-ONLY (writes fail) until you free space or upgrade. EGRESS: a public bucket or leaky query can blow the 5 GB egress quota quickly.

**How.** BACK UP YOURSELF on free tier: schedule your own logical dumps — `supabase db dump -f backup.sql` (or `pg_dump` against the connection string), run on a cron from your own machine/CI, and store off-Supabase (e.g. git-crypt repo, S3, local). For data plus schema: `supabase db dump --data-only` plus a schema dump, or full `pg_dump`. AVOID PAUSING: keep the project active (a scheduled lightweight query/health-check at least weekly), or upgrade. Restore a paused project: Dashboard > project > 'Restore project' (can take minutes). MONITOR usage: Dashboard > Reports/Usage for DB size and egress; stay under 500 MB to avoid read-only mode (vacuum/cleanup if near). WHEN TO UPGRADE to Pro (from $25/mo per org): you need automated daily backups and/or the PITR add-on, no pausing, >500 MB DB, >2 projects, more egress, or leaked-password protection. Treat free tier as dev/prototype unless you implement your own backups and pause-prevention.

**When it applies.** Free tier (prototype/side-project). Move to Pro 'Once you have users/data' you can't afford to lose, since real backups + no-pausing are Pro-only.
