# Firebase — security guide

> Firestore / Realtime Database / Storage. The web API key is public by design; **Security Rules are your entire authorization layer** — never ship "test mode". Tiers as elsewhere.

---

Firebase pushes a fundamentally different security model than a traditional server: clients (browsers, mobile apps) talk DIRECTLY to Firestore/RTDB/Storage, so Security Rules are the ONLY thing standing between an attacker and your data — there is no server-side middleware to hide behind. Vibe-coded Firebase apps fail catastrophically in one predictable way: the project is created in "test mode" (allow read, write: if true) or with overly broad rules, shipped to production, and the public web config (apiKey, projectId) is scraped from client JS. Because the API key is NOT a secret and only identifies the project, anyone who finds it can read/write the whole database. This is not theoretical: in March 2024 GitGuardian researchers scanning ~5M sites found 916 with missing/misconfigured Firebase rules exposing 125M+ records and 19.8M+ plaintext credentials (emails, names, phone numbers, billing info, and passwords among them). The correct posture is: (1) deny-by-default rules with request.auth + ownership checks, (2) treat the API key as public but lock it down with rules, (3) add App Check attestation so only YOUR real apps can call the backend, (4) write granular Storage rules + least-privilege service accounts for Cloud Functions, and (5) validate data in rules, cap query reads to control cost/DoS, and test rules in the Emulator before deploy. App Check, rules, and Auth are complementary layers — none replaces the others.

### 🟢 Deny-by-default Security Rules with request.auth + ownership checks (never ship test mode)

**What it is.** Firebase Security Rules are the server-side authorization layer that gates every direct client read/write to Firestore, Realtime Database, and Storage. The two safe defaults are Locked mode (allow read, write: if false;) and per-user ownership rules that compare request.auth.uid to the data owner. 'Test mode' generates the dangerous open rule allow read, write: if true; (RTDB: ".read": true, ".write": true) — typically with a ~30-day expiry timestamp that most people forget about.

**Why it matters / why vibe-coded apps get it wrong.** On Firebase the client talks directly to the database; rules are the ONLY safeguard. Vibe-coded apps almost universally pick 'Start in test mode' in the console because it makes the app 'just work' during development, then ship it. As Google's own docs warn, anyone who guesses your project ID can then steal, modify, or delete all data with no auth. This exact pattern caused the 2024 GitGuardian-reported exposure of 125M+ records / 19.8M+ plaintext credentials across 916 sites. A subtler trap is 'any signed-in user' (if request.auth != null) — Firebase explicitly cautions that this lets any random authenticated user read everyone else's data; you also need an ownership/role check.

**How.** Start in Locked/Production mode, not test mode. Use rules_version = '2'. Firestore ownership pattern (store owner uid in a field): match /stories/{id} { allow read, delete: if request.auth.uid == resource.data.author_uid; allow create: if request.auth.uid == request.resource.data.author_uid; allow update: if request.auth.uid == resource.data.author_uid && request.auth.uid == request.resource.data.author_uid; }. Per-user collection by path: match /users/{userId}/{doc=**} { allow read, write: if request.auth.uid == userId; }. RTDB equivalent: "users": { "$uid": { ".read": "auth.uid === $uid", ".write": "auth.uid === $uid" } }. Roles via custom claims set with the Admin SDK: allow write: if request.auth.token.admin == true;. Deploy with: firebase deploy --only firestore:rules (or storage / database). NOTE: rule updates can take up to ~1 min to affect new queries and up to ~10 min to fully propagate to active listeners. Remember RTDB rules cascade — a parent ".read": true grants read on all deeper paths and CANNOT be revoked by a stricter child rule; keep rules granular and deny at the top.

**When it applies.** Every app — this is the single highest-impact control on Firebase; with direct client access, missing/open rules = full data breach.

### 🟢 The public web API key is not a secret — but it MUST be backed by locked-down rules

**What it is.** The firebaseConfig object in your client code (apiKey, authDomain, projectId, appId, etc.) is meant to be public. A Firebase API key only IDENTIFIES your project/app to Firebase services; it does not authorize access. Authorization comes from three separate mechanisms: Firebase Security Rules, Firebase App Check, and Google Cloud IAM.

**Why it matters / why vibe-coded apps get it wrong.** Vibe-coded developers panic about the apiKey being visible in client JS and waste effort trying to 'hide' it (env vars, obfuscation) while leaving the actual gate — Security Rules — wide open. Firebase's docs state plainly that the key being exposed is fine and expected; the DATA is exposed because the rules are open, not because the key leaked. The inverse mistake is just as bad: assuming that because the key 'looks like a secret' the project is somehow protected by it. It is not. (Genuinely sensitive credentials that the security checklist says must stay secret: service-account private keys for the Admin SDK and legacy FCM server keys — never ship those to clients.)

**How.** Do NOT treat the Firebase web apiKey as a secret; it's fine in client code and public repos. Instead: (1) lock down Security Rules (see item 1) so the key alone grants nothing; (2) enable App Check (see item 3); (3) in Google Cloud Console > APIs & Services > Credentials, apply Application restrictions (HTTP referrers for web, app signature for Android/iOS) and API restrictions to scope the key. Use SEPARATE, restricted API keys for non-Firebase Google APIs (Maps, Gemini) — those can incur cost if abused. Keep service-account JSON keys and legacy FCM server keys server-side only, in a secret manager, never in the client bundle.

**When it applies.** Every app — misunderstanding this wastes effort on the wrong thing and leaves the real gate (rules) open.

**Sources:** <https://firebase.google.com/support/guides/security-checklist> · <https://firebase.google.com/docs/projects/api-keys>

### 🔵 Firebase App Check (attestation) so only your real apps can reach the backend

**What it is.** App Check verifies that incoming requests come from your authentic, untampered app on a genuine device/browser, using a platform attestation provider: Play Integrity (Android), App Attest / DeviceCheck (iOS/iPadOS), and reCAPTCHA Enterprise (web). A short-lived attestation token is attached to every request; when enforcement is on, requests without a valid token are rejected — blocking scripts, scrapers, stolen-config bots, and emulated clients hammering your DB.

**Why it matters / why vibe-coded apps get it wrong.** Rules answer 'is this USER allowed to touch this data?' App Check answers 'is this request even coming from my real app?' Vibe-coded apps skip it, so even with correct rules an attacker can lift the public config and replay requests at scale from curl/scripts to scrape data or run up your bill. Firebase's docs describe App Check and Authentication as COMPLEMENTARY (Auth protects your users; App Check protects you, the developer) — it's not a replacement for rules or Auth: it doesn't do per-document authorization, and rules don't stop a botnet impersonating your app. You need both.

**How.** Firebase Console > Build > App Check. Register each app with its provider (web: reCAPTCHA Enterprise site key; Android: Play Integrity; iOS: App Attest). Initialize the App Check SDK in your client before calling Firebase. Watch the per-service request metrics (Firestore, RTDB, Storage, Cloud Functions) FIRST to confirm legitimate traffic carries valid tokens, then click Enforce per service. Enforcement can take up to ~15 min to take effect. For local dev/CI, register a debug token (FIREBASE_APPCHECK_DEBUG_TOKEN) so test clients aren't blocked. Token TTL is configurable from 30 min to 7 days (default ~1 hr on mobile, ~1 day on web; the SDK auto-refreshes at ~half the TTL) — shorter = stronger (smaller window for a stolen token) but more attestation latency and quota use.

**When it applies.** Once you have users/data (or any public-facing app) — protects against abuse, scraping, and bill-running attacks that rules alone can't stop. Effectively required before a real launch.

**Sources:** <https://firebase.google.com/docs/app-check> · <https://firebase.google.com/docs/app-check/enable-enforcement> · <https://firebase.google.com/support/guides/security-checklist>

### 🟢 Storage rules, Cloud Functions security, and least-privilege service accounts

**What it is.** Cloud Storage for Firebase has its OWN rules language (separate from Firestore rules) controlling uploads/downloads, where you can validate file owner, content type, and size from request.resource metadata. Cloud Functions run with a runtime service account; by default both 1st-gen and 2nd-gen (Cloud Run) functions use the Compute Engine / App Engine default service account, which carries the broad project Editor role.

**Why it matters / why vibe-coded apps get it wrong.** Two distinct blind spots. (1) Teams lock Firestore but forget Storage entirely (or leave it in test mode), so anyone can upload arbitrary files (malware, huge files inflating your bill) or download other users' private files. Storage rules are configured and deployed separately and are easy to overlook. (2) The default Functions service account having Editor means a single vulnerable/compromised function (e.g. one with an injection bug or that's publicly invokable) can read, modify, or delete almost anything in the project. Vibe-coded backends never narrow this, and also leave HTTP functions unauthenticated.

**How.** STORAGE RULES (storage.rules): scope by owner and validate metadata — match /users/{userId}/{file} { allow read: if request.auth.uid == userId; allow write: if request.auth.uid == userId && request.resource.size < 5 * 1024 * 1024 && request.resource.contentType.matches('image/.*'); }. Deploy: firebase deploy --only storage. FUNCTIONS / SERVICE ACCOUNTS: create a dedicated service account per function and bind only the minimal IAM roles it needs (e.g. roles/datastore.user for one DB, or a custom role with just storage.objects.get/create, not Editor): gcloud functions deploy myFn --service-account=myfn-sa@PROJECT.iam.gserviceaccount.com. Consider enforcing the iam.automaticIamGrantsForDefaultServiceAccounts org policy so new default accounts don't auto-receive Editor. Require auth on callable/HTTP functions (verify context.auth / the ID token; don't deploy public invokers unless intended). Enable App Check enforcement on Cloud Functions too. Keep secrets in Secret Manager / functions secrets, never hardcoded.

**When it applies.** Every app that stores files (Storage rules) and once you add server logic (Functions least-privilege). Per-function service accounts move from nice-to-have to important at scale / in regulated environments.

**Sources:** <https://firebase.google.com/docs/storage/security/rules-conditions> · <https://cloud.google.com/blog/products/application-development/least-privilege-for-cloud-functions-using-cloud-iam> · <https://firebase.google.com/docs/firestore/security/iam> · <https://cloud.google.com/functions/docs/securing/function-identity>

### 🟢 Data validation in rules, capping reads against cost/DoS, and testing with the Emulator

**What it is.** Security Rules can validate the SHAPE and CONTENT of writes (required fields, types, string length, allowed values) on request.resource.data, and can cap how many documents a single query may read via request.query.limit. The Firebase Local Emulator Suite plus @firebase/rules-unit-testing lets you unit-test rules locally (assertSucceeds / assertFails) without touching production.

**Why it matters / why vibe-coded apps get it wrong.** Three failure modes. (1) Validation: with direct client writes, a malicious client can write any field/type it wants unless rules validate it — corrupting data or injecting fields (e.g. setting their own role: 'admin'). (2) Cost/DoS: Firestore bills per document read (minimum one read per query, even for zero results); an unbounded query (collection.get() with no limit) lets one client read your entire collection, and an attacker can loop it to run a huge bill — a 'silent killer' of Firebase budgets. CRUCIAL gotcha: rules are NOT filters — a query is all-or-nothing, so you must constrain queries AND cap their size in rules. (3) Testing: people edit rules blindly in the console and deploy untested, shipping holes. The Emulator removes the excuse.

**How.** VALIDATE on write: allow create: if request.resource.data.keys().hasAll(['name','email','author_uid']) && request.resource.data.name is string && request.resource.data.name.size() <= 100 && request.resource.data.author_uid == request.auth.uid;. CAP READS so clients can't scan whole collections: match /stories/{id} { allow list: if request.query.limit <= 20 && resource.data.author == request.auth.uid; } — the client MUST then query with .where('author','==',uid).limit(20) or it's denied (rules are not filters: a query that could return docs the rules forbid is rejected entirely). COST GUARDRAILS: set a Cloud Billing budget + alerts (alerts notify but do NOT cap spend), enable App Check, and consider a rate-limiter for Functions. TEST locally: firebase emulators:start --only firestore (or emulators:exec '<test cmd>'); then with @firebase/rules-unit-testing (v9+ library) — initializeTestEnvironment({ projectId:'demo-x', firestore:{ rules: fs.readFileSync('firestore.rules','utf8') } }); const alice = testEnv.authenticatedContext('alice'); await assertSucceeds(getDoc(alice.firestore().doc('users/alice'))); await assertFails(getDoc(alice.firestore().doc('users/bob'))). Run it in CI before firebase deploy.

**When it applies.** Validation + emulator testing: every app. Read caps + billing budgets/alerts: once you have users/data (real traffic and a real bill to protect). Rate limiting and per-collection query hardening: at scale.
