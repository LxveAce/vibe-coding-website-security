## The one rule
A key that reaches the browser (Supabase anon/publishable, Firebase web config) is safe **only** when
row-level access control gates every row. The admin/`service_role`/secret key bypasses that and is
**server-only, never in a client bundle.** Everything below follows from *never trust the client,
never expose the DB.*

## Universal controls — apply to every database

Nine universal controls that apply to any backend database (Postgres, MySQL, MongoDB, Redis, DynamoDB) and to managed BaaS layers (Supabase, Firebase, PlanetScale, Neon, Turso, PocketBase, Appwrite). The through-line for AI-coded ("vibe-coded") apps: defaults are insecure and AI scaffolds reproduce them. The biggest real-world losses are not exotic exploits but configuration: databases bound to 0.0.0.0 with no auth (tens of thousands of open Redis and MongoDB instances mass-scanned, ransomed, wiped, and cryptomined), Supabase tables shipped with RLS off (so the public anon key reads everything), Firebase projects left in "test mode" (allow read, write: if true), service_role/secret keys committed to repos or shipped in client bundles, and string-concatenated SQL. Each item below gives what it is, why vibe-coded apps get it wrong, a concrete how-to with exact settings/SQL/CLI, and an honest tier. Supabase notes are free-tier-specific: on Free, RLS + SSL enforcement + the publishable/anon-vs-secret key split are your entire data-layer defense; backups are NOT downloadable, there is no PITR (Pro add-on, recommended once DB > 4 GB), and inactive projects pause after a low-activity 7-day period. Note current Supabase key guidance: legacy anon/service_role/JWT secrets can no longer be rotated in place — Supabase now directs you to migrate to publishable/secret keys (sb_publishable_/sb_secret_) plus asymmetric JWT signing keys, which support zero-downtime rotation. Claims below were verified against Supabase, Firebase, PostgreSQL, MongoDB, Redis, PgBouncer, Neon, OWASP, and pgAudit primary docs; every cited URL was fetched and resolves.

### 🟢 Never expose the DB to the public internet; never put credentials/connection strings in client code

**What it is.** The database listens only on a private network/localhost/VPC and is reachable only by your backend; clients (browser, mobile) talk to your API, never directly to the DB. Connection strings and DB passwords never appear in client-side bundles, mobile binaries, or git.

**Why it matters / why vibe-coded apps get it wrong.** This is the single most damaging and most common failure. Attackers mass-scan default ports (Postgres 5432, MySQL 3306, MongoDB 27017, Redis 6379) and connect to anything open. Tens of thousands of unauthenticated Redis servers have been found internet-exposed and abused for data theft, cryptomining, and DB wiping (a 2022 cryptojacking campaign alone targeted ~39,000 unauthenticated Redis; 2025 scans by Wiz found ~330,000 Redis exposed with ~60,000 having no auth). MongoDB has been mass-ransomed for years because it historically bound to all interfaces with auth off — the 2017 wave hit on the order of tens of thousands of databases, and recent 2025 scans found ~208,500 exposed MongoDB, ~3,100 reachable with no authentication, and ~46% of those already wiped and left with ransom notes. Vibe-coded apps get it wrong by binding to 0.0.0.0 to 'make it work', running the DB on a VM with a public IP and an open security group, or by hardcoding the full connection string (with password) into front-end code or a mobile app where anyone can extract it. With BaaS like Supabase/Firebase the DB is intentionally reachable from the browser via the anon/publishable key — which is fine ONLY because RLS/Security Rules gate every row (see items 4 and 5).

**How.** OWASP Database Security Cheat Sheet: 'Configuring the database to only bind on localhost', 'Restricting access to the network port to specific hosts with firewall rules', and 'Placing the database server on a dedicated internal network segment that is isolated from the application server.' Concretely: Postgres listen_addresses='localhost' or a private IP, pg_hba.conf restricting source CIDRs; MongoDB net.bindIp set to 127.0.0.1/private IP (never 0.0.0.0) plus access control enabled; Redis bind 127.0.0.1 + protected-mode yes + a password (item 2). Cloud: put RDS/Cloud SQL in a private subnet, no public IP, security group allowing only the app's SG/CIDR; use a bastion or SSM for admin. Credentials: OWASP says creds 'should never be stored in the application source code' and must not be 'checked into source code repositories' — keep them in env vars / a secret manager (item 6). Supabase: enable Database > Settings > Network Restrictions to allowlist IPs; the anon key in the browser is expected, but the service_role/secret key must never ship to a client (item 6). Verify nothing is exposed: from outside your network, `nmap -p 5432,3306,27017,6379 <host>` should show filtered/closed.

**When it applies.** Every app — this is non-negotiable on day one; an open, unauthenticated DB is compromised within hours of exposure.

### 🟢 Least-privilege database roles (app user is not owner/superuser; separate read/write/migration roles)

**What it is.** The credential your app connects with has only the privileges it needs — typically SELECT/INSERT/UPDATE/DELETE on its own schema — and is NOT a superuser, NOT the table owner, and cannot create roles, drop tables, or run migrations. Schema changes run under a separate, higher-privileged migration role used only by CI/CD.

**Why it matters / why vibe-coded apps get it wrong.** Least privilege caps the blast radius: if the app credential leaks or SQL injection lands, the attacker inherits exactly the app's rights. A superuser/owner credential means full data exfiltration, DROP TABLE, disabling RLS, and (in Postgres) BYPASSRLS. OWASP SQL Injection cheat sheet is blunt: 'DO NOT ASSIGN DBA OR ADMIN TYPE ACCESS TO YOUR APPLICATION ACCOUNTS.' Vibe-coded apps almost always connect as the postgres/root/admin superuser because that's the default credential from the local install or the managed provider's first user, and they never create a scoped app role. They also run migrations with the same all-powerful connection the live app uses.

**How.** Postgres role attributes (CREATE ROLE): a superuser 'bypasses all permission checks' and per the docs is 'a dangerous privilege and should not be used carelessly; it is best to do most of your work as a role that is not a superuser.' Create a login role with none of SUPERUSER/CREATEDB/CREATEROLE/BYPASSRLS. Example: `CREATE ROLE app_rw LOGIN PASSWORD '…' CONNECTION LIMIT 50;` then `REVOKE ALL ON SCHEMA public FROM PUBLIC;` and grant only what's needed: `GRANT USAGE ON SCHEMA app TO app_rw; GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_rw; ALTER DEFAULT PRIVILEGES IN SCHEMA app GRANT SELECT,INSERT,UPDATE,DELETE ON TABLES TO app_rw;`. Add a read-only role for reporting (`GRANT SELECT …` only) and a separate `app_migrate` owner role used solely by migration tooling. Always `REVOKE CREATE ON SCHEMA public FROM PUBLIC` to stop arbitrary object creation. MongoDB: create a user-administrator first, then per-app users with only needed roles (readWrite on one DB), following the 'principle of least privilege' — never the root role for the app. Supabase deliberately uses low-privilege `anon`/`authenticated` Postgres roles for API traffic and reserves `service_role` (which has the BYPASSRLS attribute) for trusted server code only — do not use service_role as your general app connection.

**When it applies.** Every app — costs minutes to set up; pairs directly with injection defense (item 4).

**Sources:** <https://www.postgresql.org/docs/current/role-attributes.html> · <https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html> · <https://www.mongodb.com/docs/manual/administration/security-checklist/> · <https://supabase.com/docs/guides/api/api-keys>

### 🟢 Encryption in transit (TLS/SSL required) and encryption at rest

**What it is.** All client-to-DB and intra-cluster traffic uses TLS so credentials and data can't be sniffed or MITM'd; data on disk (and backups) is encrypted at rest so stolen disks/snapshots are useless.

**Why it matters / why vibe-coded apps get it wrong.** Without TLS, anyone on the network path (shared cloud network, coffee-shop wifi, a compromised hop) can read the plaintext password handshake and query payloads. Many DBs accept BOTH encrypted and unencrypted connections by default, so 'it connects' doesn't mean it's encrypted — the app silently falls back to plaintext. At-rest encryption matters when a snapshot, backup file, or physical disk is exfiltrated. Vibe-coded apps typically use sslmode=disable/prefer (or no TLS verification) because a self-signed cert threw an error and disabling it 'fixed' it, leaving them open to MITM.

**How.** OWASP Database Security: 'Configure the database to only allow encrypted connections', install a trusted cert, and have apps 'connect using TLSv1.2+ with modern ciphers (e.g, AES-GCM or ChaCha20)' AND 'verify that the digital certificate is correct.' Postgres: set `ssl = on`, and in pg_hba.conf use `hostssl` (not `host`) so plaintext is rejected; clients use `sslmode=verify-full` with the CA cert (verify-full also checks hostname — verify-ca/require do not fully prevent MITM). MySQL: `require_secure_transport=ON`; create users `REQUIRE SSL`. MongoDB: configure net.tls for all mongod/mongos and app connections. Redis: enable TLS (`--tls-port`, cert/key/CA) and disable the plaintext port. At rest: MongoDB WiredTiger native encryption or dm-crypt/LUKS; Postgres/MySQL via cloud KMS-backed volume encryption (RDS/Cloud SQL encrypt storage + automated backups when enabled). Supabase free tier: turn on Database > Settings > SSL Configuration > 'Enforce SSL on incoming connections' so non-TLS connections are refused; storage is encrypted at rest by the platform. PlanetScale/Neon/Turso require TLS by default.

**When it applies.** Every app for in-transit TLS (especially any cloud/managed DB). At-rest encryption is Every app on managed providers (on by default) and Once you have users/data for self-hosted disks; mandatory for Regulated/enterprise.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Database_Security_Cheat_Sheet.html> · <https://www.mongodb.com/docs/manual/administration/security-checklist/> · <https://supabase.com/docs/guides/platform/going-into-prod> · <https://www.postgresql.org/docs/current/ssl-tcp.html>

### 🟢 Parameterized queries / ORM only — never string-concatenated SQL or NoSQL (injection)

**What it is.** User input is always passed as bound parameters, never concatenated into query text. The driver/ORM sends the query template and the values separately so input can never change query structure. Applies equally to NoSQL ($where, mapReduce, operator injection) and to dynamic SQL inside stored procedures.

**Why it matters / why vibe-coded apps get it wrong.** OWASP ranks parameterized queries as the #1 (Option 1) primary defense; with binding 'the database will always distinguish between code and data, regardless of what user input is supplied.' Vibe-coded apps get this wrong constantly: an LLM produces `"SELECT * FROM users WHERE email = '" + email + "'"` because it's the most obvious string, or builds dynamic `ORDER BY`/`WHERE` clauses from request params, or in Mongo passes a raw request body straight into a query allowing operator injection (e.g. `{password: {$ne: null}}`) or `$where` JS. Escaping is rated by OWASP as 'STRONGLY DISCOURAGED' and fragile compared to other defenses.

**How.** Use prepared statements / bound params everywhere. Vulnerable (Java): a `Statement` running `statement.executeQuery(query)` with the value concatenated in. Safe (Java): `PreparedStatement p = c.prepareStatement("… WHERE user_name = ?"); p.setString(1, name);`. Node pg: `client.query('SELECT * FROM users WHERE email=$1',[email])`. Python: `cur.execute('… WHERE id=%s',(id,))`. ORMs (Prisma, Drizzle, SQLAlchemy, ActiveRecord) parameterize automatically — but raw-SQL escape hatches (`$queryRawUnsafe`, `sequelize.literal`) reintroduce risk, so keep user data in bound params even there. Identifiers that can't be bound (table/column/ORDER BY) must be validated against an allow-list, not interpolated. Mongo: never spread req.body into a query; cast/validate types so `{$ne:…}`/`{$gt:…}` can't be injected, and disable server-side JS (`--noscripting`, which kills `$where`/`$function`/mapReduce). At the connection layer, PgBouncer recommends disabling the Simple Query protocol because, unlike the Extended Query protocol, 'Simple Query allows multiple queries in one packet, which allows some classes of SQL-injection attacks.' Supabase: prefer the PostgREST/SDK query builder (`.eq()`, `.filter()`) which parameterizes; if you write SQL RPC functions, use parameters, not string-built dynamic SQL.

**When it applies.** Every app — input validation is a useful secondary layer but parameterization is the primary, mandatory control.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Query_Parameterization_Cheat_Sheet.html> · <https://www.mongodb.com/docs/manual/administration/security-checklist/> · <https://www.pgbouncer.org/config.html>

### 🟢 Tenant isolation / row-level access control for multi-user data

**What it is.** Every row carries an owner/tenant identifier and the database (or rules engine) enforces that a request can only touch rows it owns — so being authenticated is not enough; you must also be authorized for each specific row. In Postgres this is Row Level Security (RLS); in Firebase, Security Rules; in DynamoDB, IAM leading-key conditions / scoped access.

**Why it matters / why vibe-coded apps get it wrong.** This is THE defining failure mode of vibe-coded BaaS apps. Supabase exposes your tables to the internet through PostgREST using the public anon key; if a table has RLS off (the default for new tables created via SQL) anyone with that key — which ships in your front-end bundle — can read and modify the entire table. Symptoms: a user sees another account's records on their dashboard. Firebase's equivalent: shipping `allow read, write: if true` (test mode) so the whole database is world-readable/writable. AI generators write the read/write code against `users/{uid}` but never publish a rules/RLS file, and the providers' quickstarts seed permissive defaults that get copy-pasted to production.

**How.** Supabase/Postgres: `alter table profiles enable row level security;` — once RLS is on, 'no data will be accessible via the API when using a publishable key, until you create policies' (fail-closed), then add scoped policies, e.g. `create policy "own rows" on profiles for select to authenticated using ((select auth.uid()) = user_id);` and a matching `with check` on INSERT/UPDATE so users can't write rows they don't own. Critical gotcha: the SQL Editor and the service_role key BYPASS RLS, so always test policies from the client SDK as a logged-in user, not from the dashboard. Performance/correctness: wrap `auth.uid()` in a subselect `(select auth.uid())` so Postgres caches it per-statement, index the tenant column (`create index on profiles(user_id)`), specify `to authenticated`, and store tenant/role claims in `raw_app_meta_data` (user-editable `raw_user_meta_data` must never hold authz data). Firebase: deny-by-default `match /{document=**}{ allow read, write: if false; }` then scope: `match /some_collection/{userId}/{document}{ allow read, write: if request.auth != null && request.auth.uid == userId; }`. Multi-tenant SaaS on plain Postgres without RLS: filter by tenant_id in every query via a mandatory data-access layer, or adopt RLS — never rely on hand-written WHERE clauses alone. Free-tier note: Supabase RLS is fully available on Free and is your primary data-layer defense there.

**When it applies.** Every app the moment more than one user's data lives in one DB/table — i.e. essentially all multi-user apps from day one.

**Sources:** <https://supabase.com/docs/guides/database/postgres/row-level-security> · <https://supabase.com/docs/guides/api/api-keys> · <https://firebase.google.com/docs/rules/basics> · <https://makerkit.dev/blog/tutorials/supabase-rls-best-practices>

### 🟢 Secrets management for DB credentials (env / secret manager, rotation) — not in the repo

**What it is.** DB passwords, connection strings, and privileged keys live in environment variables or a dedicated secret manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault, Doppler/Infisical), injected at runtime, never committed to git, and rotated on a schedule and immediately on suspected exposure.

**Why it matters / why vibe-coded apps get it wrong.** OWASP: credentials must not be 'checked into source code repositories.' Secret scanners continuously crawl public and even private GitHub for connection strings and provider keys; a committed Supabase service_role/secret key, Mongo URI, or AWS key is harvested within minutes. Vibe-coded apps hardcode the connection string in source, paste the service_role/secret key into client code, or commit `.env`. They also never rotate, so a key leaked once stays valid forever. Supabase is explicit: the anon/publishable key is public by design and safe in the browser, but the service_role/secret key uses the BYPASSRLS service_role and 'has full access to your project's data' — 'never expose your secret keys publicly,' and keep them only in server/Edge Function env, never a browser.

**How.** Add `.env`, `.env.local`, `*.pem` to `.gitignore` BEFORE the first commit; provide a committed `.env.example` with empty values. Load via env at runtime (`process.env.DATABASE_URL`). Better: store in a secret manager and fetch with the SDK so creds never sit in a file — AWS Secrets Manager can auto-rotate an RDS password on a schedule; Vault can issue dynamic, short-TTL per-request DB creds (e.g. 1h prod TTLs). Supabase free tier: keep the anon/publishable key in client env (fine), keep service_role/secret keys only in server-side env and Edge Function secrets (`supabase secrets set DB_URL=…`). Current Supabase key guidance has changed: the legacy anon/service_role/JWT secrets can no longer be rotated in place — instead migrate to the new publishable/secret keys (`sb_publishable_…` / `sb_secret_…`), which you can create and revoke individually (e.g. one secret key per backend so you can rotate them independently), and to asymmetric JWT signing keys, which support zero-downtime rotation (non-expired tokens signed by an old key keep working until you revoke it). Rotate at least annually and on any suspected leak. If a secret was ever committed, rotate/revoke it AND purge history (BFG/filter-repo) — deletion alone doesn't un-leak it. Add a pre-commit secret scanner (gitleaks/trufflehog) and enable provider push-protection.

**When it applies.** Every app for env-not-in-repo + .gitignore. Managed secret manager + automatic rotation + dynamic creds is Once you have users/data, and required for Regulated/enterprise.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Secrets_Management_Cheat_Sheet.html> · <https://supabase.com/docs/guides/api/api-keys> · <https://supabase.com/docs/guides/getting-started/migrating-to-new-api-keys> · <https://supabase.com/docs/guides/auth/signing-keys> · <https://supabase.com/docs/guides/functions/secrets>

### 🟢 Backups + point-in-time recovery + TESTED restores; migrations under version control

**What it is.** Automated, regular backups plus transaction-log-based PITR so you can restore to any moment; restores are periodically rehearsed (a backup is only real once you've restored from it); and all schema changes go through versioned, reviewed, forward-only migration files in the repo.

**Why it matters / why vibe-coded apps get it wrong.** Backups are the recovery path from the very attacks above (ransomware wiping a DB), from a bad migration, and from an accidental `DELETE`/`DROP`. The classic failure is discovering during an incident that backups were never enabled, are corrupt, or have never been restore-tested — best practice is to 'periodically build nonproduction databases from production backups … as a restore practice' to validate the integrity of the entire backup chain. Migrations off-version-control (changes typed straight into prod) make environments drift and incidents unreproducible. Vibe-coded apps lean on the provider's defaults without checking retention/PITR, and edit schema by hand in the dashboard.

**How.** Postgres: continuous archiving / WAL for PITR (pg_basebackup + archive_command, or provider PITR); test by restoring to a scratch instance and running app smoke tests; automate a monthly restore drill. MySQL: full backups + binlog for PITR. Keep migrations in a tool (Prisma Migrate, Drizzle, Flyway, Liquibase, Alembic, Rails) with files committed and run in CI by the dedicated migrate role (item 2) — never edit prod schema by hand. Supabase free tier specifics: daily backups exist but 'Database backups are not available for download for Free Plan projects' and there is NO PITR on Free — PITR is a paid add-on with second-level granularity, and Supabase recommends enabling it once you expect your DB to exceed 4 GB; for resilience also keep your own `pg_dump`/logical export on a schedule. Also note Free projects with low activity over a 7-day period may be paused (plan around it). GDPR caveat: backups taken before a user-deletion still contain that PII, so document a backup-retention/erasure procedure (item 9).

**When it applies.** Backups + migrations in version control: Every app. Tested restores + PITR: Once you have users/data (any data you can't afford to lose). Documented restore runbooks and RTO/RPO targets: Regulated/enterprise.

**Sources:** <https://supabase.com/docs/guides/platform/going-into-prod> · <https://www.isaca.org/resources/isaca-journal/past-issues/2012/database-backup-and-recovery-best-practices> · <https://www.red-gate.com/simple-talk/data-security-privacy-compliance/its-2026-why-are-databases-still-failing-gdpr-compliance-audits/>

### 🟢 Connection pooling, query timeouts, and resource caps to resist DoS

**What it is.** A connection pooler (PgBouncer, Supabase's Supavisor/pooler, ProxySQL, RDS Proxy) sits between app and DB to bound concurrent connections and reuse them; per-statement timeouts kill runaway queries; and per-user/per-DB connection caps and row/time limits prevent one client (or attacker) from exhausting the DB.

**Why it matters / why vibe-coded apps get it wrong.** Databases have a hard max connection ceiling; each connection costs memory. Serverless/edge functions and AI-generated code that opens a new DB connection per request (and forgets to close it) exhaust connections under modest load — a self-inflicted DoS — and also make the DB trivially exhaustible by an attacker. No statement timeout means one heavy/pathological query (or a deliberately expensive injected one) pins CPU and blocks everyone. Vibe-coded apps rarely set timeouts, pool sizes, or per-user caps.

**How.** Put a pooler in front of the DB. PgBouncer in transaction mode multiplexes many clients onto few server connections; set `max_client_conn`, `default_pool_size`, and `query_wait_timeout` (which is 'used to prevent unresponsive servers from grabbing up connections'); recent PgBouncer adds `max_user_client_connections` and `max_db_client_connections` to cap per-user/per-DB; and disabling the Simple Query protocol both hardens against an injection class and is good hygiene. Set timeouts at the DB: Postgres `statement_timeout` (e.g. 15s for web traffic), `idle_in_transaction_session_timeout`, and `LIMIT` on unbounded reads; PgBouncer `query_timeout` should be slightly below the server-side `statement_timeout`. Cap connections per role: `ALTER ROLE app_rw CONNECTION LIMIT 50;`. Supabase: use the connection pooler endpoint (transaction mode, port 6543) from serverless/edge code rather than the direct 5432 connection — the Free tier has a low direct-connection ceiling, so pooling is effectively required there. Neon/PlanetScale ship pooled endpoints (Neon's `-pooler` hostname runs PgBouncer in transaction mode); use them for serverless. Pair with app-layer rate limiting on API routes.

**When it applies.** Query timeouts + a pooler for any serverless/edge or multi-instance app: Once you have users/data (and effectively Every app on Supabase Free due to low connection limits). Per-user connection caps and fine-grained resource governance: At scale.

**Sources:** <https://www.pgbouncer.org/config.html> · <https://neon.com/docs/connect/connection-pooling> · <https://oneuptime.com/blog/post/2026-01-25-manage-connection-limits-postgresql/view> · <https://supabase.com/docs/guides/platform/going-into-prod>

### 🔴 Audit logging, monitoring & alerting on the data layer; PII classification, retention & deletion

**What it is.** Record who did what to which data (logins, privileged actions, schema changes, bulk reads/exports), centralize and alert on anomalies, classify which columns hold PII, and enforce retention/deletion so personal data isn't kept forever or recoverable after a deletion request.

**Why it matters / why vibe-coded apps get it wrong.** Without audit logs you can't detect or investigate a breach — you won't know if the leaked key was used or which rows were exfiltrated. Monitoring/alerting turns 'breached for 6 months unnoticed' into 'caught in minutes.' PII classification + retention is both a security control (less data = smaller blast radius) and a legal one (GDPR/CCPA right-to-erasure; databases routinely fail compliance audits because no one mapped where PII lives or set retention). Vibe-coded apps log nothing at the data layer, store PII in plaintext columns indefinitely, and have no deletion path — so a 'delete my account' request can't actually purge the data (and stale backups still hold it: once a backup taken before the deletion is restored, 'the deleted data reappears in the live system').

**How.** Enable DB-native auditing: Postgres pgAudit 'provides detailed session and/or object audit logging via the standard PostgreSQL logging facility,' capturing SELECT/INSERT/UPDATE/DELETE, DDL, and role activity for HIPAA/GDPR/SOX evidence (it records who touched which tables/when without copying the data); MongoDB Enterprise auditing for connection/auth/operation events; cloud DBs ship audit logs to CloudWatch/Cloud Logging. Centralize logs and alert on signals: spikes in rows returned/exported, failed-auth bursts, use of privileged/service_role credentials, and out-of-hours admin access. Supabase: review Logs/Reports, alert on auth anomalies, and remember service_role usage bypasses RLS so audit its use. Classify PII by scanning column names/types (email, phone, ssn, dob, address, iban) and tag sensitive columns (tools like pgcomply automate classification and drift detection in Postgres); apply column-level encryption/masking where warranted. Retention/deletion: set policies (e.g. delete consumer PII after account closure + 30–90 day grace; retain financial/transactional data 5–7 yrs for tax/audit), implement a real erasure path, and document a backup-retention procedure since backups taken before a deletion still contain the erased data in recoverable form. Keep monitoring on connection counts, slow queries, and error rates to catch the DoS conditions in item 8.

**When it applies.** Basic monitoring/alerting (error/connection/auth spikes): Once you have users/data. Full audit logging, PII classification, and documented retention/erasure: Regulated/enterprise (GDPR/HIPAA/SOC 2/PCI) — and prudent for any app storing real personal data.
