# Embedded/edge DBs & open-source BaaS

> SQLite/libSQL/Turso/Cloudflare D1 for small/edge apps, and open-source BaaS (PocketBase, Appwrite, Convex, Nhost) as Supabase alternatives — with how to choose.

---

For small or edge AI-coded apps, embedded/edge SQLite-family databases (SQLite, libSQL, Turso, Cloudflare D1) and open-source BaaS (PocketBase, Appwrite, Convex, Nhost) are attractive because they are cheap, simple, and self-hostable — but their security models differ fundamentally from Supabase's Postgres RLS, and vibe-coded apps repeatedly ship them with the gates wide open. The recurring failure mode is "default-allow": a SQLite file readable by every process on a shared host, a Turso/D1 token with full write scope embedded in a frontend, a PocketBase/Appwrite collection left with an empty (public) rule, a Firebase database left in test mode (allow read, write: if true), or a Supabase table created in raw SQL without RLS so the public anon key returns the whole table. The same class of bug caused the 2017 MongoDB ransom wave (port 27017 open, no auth), ongoing exposed-Redis compromises (binds broadly, historically no auth), and CVE-2025-48757 / the Lovable exposure where 303 endpoints across 170 vibe-coded Supabase apps (10.3% of 1,645 analyzed) leaked user data through missing RLS. Concrete guidance below covers file permissions and SQLCipher/libSQL encryption, edge auth-token scoping (read-only vs full), each BaaS access-rule model and self-host hardening, and how to choose between Supabase free tier and Postgres/Neon/PlanetScale/Firebase including migration and lock-in. Supabase specifics are free-tier-accurate as of 2026: RLS is available on free; it is now ENABLED BY DEFAULT on tables created via the dashboard Table Editor, but you must still enable it yourself on tables created in raw SQL / the SQL editor — the dangerous gap is the raw-SQL table, not the Table Editor table. Free projects are 500 MB DB / 1 GB storage / 5 GB egress/month, capped at 2 active projects, auto-pause after 7 days of inactivity, and have NO automated daily backups (only a single rolling ~7-day snapshot; daily backups start on the $25/mo Pro plan). A forward-looking change: new Supabase projects default to NOT auto-exposing new tables to the Data/GraphQL API from 2026-05-30 (opt-in available from 2026-04-28), and existing projects are migrated by 2026-10-30, after which a table needs an explicit Postgres GRANT before PostgREST/GraphQL can see it. The governing principle for every item: deny by default, scope every credential to least privilege, and never trust a client — enforce authorization server-side or in the database, never only in the frontend the AI generated.

### 🟢 SQLite file permissions and OS-level isolation

**What it is.** Plain SQLite (and the local file used by libSQL embedded replicas, D1 local dev, PocketBase, Appwrite) is just a file on disk with no network auth layer; access control is entirely the filesystem's job.

**Why it matters / why vibe-coded apps get it wrong.** Vibe-coded apps drop app.db in the project/web root or a world-readable container layer. On shared hosting, a Docker COPY of the whole repo, or a misconfigured static server, the .db file (and its -wal/-shm sidecars) gets served or read by other tenants/processes. There is no password to stop them — possessing the file IS full read/write access.

**How.** Store the DB OUTSIDE any served/public directory (never under web root or a Next.js /public) — the standard SQLite-security guidance is explicit that a web-app DB must not sit in a publicly downloadable directory. Set tight perms: `chmod 600 app.db` (owner-only) and a dedicated service user (`chown app:app`). Lock down the directory too (`chmod 700`). The -wal and -shm sidecars matter: with a typical umask, a 0600 DB yields 0600 sidecars, and any concurrent writer needs write permission on the *directory* (SQLite must create -wal/-shm there even for read-only opens) — keep the whole directory owned by the single service user. In Docker, put the file on a named volume, not in the image, and run as a non-root USER. Never commit *.db/*.sqlite to git — add to .gitignore. For genuine secrecy at rest, file perms are not enough; layer SQLCipher/libSQL encryption (next item).

**When it applies.** Every app — any app shipping a SQLite-family file must get file location and permissions right; it is the entire security boundary.

**Sources:** <https://cmsqlite.net/how-to-secure-your-sqlite-database-from-common-threats/> · <https://sqlite.org/forum/info/87824f1ed837cdbb>

### 🔵 Encryption at rest: SQLCipher and libSQL/Turso encryption

**What it is.** SQLCipher provides transparent full-database AES-256 encryption of the entire SQLite file (pages, metadata, tables, indexes, and journal/WAL files), with PBKDF2 key derivation. libSQL (Turso's fork) ships open-source encryption at rest built on SQLite3 MultipleCiphers; Turso Cloud and Cloudflare D1 encrypt at rest by default (AES-256-GCM).

**Why it matters / why vibe-coded apps get it wrong.** File permissions only stop same-host actors with the wrong user; they do nothing if a backup, volume snapshot, laptop, or container image is stolen/leaked. Vibe-coded local-first and edge apps often store sensitive data (tokens, PII) in an unencrypted file synced to devices. Encryption makes a leaked file useless without the key — but only if the key is not stored next to the file.

**How.** libSQL/Turso client: `createClient({ url: 'file:local.db', encryptionKey: process.env.ENCRYPTION_KEY })`; for an encrypted embedded replica add it alongside sync: `createClient({ url:'file:replica.db', syncUrl: env.TURSO_DATABASE_URL, authToken: env.TURSO_AUTH_TOKEN, encryptionKey: env.ENCRYPTION_KEY, syncInterval: 60 })`. Cipher options follow MultipleCiphers. You generate and manage the key — there is NO automatic key management; load it from env/KMS/secret manager, never hard-code it or commit it, and store it separately from the .db file. Raw SQLCipher: run `PRAGMA key='...'` as the first statement after opening, before any other operation; use a high-entropy key, not a short password (SQLCipher default is AES-256-CBC with PBKDF2). Turso Cloud and Cloudflare D1 are encrypted at rest automatically (AES-256-GCM) with provider-managed keys — no action needed, but that protects the provider's disks, not your local embedded replica, which you must encrypt yourself.

**When it applies.** Once you have users/data — mandatory once the file holds PII/secrets or syncs to client devices/laptops; optional for throwaway local caches.

**Sources:** <https://turso.tech/blog/fully-open-source-encryption-for-sqlite-b3858225> · <https://docs.turso.tech/features/embedded-replicas/introduction> · <https://oneuptime.com/blog/post/2026-02-02-sqlcipher-encryption/view> · <https://developers.cloudflare.com/d1/reference/data-security/>

### 🟢 Turso / libSQL edge auth tokens — scope to read-only and least privilege

**What it is.** Remote Turso/libSQL databases authenticate every connection with a JWT-style auth token over TLS; you mint tokens per database (or group), optionally read-only, optionally with an expiration. Embedded replicas sync using that same token.

**Why it matters / why vibe-coded apps get it wrong.** The single most common edge-DB mistake: shipping ONE full-access token to the browser/client or committing it to git, so anyone who reads the bundle can write/drop the whole DB. Edge functions and embedded replicas distribute the token widely, multiplying exposure. Note Turso tokens are database/group-scoped, not per-end-user — they do NOT give you row-level isolation, so you still need server-side authorization for multi-tenant data.

**How.** Mint scoped tokens with the CLI: `turso db tokens create <db> --read-only` for any client that only reads, and a separate full token only for trusted server code. Set short expirations: `turso db tokens create <db> --read-only --expiration 90d` (also accepts forms like 30d/1w/'never'). Rotate/revoke with `turso db tokens invalidate <db>` (invalidates all tokens for the DB). Store in env vars (TURSO_AUTH_TOKEN), never in client-shipped code; for browser/mobile, proxy DB access through your own server/edge function that holds the token rather than embedding it. Connections are TLS-encrypted by default. Treat the token like a password: one per environment, least privilege, rotate on suspected leak.

**When it applies.** Every app using Turso/libSQL remote or replicas — token scoping is the network access boundary.

**Sources:** <https://docs.turso.tech/cli/db/tokens/create> · <https://vibeappscanner.com/best-practices/turso> · <https://docs.turso.tech/features/embedded-replicas/introduction>

### 🟢 Cloudflare D1 — bindings, scoped API tokens, and proxy Workers

**What it is.** D1 is Cloudflare's serverless SQLite. Workers access it via a binding (env.DB) with no separate password; the HTTP/REST API and Wrangler use a Cloudflare API token. All data is encrypted at rest (AES-256-GCM) and in transit (TLS); keys are Cloudflare-managed.

**Why it matters / why vibe-coded apps get it wrong.** D1 has no row-level security or end-user auth of its own — a Worker bound to the DB can read/write everything, so ALL authorization must be enforced in your Worker code. Vibe-coded apps either expose a public Worker endpoint that runs arbitrary queries, or create an over-broad API token (account-wide D1:Edit) and leak it. Forgetting D1:Edit on a token silently breaks writes, prompting devs to over-grant.

**How.** For Worker access, use a binding in wrangler.toml (`[[d1_databases]]` with binding, database_name, database_id) — never put credentials in client code; the binding is server-side only. For REST/CI, create a scoped API token (My Profile > API Tokens > Custom token) with exactly the D1 permission needed: `D1:Read` for read-only, `D1:Edit` only where writes are required, scoped to the specific account; pass as `Authorization: Bearer <token>`. Build a proxy Worker that validates the user's session/JWT and authorizes each query BEFORE touching env.DB — D1 itself will not. Use parameterized statements (`db.prepare(sql).bind(...)`) to prevent SQL injection; never string-concatenate user input.

**When it applies.** Every app on D1 — binding hygiene and in-Worker authorization are non-optional since D1 has no built-in row security.

**Sources:** <https://developers.cloudflare.com/d1/reference/data-security/> · <https://developers.cloudflare.com/d1/tutorials/build-an-api-to-access-d1/> · <https://developers.cloudflare.com/fundamentals/api/get-started/create-token/>

### 🟢 PocketBase access rules (API rules) and self-host hardening

**What it is.** PocketBase is a single-binary Go BaaS over SQLite. Authorization is per-collection 'API rules': List, View, Create, Update, Delete (auth collections add a Manage rule). Each rule is a filter expression; null (locked) = superusers only and is the default, '' (empty) = anyone (superusers, authed users, and guests), non-empty = only requests matching the filter.

**Why it matters / why vibe-coded apps get it wrong.** The empty-string rule means PUBLIC — and the UI shows it as an unlocked, blank field that looks 'done'. Vibe-coded apps leave Create/Update/Delete empty (world-writable) or use a rule that returns all rows. The superuser dashboard is also reachable from the internet by default. Note: an unsatisfied List rule returns 200 with an empty list (acts as a filter), while Create returns 400 and View/Update/Delete return 404 — so a too-loose List rule silently leaks rather than erroring.

**How.** Restrict every collection to the owner: set List/View/Update/Delete rules to `@request.auth.id != '' && user = @request.auth.id` (where `user` is your owner relation field) and Create to `@request.auth.id != ''` plus a check that the new record's owner equals the caller. Never leave a write rule blank unless you truly want the world to write. Use `@request.auth.*` for the caller and `@request.body.*`/`@request.data.*` to validate incoming fields (PocketBase has no column-level rules, so guard sensitive fields here or in Go hooks). Hardening: put the admin/superuser UI (`/_/`) behind a reverse proxy with IP allowlist/VPN/extra auth or keep it off the public internet, serve only via HTTPS (TLS-terminating proxy like Caddy), create a strong unique superuser, validate input in hooks, and keep the binary updated (subscribe to GitHub releases). Test every rule from an anonymous and a different-user session.

**When it applies.** Every app on PocketBase — rules are the whole access model; an empty rule equals a public breach.

**Sources:** <https://pocketbase.io/docs/api-rules-and-filters/> · <https://pocketbase.io/docs/authentication/> · <https://www.valtikstudios.com/blog/pocketbase-self-hosted-security>

### 🟢 Appwrite permissions (collection + document/row-level security) and server-key danger

**What it is.** Appwrite is an open-source Firebase alternative. Access is role-based per collection/bucket and optionally per document/file. Roles: Role.any() (world), Role.guests(), Role.users(), Role.user(ID), Role.team(ID[,role]), Role.label(ID). Permission types: read, create, update, delete (write = create+update+delete).

**Why it matters / why vibe-coded apps get it wrong.** Default collection permissions often ship wider than intended (e.g. Role.any() read). Crucially, enabling per-document security still requires you to SET permissions on each document at create time or it is inaccessible/over-shared. And a Server API key with appropriate scopes BYPASSES all resource permissions entirely — embedding a server key in frontend code (a very common vibe-coded mistake) hands an attacker access to any resource regardless of any rule.

**How.** Set least-privilege collection permissions, e.g. `Permission.read(Role.user(userId))`, `Permission.update(Role.team('owners','owner'))`; avoid Role.any() unless the data is truly public. Turn ON 'Document/Row security' for per-record ownership and assign per-document permissions at create time (e.g. read/update/delete = Role.user(ownerId)). Use the WEB/client SDK with project-scoped client access in the browser; keep the SERVER API key only on your backend, scoped to the minimum required scopes, never committed or shipped — with the right scopes it ignores all resource permissions. Self-host hardening: front Appwrite with a reverse proxy (Caddy/Nginx/Traefik) terminating TLS via Let's Encrypt (it ships HTTP by default), patch the OS/Docker, firewall internal services, and lock the console behind allowlists.

**When it applies.** Every app on Appwrite — default-wide permissions and a leaked server key are the two classic breaches.

**Sources:** <https://appwrite.io/docs/advanced/platform/permissions> · <https://appwrite.io/docs/products/auth/security> · <https://betterstack.com/community/guides/databases/appwrite-self-hosted/>

### 🟢 Convex and Nhost authorization models (code-based and Hasura/Postgres)

**What it is.** Convex is a TypeScript reactive backend with no RLS — every public query/mutation/action runs your code and receives ctx.auth, so you write the authorization check at the top of each function. Nhost is an open-source Firebase alternative bundling Postgres + Hasura GraphQL + Auth + Storage + Functions; authorization is Hasura's role/permission rules keyed off JWT session variables (x-hasura-user-id).

**Why it matters / why vibe-coded apps get it wrong.** Convex's flexibility is a double-edged sword: if a function forgets the auth/ownership check, that endpoint is wide open — there is no database backstop. Nhost/Hasura apps fail when devs add an over-broad 'select with no row filter' rule that returns all rows (Hasura's default of no-permission is deny, which is safe), or expose the admin secret to the client.

**How.** Convex: start every public function by deriving identity from the request, e.g. `const identity = await ctx.auth.getUserIdentity(); if (!identity) throw new Error('unauthorized');` (or the `getAuthUserId(ctx)` helper shown in the docs), then verify the user owns the target record before reading/writing; centralize repeated checks in helpers. Never trust client-passed user IDs — derive identity from ctx.auth. Self-hosting Convex uses a generated admin key to authorize deploys/dashboard — keep it secret. Nhost/Hasura: configure per-role (user/anonymous) permissions on each table with a row filter like `{ user_id: { _eq: 'X-Hasura-User-Id' } }` and column allowlists; never expose HASURA_GRAPHQL_ADMIN_SECRET to clients (it bypasses all permissions, like a service key). Self-host the docker-compose stack behind TLS and patched infra.

**When it applies.** Every app on Convex/Nhost — Convex needs a per-function check; Nhost needs per-table Hasura rules and a protected admin secret.

**Sources:** <https://docs.convex.dev/auth> · <https://stack.convex.dev/authentication-best-practices-convex-clerk-and-nextjs> · <https://docs.nhost.io/products/graphql/permissions> · <https://github.com/nhost/nhost>

### 🟢 Supabase Row Level Security — enable it on EVERY exposed table (free-tier specific)

**What it is.** Supabase exposes Postgres directly to the browser via PostgREST using the public anon key; the only thing protecting your data is Postgres Row Level Security. RLS is available on the free tier. It is ENABLED BY DEFAULT on tables you create in the dashboard Table Editor, but you must enable it yourself on tables created in raw SQL or the SQL editor.

**Why it matters / why vibe-coded apps get it wrong.** This is the #1 vibe-coded backend breach. A table without RLS is fully readable/writable by anyone with your project URL + anon key (which ships in client code). The high-risk path today is the raw-SQL table (Table Editor tables now default to RLS-on), and AI codegen frequently emits raw `create table` SQL with no `enable row level security`. This pattern caused CVE-2025-48757 / the 2025 Lovable exposure — 303 endpoints across 170 apps (10.3% of 1,645 analyzed) leaked emails, messages, and private records via the anon key. The anon key is safe to publish ONLY when RLS is on every exposed table.

**How.** For each table created in SQL: `alter table profiles enable row level security;` (and verify the project-level 'Enable RLS on new tables' setting). Add owner policies, e.g. select: `create policy "own rows" on profiles for select to authenticated using ((select auth.uid()) = user_id);` and insert: `... for insert to authenticated with check ((select auth.uid()) = user_id);`; repeat update with both using + with check. Always wrap `auth.uid()` as `(select auth.uid())` (caches per-statement — the docs benchmark this as dramatically faster than per-row evaluation) and scope policies with `to authenticated`/`to anon`. Keep the service_role key server-side ONLY — it is a privileged key that bypasses RLS; never ship it to clients. Forward-looking: new projects (default from 2026-05-30) and existing projects (migrated by 2026-10-30) no longer auto-expose new tables to the Data/GraphQL API — add an explicit Postgres `grant` (or `alter default privileges`) for the anon/authenticated/service_role roles to expose a table (error 42501 = missing grant). Consider a dedicated `api` schema and disable the Data API if unused. Test every policy as anon and as another user.

**When it applies.** Every app on Supabase — RLS is the entire data boundary; a single un-RLS'd exposed table is a public breach.

**Sources:** <https://supabase.com/docs/guides/database/postgres/row-level-security> · <https://supabase.com/docs/guides/api/securing-your-api> · <https://vibeappscanner.com/supabase-row-level-security>

### 🟢 Choosing a backend: Supabase free tier vs Postgres/Neon/PlanetScale/Firebase

**What it is.** When the Supabase free tier fits vs when to graduate. Free Supabase: 500 MB DB, 1 GB storage, 5 GB egress/month, max 2 active projects, auto-pause after 7 days with no API requests, and NO automated daily backups (only a rolling ~7-day point-in-time snapshot; daily backups begin on Pro). Alternatives: Neon (serverless Postgres, branching, scale-to-zero), PlanetScale (MySQL/Postgres with Git-style branching + deploy requests), Firebase (NoSQL Firestore + Security Rules).

**Why it matters / why vibe-coded apps get it wrong.** Vibe-coded apps either outgrow the free tier silently (hit egress/DB caps, or a 'down' app is just paused) or pick a backend whose security model they don't understand — and they often assume their data is being backed up when the free tier has no automated daily backup. Firebase's test mode (allow read, write: if true) is the Firebase equivalent of missing RLS and is the most common Firebase breach.

**How.** Use Supabase free for prototypes, internal tools, and low-traffic side projects that tolerate cold starts — but never for production needing 24/7 uptime (the 7-day pause will take you offline), data you can't afford to lose (no automated daily backups on free — take your own pg_dump), or >500 MB/>5 GB egress; upgrade to Pro ($25/mo) or move. Pick Neon when you want serverless Postgres with cheap branches and scale-to-zero; enable SSL `verify-full` (Neon's strictest, supported mode), use IP Allow (Scale plan) and Protected Branches (auto-rotate role passwords on child branches, undeletable) for prod. Pick PlanetScale when you want safe non-blocking schema changes: branch -> deploy request -> merge, with branch-scoped passwords (a password cannot reach another branch) and per-password IP restrictions (PlanetScale stores only hashes, never plaintext passwords). Pick Firebase for realtime/offline-first mobile and rapid NoSQL prototyping — but you MUST replace test-mode rules before launch: use owner rules like `allow read, write: if request.auth.uid == resource.data.author_uid;` (and check request.resource.data on create), avoid the over-broad `auth != null` (which lets ANY logged-in user read everything), and enable App Check to block non-app traffic.

**When it applies.** Every app at the design stage — choosing wrong forces a painful migration later; security model and backup posture differ per choice.

**Sources:** <https://www.itpathsolutions.com/supabase-free-tier-limits> · <https://neon.com/docs/security/security-overview> · <https://planetscale.com/docs/concepts/connection-strings> · <https://firebase.google.com/docs/rules/insecure-rules>

### 🔴 Migration and vendor lock-in: Postgres portability vs proprietary stores

**What it is.** Lock-in risk varies enormously by store. Anything built on standard Postgres (Supabase, Neon, self-hosted) is portable via pg_dump/restore and standard SQL; proprietary NoSQL (Firestore) and provider-specific edge stores (D1's binding model) carry heavier rewrite cost.

**Why it matters / why vibe-coded apps get it wrong.** Vibe-coded apps lean hard on one provider's proprietary features (Firestore Security Rules + query model, D1 bindings, Convex functions). If you must leave (cost, outage, acquisition), Firestore migration is typically a multi-month project because schema design, access rules, and app code must all be rewritten; Postgres-based stacks move much faster.

**How.** Prefer standard-Postgres backends when lock-in matters: your schema, functions, triggers, and RLS policies are portable; back up with the `pg_dump` command Supabase shows in the dashboard and restore on RDS/Neon/Railway/your own server (or use logical replication for near-zero downtime on Postgres 10+). Keep migrations in version-controlled SQL, not just dashboard clicks, so they replay anywhere. For D1/Turso, keep DDL/queries in standard SQLite SQL and avoid provider-only extensions so you can dump and reload elsewhere. For Firestore, isolate data-access behind a repository layer so a future move doesn't touch every component, and export via the managed export tooling. Budget the migration cost into the original backend choice — portability is a security/continuity property, not just convenience.

**When it applies.** At scale / when continuity matters — early prototypes can ignore this; production and regulated apps should weigh exit cost up front.

**Sources:** <https://supabase.com/docs/guides/platform/migrating-to-supabase/postgres> · <https://www.thestackcanary.com/supasafe-supabase/> · <https://dev.to/pockit_tools/supabase-vs-firebase-in-2026-the-honest-comparison-after-using-both-in-production-3e5>

### 🟢 Real-world cautionary failures: open MongoDB/Redis and missing-RLS exposures

**What it is.** The canonical default-allow disasters that prove why the items above matter: the 2017 MongoDB ransom wave, ongoing exposed-Redis compromises, and the 2025 Supabase/Lovable missing-RLS leak (CVE-2025-48757).

**Why it matters / why vibe-coded apps get it wrong.** These are not theoretical. In Jan 2017 automated scripts scanned port 27017, connected to MongoDB instances with no auth, copied and deleted data, and left ransom notes — tens of thousands hit in days. Redis is designed for trusted networks: it historically shipped with no auth and is not meant to be exposed to the internet; open instances lead to data theft, RCE, and ransom. The 2025 Supabase exposure leaked private user data from 170+ vibe-coded apps (303 endpoints, 10.3% of 1,645 analyzed) via missing RLS + the public anon key. The common thread: a datastore reachable by default with no/weak auth.

**How.** MongoDB: never run without authentication (`security.authorization: enabled`); do not bind to 0.0.0.0 unless required — bind only to the interfaces your apps use (e.g. 127.0.0.1 plus a private IP), create least-privilege users, firewall port 27017 off the public internet, and require TLS for client and intra-cluster traffic. Redis: bind to a specific/loopback interface rather than all interfaces, keep `protected-mode yes` (it restricts to loopback when unconfigured and unauthenticated), require a strong password via ACL users (preferred) or `requirepass` with minimal command sets, never expose 6379 to the internet, and restrict/rename dangerous commands for untrusted clients. General rule that prevents all three: deny by default, require authentication on every datastore, bind to the smallest network surface, and verify from an external/unauthenticated client that you get rejected — then re-check after every AI-generated change.

**When it applies.** Every app — these failures recur constantly; the mitigation (auth on, bound tight, deny-by-default) is universal and cheap.

**Sources:** <https://redis.io/docs/latest/operate/oss_and_stack/management/security/> · <https://oneuptime.com/blog/post/2026-03-31-mongodb-harden-security-threats/view> · <https://vibeappscanner.com/supabase-row-level-security>
