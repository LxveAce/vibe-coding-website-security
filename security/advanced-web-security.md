## Transport, TLS, DNS & network security

This layer covers the cryptographic and naming/routing substrate that AI-coded apps almost never configure deliberately: they inherit whatever TLS their PaaS or reverse proxy ships, never touch DNS beyond an A/CNAME record, and assume the hosting provider "handles DDoS." The deep gaps here are: (1) Modern TLS hardening - TLS 1.3 + a curated TLS 1.2 cipher list, OCSP stapling (where still served), and disabling legacy protocols, benchmarked against the TLS configuration generator that was, until May 2026, Mozilla's SSL Config Generator and is now the community-run TLSRef Configurator (configurator.tlsref.org), plus Qualys SSL Labs. A major 2024-2025 shift: Let's Encrypt deprecated OCSP entirely (responders shut down Aug 6 2025) in favor of CRLs and short-lived certs, which changes revocation guidance. (2) HSTS with preload, plus the now-mandatory Certificate Transparency ecosystem and the deprecated Expect-CT header (deprecated, not removed, from Chromium 107; CT is now enforced by default so the header is obsolete). (3) DNS-layer security - DNSSEC chain of trust, CAA records (RFC 8659) with account/method binding (RFC 8657), registrar/registry lock, and NS hygiene (AXFR lockdown). CA/Browser Forum ballot SC-085v2 (passed June 2025, effective March 15 2026) now requires CAs to validate DNSSEC when present during CAA/DCV lookups, raising the value of DNSSEC. (4) Subdomain takeover via dangling CNAMEs - a classic that AI scaffolding makes worse because it spins up and tears down cloud resources and SaaS endpoints without DNS lifecycle hygiene; covered by the OWASP Subdomain Takeover Prevention Cheat Sheet with service fingerprints and a strict decommissioning order. (5) Edge defense - L3/4 and L7 DDoS protection, WAF managed rulesets, rate limiting, and bot management (Cloudflare/AWS/Akamai), including origin-cloaking patterns (Authenticated Origin Pulls / mTLS) that AI-coded apps leave wide open by exposing the raw origin IP. Every item below names exact directives, headers, RFCs, and tooling, with an honest tier for when it actually matters. Note on currency: the Mozilla server-side TLS tooling and guidelines were migrated to the community TLSRef project in May 2026; old ssl-config.mozilla.org and wiki.mozilla.org URLs now redirect, and this section uses the live TLSRef sources.

### 🟢 Modern TLS configuration: TLS 1.3 + curated TLS 1.2, disable legacy, OCSP stapling

**What it is.** A deliberate server TLS profile: enable TLS 1.3 and (for compatibility) TLS 1.2 only, with a vetted AEAD cipher list, ECDHE forward secrecy, OCSP stapling where the CA still serves OCSP, and TLS 1.0/1.1/SSLv3 plus CBC/3DES/RC4 fully disabled. Benchmarked with the TLSRef Configurator (the community successor to Mozilla's SSL Config Generator, which was archived May 2026) and Qualys SSL Labs (target grade A/A+).

**Why it matters / why AI-coded apps miss it.** AI-coded apps inherit whatever the PaaS/reverse-proxy default is and never run SSL Labs, so they silently ship TLS 1.0/1.1, weak CBC suites, no OCSP stapling, or a missing intermediate chain (the #1 cause of 'works in Chrome, fails on mobile/curl'). Weak suites enable downgrade and padding-oracle attacks; a broken chain breaks non-browser clients. AI tools also still emit ssl-config.mozilla.org config blobs that now redirect, so practitioners need the current TLSRef source.

**How to implement.** Generate config from the TLSRef Configurator (https://configurator.tlsref.org/, formerly ssl-config.mozilla.org which now redirects there; guidelines at https://docs.tlsref.org/server-side-tls.html, formerly wiki.mozilla.org/Security/Server_Side_TLS) using the 'Intermediate' profile as the public-internet baseline (TLS 1.2+1.3) and 'Modern' (TLS 1.3-only) where the client base allows. TLS 1.3 suites: TLS_AES_128_GCM_SHA256, TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256. TLS 1.2 (Intermediate) OpenSSL list: ECDHE-ECDSA-AES128-GCM-SHA256, ECDHE-RSA-AES128-GCM-SHA256, ECDHE-ECDSA-AES256-GCM-SHA384, ECDHE-RSA-AES256-GCM-SHA384, ECDHE-ECDSA-CHACHA20-POLY1305, ECDHE-RSA-CHACHA20-POLY1305, plus DHE-RSA variants. nginx: ssl_protocols TLSv1.2 TLSv1.3; ssl_prefer_server_ciphers off (let the TLS 1.3 client pick); ssl_ciphers <list>; ssl_ecdh_curve X25519:prime256v1; ssl_stapling on; ssl_stapling_verify on; resolver ...; serve the full chain (fullchain.pem). Prefer ECDSA P-256 certs (smaller, faster); dual ECDSA+RSA only if you need legacy clients. Verify with 'sslyze --regular host' or SSL Labs; in CI use testssl.sh. Caddy and modern Traefik do most of this by default - verify, do not assume.

**When it applies.** Every app (any service terminating its own TLS). Drops to 'verify-only' if you sit fully behind Cloudflare/Fastly/an ALB that terminates TLS - but then harden the edge profile and the origin link instead.

**Sources:** <https://configurator.tlsref.org/> · <https://docs.tlsref.org/server-side-tls.html> · <https://www.ssllabs.com/ssltest/> · <https://developer.mozilla.org/en-US/docs/Web/Security/Practical_implementation_guides/TLS>

### 🔵 Certificate revocation in 2025: OCSP stapling sunset, CRLs, and short-lived certs

**What it is.** The revocation-checking model is changing. OCSP (including stapling and OCSP Must-Staple) is being retired across the Web PKI; the replacement strategy is CRLs (browser-aggregated, e.g. Chrome CRLSets / Mozilla CRLite) plus much shorter certificate lifetimes so a leaked key has a small exposure window.

**Why it matters / why AI-coded apps miss it.** Guides written before 2025 tell you to enable 'OCSP Must-Staple' - following that now causes issuance failures or hard-fails. AI tools trained on older content will confidently emit obsolete config. Practitioners need the current picture: stapling is still fine where supported but is no longer the strategic answer, and you should not request Must-Staple from Let's Encrypt.

**How to implement.** Let's Encrypt timeline (confirmed): 2025-01-30 OCSP Must-Staple requests fail for accounts that hadn't previously issued a Must-Staple cert; 2025-05-07 OCSP URLs removed from newly issued certs (replaced by CRL URLs) and all Must-Staple requests fail; 2025-08-06 OCSP responders shut down. Action items: (1) Do NOT enable OCSP Must-Staple on new Let's Encrypt certs. (2) Keep ssl_stapling on only where your CA still serves OCSP (harmless and helps); LE certs issued after 2025-05-07 have no OCSP URL, so stapling there is a no-op - do not depend on it. (3) Move to short-lived certs (the TLSRef Modern profile recommends a 90-day maximum lifetime with ECDSA to force automation; LE/most ACME CAs auto-renew) and fully automate renewal (certbot, acme.sh, lego, or built-in Caddy/Traefik ACME) - a missed renewal is now a more likely outage than a revocation gap. (4) Non-browser TLS clients (VPN concentrators, legacy SDKs) that hard-fail on OCSP must be reconfigured. Set CAA accounturi binding (see DNS item) so automation is the only thing that can issue.

**When it applies.** Once you have users/data (anyone running production TLS with auto-renewed certs). Critical for non-browser/IoT/VPN stacks that did OCSP hard-fail.

**Sources:** <https://letsencrypt.org/2024/12/05/ending-ocsp/> · <https://www.abetterinternet.org/post/ending-ocsp/> · <https://docs.tlsref.org/server-side-tls.html>

### 🟢 HSTS with preload (and the Certificate Transparency ecosystem)

**What it is.** Strict-Transport-Security forces browsers to use HTTPS for your domain for a set period, eliminating the first-request SSL-strip window; the preload list bakes this into browsers so even the very first visit is protected. Certificate Transparency (CT) is the public-log system that makes any cert issued for your domain auditable.

**Why it matters / why AI-coded apps miss it.** AI scaffolds redirect HTTP->HTTPS but rarely add HSTS, leaving a downgrade/MITM window on the first hit and on any link typed without https. Worse, they sometimes add HSTS with includeSubDomains;preload to a domain whose other subdomains are NOT all HTTPS-ready, which bricks those subdomains in every browser until the (slow) preload-removal process completes. CT is now mandatory (browsers reject certs lacking SCTs) yet teams never monitor logs, so a mis-issued/rogue cert for their domain goes unnoticed.

**How to implement.** Header (HTTPS responses only, never over HTTP): Strict-Transport-Security: max-age=63072000; includeSubDomains; preload. Preload requirements (hstspreload.org, confirmed): max-age >= 31536000 (1 year), includeSubDomains present, preload present, valid cert, served on the HTTPS base domain, and HTTP (port 80) redirects to HTTPS on the same host. Roll out safely: ship a short max-age (e.g. 300) first, then 1 day, then ramp to 2 years, and only add 'preload' after every subdomain is confirmed HTTPS - removal from the preload list is slow and global. CT: you do not 'enable' it (CAs embed Signed Certificate Timestamps / SCTs at issuance automatically), but you MUST monitor: subscribe to crt.sh, Cloudflare CT Monitoring, or cert-spotter/Certspotter and alert on unexpected issuance. Pair with CAA (next item) to prevent unauthorized issuance in the first place. IMPORTANT: Expect-CT is deprecated (not removed) - it was deprecated in Chromium 107 because Chromium now enforces CT by default, and it is dropped from standards. Do NOT add the Expect-CT header; CT enforcement is automatic in browsers now and the header is obsolete (effectively a no-op).

**When it applies.** Every app for HSTS (basic max-age+includeSubDomains). 'Once you have users/data' for preload (commit carefully) and CT monitoring (matters once your domain/brand is worth impersonating).

**Sources:** <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Strict-Transport-Security> · <https://hstspreload.org/> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Expect-CT> · <https://developers.cloudflare.com/ssl/edge-certificates/additional-options/certificate-transparency-monitoring/>

### 🟢 DNS security: DNSSEC, CAA records (with account binding), registrar/registry lock, NS hygiene

**What it is.** Hardening the naming layer: DNSSEC cryptographically signs zone data (RRSIG/DNSKEY) anchored via a DS record in the parent zone to stop spoofing/cache poisoning; CAA records (RFC 8659) restrict which CAs may issue certs for your domain, with RFC 8657 extending this to a specific ACME account (accounturi) and validation method (validationmethods); registrar/registry lock blocks unauthorized domain transfers/NS changes; NS hygiene means locking down zone transfers and removing stale records.

**Why it matters / why AI-coded apps miss it.** AI-coded projects treat DNS as 'point the CNAME and forget it.' None of DNSSEC, CAA, or registry lock is ever set, so the domain is one phished registrar login or one DNS-hijack away from total takeover - and an attacker who controls DNS can also obtain a valid TLS cert (defeating all your TLS work) unless CAA stops them. As of CA/Browser Forum ballot SC-085v2 (passed June 2025, effective 2026-03-15), CAs must validate DNSSEC when present during CAA and domain-control-validation lookups, so DNSSEC now directly strengthens issuance security.

**How to implement.** DNSSEC: enable via managed DNS provider toggle/API (Cloudflare, Route 53, NS1) which generates KSK+ZSK and signs the zone; then publish the DS record at your registrar to complete the chain of trust (the step people forget - DNSSEC does nothing until the DS is in the parent). Plan key rollovers; a botched rollover causes a hard SERVFAIL outage, so prefer provider-managed/automatic key management. CAA: add at apex - example: 'example.com. IN CAA 0 issue "letsencrypt.org; accounturi=https://acme-v02.api.letsencrypt.org/acme/acct/12345; validationmethods=dns-01"', a matching issuewild line (or 'issuewild ";"' to forbid wildcards), and 'iodef "mailto:security@example.com"' for violation reports. Account/method binding (RFC 8657) is the strong form: even an attacker who controls DNS briefly cannot get a cert from a different account/method. Registry/registrar: enable Registry Lock for apex/high-value domains, registrar-level Transfer Lock, 2FA + SSO on the registrar account, disable SMS recovery, restrict authorized contacts. NS hygiene: 'allow-transfer { none; };' (BIND) or confirm AXFR is disabled at your managed provider so attackers can't pull the whole zone; remove stale/orphan records (feeds subdomain takeover); restrict and minimize wildcard records. Audit with a DNS security checker, dig CAA/DNSKEY/DS, and DNSViz for the chain.

**When it applies.** Once you have users/data for DNSSEC + registry lock + NS hygiene. CAA is cheap enough to be 'Every app' (one DNS record) and is the single highest-leverage DNS control. Registry Lock specifically is 'Regulated/enterprise / high-value brand' (it adds manual change friction).

**Sources:** <https://www.rfc-editor.org/rfc/rfc8659> · <https://www.rfc-editor.org/rfc/rfc8657> · <https://cabforum.org/2025/06/18/ballot-sc-085v2-require-validation-of-dnssec-when-present-for-caa-and-dcv-lookups/> · <https://www.cloudflare.com/learning/dns/dnssec/how-dnssec-works/> · <https://developers.cloudflare.com/ssl/edge-certificates/caa-records/>

### 🟠 Subdomain takeover via dangling CNAMEs: detection and prevention

**What it is.** When a DNS record (typically a CNAME or NS) points to a deprovisioned third-party resource (S3 bucket, Azure App Service, GitHub Pages, Heroku, Netlify, Fastly, Shopify, Zendesk, etc.), an attacker re-registers that resource name on the provider and serves attacker content on YOUR subdomain - inheriting its trust, cookies, CSP allowlisting, and OAuth redirect whitelisting.

**Why it matters / why AI-coded apps miss it.** This is the prototypical AI-vibe-coding failure: agents and scaffolds spin up and tear down SaaS/cloud endpoints (preview deploys, demo buckets, marketing pages) and add DNS records, but the teardown order is wrong (resource deleted first, DNS left dangling) and there is no DNS inventory. The hijacked subdomain enables convincing phishing, cookie theft (if cookies are scoped to the parent domain), CSP bypass, and OAuth account takeover via a whitelisted redirect.

**How to implement.** Correct decommissioning ORDER (OWASP cheat sheet, confirmed): (1) repoint/redirect the subdomain to a maintenance page or remove the DNS record FIRST, (2) wait for TTL/propagation, (3) only THEN delete the cloud resource - the common mistake is doing this in reverse, which opens an immediate takeover window. Prevention: keep a version-controlled DNS source of truth (Terraform, OctoDNS, or DNSControl) so every add/remove is reviewed and logged, and a mapping (CMDB/spreadsheet) of each record to its owning resource. Avoid broad wildcard records; use provider domain-verification (Azure TXT asuid, Google site verification) so a name can't be claimed by another tenant. Reduce blast radius: don't scope sensitive cookies to the parent domain, use exact-match (not wildcard) OAuth redirect URIs, and list exact hosts (not wildcards) in CSP. Detection: run a fingerprint scanner at least weekly - dnsReaper (40+ provider fingerprints), nuclei takeover templates, subjack/subzy; cross-reference EdOverflow/can-i-take-over-xyz (the canonical 'Can I take over XYZ?' list, source of the fingerprints used by most tools) for which services are currently vulnerable; monitor crt.sh / certspotter for unexpected certs on your subdomains. Alert specifically on provider fingerprints such as 'NoSuchBucket', "There isn't a GitHub Pages site here", 'No such app', or 'Fastly error: unknown domain'.

**When it applies.** Once you have users/data, and rises sharply with org size and number of SaaS integrations. For a single-domain monolith with no third-party CNAMEs it is low-risk; for any company using cloud storage + multiple SaaS subdomains it is a near-certainty over time and belongs in weekly automation.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Subdomain_Takeover_Prevention_Cheat_Sheet.html> · <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/02-Configuration_and_Deployment_Management_Testing/10-Test_for_Subdomain_Takeover> · <https://github.com/EdOverflow/can-i-take-over-xyz>

### 🟢 Edge defense: DDoS protection, WAF managed rules, rate limiting, bot management, origin cloaking

**What it is.** A layered edge in front of the origin: L3/4 network DDoS mitigation (SYN/UDP/DNS floods), L7 application DDoS mitigation, a WAF with managed rulesets (OWASP-style SQLi/XSS/CMS probes), rate limiting on sensitive endpoints, bot management/challenges, and origin cloaking so the real server IP can't be hit directly (Cloudflare Authenticated Origin Pulls / mTLS, AWS Shield + ALB-only ingress).

**Why it matters / why AI-coded apps miss it.** AI-coded apps assume 'the host handles DDoS' and expose the origin's public IP directly, so any volumetric or L7 flood (or a $5 booter) takes them down and bypasses any CDN they thought was protecting them. They also ship login/OTP/signup/search/GraphQL endpoints with zero rate limiting, enabling credential stuffing, OTP brute force, scraping, and cost-amplification (each unmetered request burns serverless/DB/LLM spend). A WAF is often toggled on but left in 'log only' or with no managed ruleset.

**How to implement.** Cloudflare baseline: enable the Cloudflare Managed Ruleset (covers SQLi/XSS/common probes), set the HTTP DDoS Attack Protection managed ruleset to an active action (its origin-error mitigation triggers at the High default sensitivity of 1,000 errors/sec; Adaptive DDoS Protection additionally learns a per-zone origin-error profile and mitigates deviations), and enable the Network-layer ruleset for L3/4. Rate limiting (Cloudflare best practices, confirmed values): count by IP, session cookie, header (x-api-key), or JA3/JA4 TLS fingerprint - concrete starting points: login 4 req/min (Managed Challenge) escalating to 20 req/hr (Block), OTP 5 req/min by cookie with a 10-min block, scraping 10 req/2min, per-product-ID 50 req/10s; count only error responses (e.g. status in 401/403) to avoid blocking legit users; escalate action Managed Challenge -> Block -> long block. Bot management: turn on bot challenges for high-value/automatable paths. Origin cloaking: enable Authenticated Origin Pulls (mTLS) so the origin only accepts connections bearing Cloudflare's client cert - a zone-level or per-hostname cert is stronger than the shared Global cert (Global AOP only proves traffic came from Cloudflare's network, not from YOUR account) - and firewall the origin to the CDN IP ranges only. AWS equivalent: AWS Shield (Standard always-on; Advanced for L7 + cost protection) + AWS WAF with AWSManagedRules (Common, KnownBadInputs, SQLi, IP reputation, Bot Control) + rate-based rules in a 5-minute sliding window, layered (one blanket rule, e.g. 500/5min, plus tighter per-URI scope-down rules as low as 100/5min), with ingress restricted to the ALB/CloudFront; rate-based rules in Block mode are a prerequisite for Shield Advanced cost-protection credits. Akamai App & API Protector is the enterprise analog. Always run WAF in count/observe mode briefly, tune false positives, then enforce.

**When it applies.** Rate limiting on auth/OTP/expensive endpoints is 'Every app' (cheap, prevents brute force and cost blowups). Full WAF managed rules + bot management + L7 DDoS + origin cloaking is 'Once you have users/data' and becomes mandatory 'At scale' or for any app that is a plausible attack/extortion target. Shield Advanced / Akamai-tier is 'Regulated/enterprise' due to cost.

## Advanced browser-platform hardening

This layer covers the browser's own defense-in-depth primitives that sit a level above "use HTTPS and set SameSite cookies" — the headers and HTML attributes that turn the browser into an enforcement engine against XSS, supply-chain script tampering, clickjacking, cross-window/cross-origin leaks, and cookie scoping confusion. AI code generators consistently produce apps that are functionally correct but ship with none of these: there is no CSP at all (or a uselessly permissive one with unsafe-inline), third-party CDN scripts have no integrity hashes, cookies are set with bare names instead of __Host- prefixes, and links/postMessage handlers trust whatever the other side sends. Each item below names the exact header or attribute, the exact value to use, the deployment path (almost always: ship Report-Only first, watch violations, then enforce), and an honest tier because several of these (cross-origin isolation, Trusted Types) are genuinely "only if you need them" rather than universal. Currency notes verified during research (June 2026): Trusted Types reached cross-browser Baseline "Newly available" in February 2026 when Firefox 148 shipped it (Safari 26 added it Sept 2025; Chrome/Edge since 83), so the long-standing "Chromium-only" caveat is now obsolete — it is finally deployable as a real defense rather than a Chrome-only nicety. CHIPS/Partitioned has likewise gone cross-browser (Chrome 114+, Firefox 141+, Safari 26.2+), so it is no longer a Chromium experiment either. Sources verified to resolve and support each claim: W3C CSP3 (Working Draft, 5 May 2026), MDN, Google web.dev, Google Privacy Sandbox, and OWASP Cheat Sheets (Content Security Policy, Clickjacking Defense, HTML5 Security, Third-Party JavaScript, Unvalidated Redirects).

### 🟢 Strict CSP: nonces + strict-dynamic (not host allowlists)

**What it is.** A Content-Security-Policy Level 3 header that gates script execution on a per-response cryptographic nonce (or, for cached/static pages, a hash) plus 'strict-dynamic', instead of the legacy approach of allowlisting CDN hostnames. The browser only runs <script> tags carrying the matching nonce/hash, and 'strict-dynamic' propagates that trust to scripts those trusted scripts inject, so you do not have to enumerate every transitive dependency.

**Why it matters / why AI-coded apps miss it.** Host-allowlist CSPs are trivially bypassable (a single JSONP endpoint or open redirect on an allowlisted CDN defeats them) and are what most generators emit if they emit CSP at all — more commonly AI-coded apps ship no CSP, or one containing 'unsafe-inline'/'unsafe-eval' which is equivalent to no XSS protection. Nonce + strict-dynamic is the only CSP shape Google and OWASP consider actually effective against injection.

**How to implement.** Header: Content-Security-Policy: script-src 'nonce-{RANDOM}' 'strict-dynamic'; object-src 'none'; base-uri 'none'. Generate a fresh 128-bit+ base64 nonce per HTTP response (never reuse, never on a static/cached page — for those use a hash source 'sha256-{HASH}' instead), inject it server-side into every first-party <script nonce="{RANDOM}"> and remove all inline event handlers (onclick=) and javascript: URLs. object-src 'none' kills Flash/plugin vectors; base-uri 'none' stops <base> tag hijacking of relative script URLs. Backward-compat line for old browsers (modern browsers ignore unsafe-inline when a nonce/hash is present): script-src 'nonce-{RANDOM}' 'strict-dynamic' https: 'unsafe-inline'. Deploy via Content-Security-Policy-Report-Only with a report-to/report-uri endpoint first, watch for false positives, then flip to the enforcing header. Note: strict-dynamic makes SRI on dynamically-injected scripts moot (the propagated nonce trust governs), so the two are complementary, not redundant.

**When it applies.** Once you have users/data — every app that renders any user- or third-party-influenced content should run a strict CSP; it is the single highest-leverage XSS mitigation but requires templating discipline (nonce plumbing), so it is a deliberate hardening step rather than a one-line default.

**Sources:** <https://web.dev/articles/strict-csp> · <https://cheatsheetseries.owasp.org/cheatsheets/Content_Security_Policy_Cheat_Sheet.html> · <https://www.w3.org/TR/CSP3/>

### 🔵 Trusted Types: require-trusted-types-for 'script' (DOM-XSS defense)

**What it is.** A CSP mechanism (require-trusted-types-for 'script', optionally with the trusted-types directive to allowlist policy names) that makes dangerous DOM injection sinks — innerHTML, outerHTML, insertAdjacentHTML, document.write, iframe.srcdoc, script.src, eval, setTimeout(string), new Function() — refuse plain strings and accept only TrustedHTML/TrustedScript/TrustedScriptURL objects minted by a named, audited policy. These two directives are defined in the W3C Trusted Types specification (a CSP extension), surfaced as CSP directives.

**Why it matters / why AI-coded apps miss it.** Nonce-based CSP stops attacker-injected <script> tags but does NOT stop DOM-XSS where trusted first-party code passes attacker-controlled data into innerHTML/eval. AI-generated front-end code is full of element.innerHTML = userData and similar sinks because that is the most obvious way to render dynamic content; Trusted Types is the only browser primitive that systematically converts those sinks into enforced choke points instead of leaving them as latent vulnerabilities scattered across the codebase.

**How to implement.** Header: Content-Security-Policy: require-trusted-types-for 'script'; trusted-types myPolicy. Define a policy in JS: trustedTypes.createPolicy('myPolicy', { createHTML: s => DOMPurify.sanitize(s) }). To cover un-modifiable third-party code, create a policy literally named 'default' (trustedTypes.createPolicy('default', {...})) — the browser auto-invokes the default policy at every protected sink — and allowlist that name in the trusted-types directive; sanitize there with e.g. DOMPurify.sanitize(str, {RETURN_TRUSTED_TYPE:true}). The trusted-types directive controls allowed policy names ('none' to forbid all policies, '*' for any unique name, 'allow-duplicates' to permit reusing a name). Roll out with Content-Security-Policy-Report-Only: require-trusted-types-for 'script' plus a report endpoint, fix each flagged sink (rewrite to textContent or route through the policy), then enforce. Currency: as of February 2026 Trusted Types reached cross-browser Baseline 'Newly available' (Chrome/Edge 83+, Safari 26+, Firefox 148+) — the historical 'Chromium-only' excuse for skipping it is gone, and non-supporting legacy browsers simply ignore the directive, so it is safe to ship.

**When it applies.** Once you have users/data (for any app with rich client-side DOM rendering). Highest value for SPAs/React-with-dangerouslySetInnerHTML, dashboards, and anything injecting server data into the DOM; lower priority for purely server-rendered, escape-by-default templates.

**Sources:** <https://web.dev/articles/trusted-types> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy/trusted-types> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy/require-trusted-types-for>

### 🔵 Subresource Integrity (SRI) for third-party scripts/styles

**What it is.** An integrity attribute carrying a base64 cryptographic hash (sha256/sha384/sha512) on <script> and <link rel=stylesheet|preload|modulepreload> elements. The browser computes the fetched resource's hash and refuses to execute/apply it on any mismatch, so a tampered or swapped CDN file is blocked rather than run.

**Why it matters / why AI-coded apps miss it.** Pulling jQuery/Bootstrap/analytics straight off a public CDN with no integrity check is the default pattern in generated HTML and is a direct software-supply-chain hole: if the CDN, its account, or a transitive dependency is compromised (Polyfill.io-class incidents), every visitor silently runs attacker JavaScript with full page privileges. SRI pins the exact reviewed bytes. AI tooling essentially never adds integrity/crossorigin attributes.

**How to implement.** Example: <script src="https://cdn.example.com/lib.js" integrity="sha384-…" crossorigin="anonymous"></script>. Generate the hash with: cat lib.js | openssl dgst -sha384 -binary | openssl base64 -A (or use srihash.org / a bundler plugin like vite-plugin-sri / webpack-subresource-integrity). crossorigin="anonymous" is mandatory for cross-origin resources — without CORS the browser fetches in no-cors/opaque mode and cannot read the body to hash it, so SRI silently fails; the CDN must send Access-Control-Allow-Origin. Prefer sha384 or sha512; you can list multiple space-separated hashes (browser accepts the resource if any matches, and uses the strongest it supports). Caveat: SRI pins a specific version, so use it on pinned/immutable URLs, not auto-updating 'latest' endpoints. require-sri-for is non-standard/removed, so enforce 'SRI everywhere' via build process/CI lint rather than a CSP directive.

**When it applies.** Once you have users/data — applies to any page loading scripts/styles from a domain you do not control; cheap, static, and a clean supply-chain win, but only meaningful for third-party/CDN assets (self-hosted same-origin bundles gain little).

**Sources:** <https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity> · <https://owasp.org/www-community/controls/SubresourceIntegrity> · <https://cheatsheetseries.owasp.org/cheatsheets/Third_Party_Javascript_Management_Cheat_Sheet.html>

### 🟢 Cookie name prefixes __Host- / __Secure- and Partitioned (CHIPS)

**What it is.** Magic cookie-name prefixes that the browser enforces: __Secure- requires the Secure attribute (HTTPS only); __Host- additionally forbids a Domain attribute and forces Path=/, locking the cookie to the exact host that set it. Partitioned (CHIPS) gives a cross-site cookie a separate jar per top-level site so it cannot be used for cross-site tracking and cannot be shared across embedding contexts.

**Why it matters / why AI-coded apps miss it.** Generated auth code sets session=… with no prefix and often no Domain control, which leaves the door open to cookie-fixation and subdomain-injection attacks: a compromised or attacker-controlled sibling subdomain (xss.victim.com) can write a Domain=victim.com cookie that overrides the real session, because plain cookies carry no integrity about who set them. The __Host- prefix is a free, browser-enforced guarantee that a cookie was set Secure, host-only, on path / — exactly the properties a session cookie needs — yet AI code virtually never uses it.

**How to implement.** Session cookie: Set-Cookie: __Host-session=<value>; Secure; Path=/; HttpOnly; SameSite=Lax (no Domain attribute — adding one makes the browser reject the whole cookie). Use __Secure- when you genuinely need a Domain (cross-subdomain) cookie but still want the Secure guarantee. For a cookie that must be sent in a third-party/embedded context, opt into CHIPS: Set-Cookie: __Host-embed=<value>; Secure; Path=/; SameSite=None; Partitioned — the Partitioned attribute itself strictly requires only Secure (the browser rejects Partitioned without Secure); SameSite=None is what actually lets the cookie travel in the third-party context, and __Host- is the recommended companion so it is bound to the hostname rather than the registrable domain. Combine with HttpOnly (block JS theft) and the right SameSite. Verify the browser actually accepted prefixed cookies in DevTools, since a single violated constraint causes a silent reject. Currency: CHIPS/Partitioned is now cross-browser (Chrome 114+, Firefox 141+, Safari 26.2+), no longer Chromium-only.

**When it applies.** Every app that sets a session/auth cookie — __Host- is a zero-cost rename plus correct attributes. Partitioned/CHIPS is conditional: only for cookies your site needs when embedded as a third party (widgets, embeds, SSO iframes).

**Sources:** <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie> · <https://privacysandbox.google.com/cookies/chips> · <https://github.com/privacycg/CHIPS>

### 🟢 Clickjacking defense: CSP frame-ancestors (+ X-Frame-Options fallback)

**What it is.** Response headers that tell the browser who, if anyone, may embed your page in a frame/iframe. CSP frame-ancestors is the modern CSP2+ control (supports 'none', 'self', or an explicit origin list); X-Frame-Options (DENY / SAMEORIGIN) is the older single-purpose header kept for legacy-browser coverage.

**Why it matters / why AI-coded apps miss it.** Without a framing policy, an attacker overlays your authenticated page invisibly under decoy UI and harvests real clicks (UI-redress / clickjacking) — toggling settings, confirming transactions, approving OAuth. Generated apps almost never set either header, and X-Frame-Options alone is insufficient (its ALLOW-FROM value is obsolete and dead in modern browsers, and it cannot express multi-origin allowlists). Client-side 'frame-busting' JavaScript is independently defeatable (double-framing, sandbox, onBeforeUnload, 204-flushing).

**How to implement.** Primary: Content-Security-Policy: frame-ancestors 'none'; (page must never be framed) or 'self' (only your own origin) or 'self' https://trusted-partner.example for explicit embedders. Note frame-ancestors does NOT fall back to default-src, so it must be set explicitly. Fallback for old browsers: X-Frame-Options: DENY (or SAMEORIGIN). Set both — per the CSP spec, browsers that understand frame-ancestors should honor it over X-Frame-Options, so frame-ancestors governs where supported and XFO covers the rest. Defense-in-depth: set session cookies SameSite=Lax/Strict so a framed cross-site request carries no auth (limits, does not replace, the headers). Do NOT rely on JS frame-busters as the control. Apply the policy on every HTML response, including error pages.

**When it applies.** Every app with authenticated state or state-changing actions — two static headers, no app logic, and a standard finding in any security review.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Clickjacking_Defense_Cheat_Sheet.html> · <https://owasp.org/www-community/attacks/Clickjacking> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Content-Security-Policy/frame-ancestors>

### 🟢 Reverse tabnabbing, open redirects, and postMessage origin checks

**What it is.** Three related cross-window/cross-origin trust failures: (1) reverse tabnabbing — a target=_blank link lets the opened page rewrite window.opener.location to a phishing clone; (2) open redirect — a redirect endpoint that forwards to an attacker-supplied URL; (3) postMessage handlers that trust messages without verifying event.origin, and senders that post with targetOrigin '*'.

**Why it matters / why AI-coded apps miss it.** All three are 'trust the other side' bugs that AI code routinely introduces: it emits <a target=_blank> with no rel, redirect handlers that do res.redirect(req.query.url) verbatim, and message listeners that act on event.data with no origin check (or post secrets with '*'). Each enables credential phishing, OAuth/token theft via redirect laundering, or cross-origin data exfiltration — and none is caught by transport security or CSP. (Modern evergreen browsers now imply rel=noopener on target=_blank, but stating it explicitly is still the correct hardening because older/embedded webviews do not.)

**How to implement.** Reverse tabnabbing: add rel="noopener noreferrer" to every target=_blank link (noreferrer implies noopener); for JS use window.open(url, name, 'noopener,noreferrer'). Open redirects: never redirect to a raw user-supplied URL — map a short token/ID server-side to a fixed destination, or validate against an allowlist of hosts/paths (allow-list, not deny-list). As a practitioner add-on beyond the cheat sheet text: also decode before checking and reject protocol-relative //evil.com and backslash variants, which slip past naive 'starts with /' checks. postMessage receiver: first line of the handler must be an exact-match origin check — if (event.origin !== 'https://trusted.example') return; — never substring/indexOf (the OWASP sheet flags message.origin.indexOf('.owasp.org') as insecure because owasp.org.attacker.com would pass), then validate the data shape, and never feed it to innerHTML/eval (use textContent). Sender: always pass the explicit expected targetOrigin as the 2nd arg to postMessage, never '*'. Reinforce with Referrer-Policy: no-referrer or strict-origin-when-cross-origin.

**When it applies.** Every app — these are pure code-pattern fixes (link attributes, a redirect allowlist, an origin guard) with no infra cost; the open-redirect and postMessage cases in particular are high-frequency, high-impact findings.

**Sources:** <https://owasp.org/www-community/attacks/Reverse_Tabnabbing> · <https://cheatsheetseries.owasp.org/cheatsheets/HTML5_Security_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Unvalidated_Redirects_and_Forwards_Cheat_Sheet.html>

### 🟠 Cross-origin isolation: COOP / COEP / CORP

**What it is.** A trio of headers that put your top-level document in a 'cross-origin isolated' agent cluster. COOP (Cross-Origin-Opener-Policy) severs the window.opener relationship to cross-origin windows; COEP (Cross-Origin-Embedder-Policy) blocks embedding any cross-origin subresource that has not explicitly opted in; CORP (Cross-Origin-Resource-Policy) is the per-resource opt-in that says which sites may embed that resource. Together COOP:same-origin + COEP:require-corp make self.crossOriginIsolated === true.

**Why it matters / why AI-coded apps miss it.** Two distinct payoffs. First, isolation re-enables powerful APIs that were disabled after Spectre — SharedArrayBuffer, high-resolution performance.now(), measureUserAgentSpecificMemory() — so apps using WASM threads, ffmpeg.wasm, SQLite-wasm, or multithreaded compute simply will not work without it, and AI-generated setups for those libraries omit the headers and then fail cryptically. Second, COOP independently mitigates cross-window XS-leaks and opener-based attacks even if you never need SharedArrayBuffer. The COEP requirement is the part that breaks naive setups because every cross-origin image/font/script suddenly needs a CORP/CORS opt-in.

**How to implement.** On the top-level document: Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Embedder-Policy: require-corp. Then every cross-origin resource you embed must send Cross-Origin-Resource-Policy: cross-origin (or same-site / same-origin to scope tighter) or be loaded with CORS (crossorigin attribute + Access-Control-Allow-Origin). If you cannot add CORP to third-party resources, use Cross-Origin-Embedder-Policy: credentialless (Chromium), which loads them without credentials instead of requiring CORP. Roll out safely with Cross-Origin-Opener-Policy-Report-Only and Cross-Origin-Embedder-Policy-Report-Only plus a reporting endpoint to find every resource that would break, then enforce; confirm success at runtime via self.crossOriginIsolated. If you only want opener hardening without the COEP pain, COOP: same-origin (or same-origin-allow-popups for OAuth popups) alone is a valid standalone XS-leak defense.

**When it applies.** At scale / specialized — COOP alone is broadly useful as an XS-leak/opener defense, but full COOP+COEP cross-origin isolation is required mainly when you need SharedArrayBuffer / WASM threads / precise timers, and it carries real breakage cost (every embedded cross-origin asset must opt in), so it is a deliberate, tested rollout rather than a default header to bolt on.

**Sources:** <https://web.dev/articles/coop-coep> · <https://web.dev/articles/cross-origin-isolation-guide> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Cross-Origin-Opener-Policy>

## Identity & account security (deep)

This is the advanced identity layer for AI-coded apps that already have basic login, headers, and a top-50 checklist. It assumes the obvious controls (bcrypt/argon2 hashing, HTTPS, "have MFA") are done and drills into the failure modes that ship in vibe-coded auth: passwords that aren't checked against breach corpora, login/reset/signup endpoints that leak which accounts exist via message or timing, OAuth/OIDC clients wired up with copy-pasted flows that skip PKCE S256, exact redirect-URI matching, and state/nonce validation, SAML service-provider code that trusts an assertion's signature without checking what was signed or who it was issued to, and reset/email-change flows that allow account takeover because they don't verify ownership of both addresses or re-authenticate for sensitive changes. The authoritative spine is the current OWASP Cheat Sheets (Credential Stuffing, Authentication, Forgot Password, SAML Security), RFC 9700 / BCP 240 "Best Current Practice for OAuth 2.0 Security" (published January 2025), RFC 9207 (iss for mix-up), RFC 9449 (DPoP, for sender-constrained tokens), RFC 9470 (step-up challenge), the IETF "OAuth 2.0 for Browser-Based Apps" BCP draft (now -26, in the RFC Editor queue as of Dec 2025), the Have I Been Pwned k-anonymity Pwned Passwords API, and the OWASP "Changing a User's Registered Email Address" control. Each item below names the exact header, parameter, claim, or library to use, not just the principle. Tiers are honest: breached-password checks, enumeration-uniform responses, and correct OAuth/OIDC validation belong in essentially every app with accounts; device fingerprinting, sender-constrained tokens, BFF token storage, and SAML hardening kick in once you have real users, federation, or enterprise customers.

### 🟢 Breached-credential (Pwned Passwords) checks at password-set time, layered with MFA

**What it is.** On every signup, password change, and (ideally) login, check the chosen password against a corpus of known-breached passwords and require/strongly encourage MFA. Credential stuffing replays username/password pairs leaked from other sites; the only structural fix is making sure your users' passwords aren't already in the dumps and that a known password alone is not sufficient to log in.

**Why it matters / why AI-coded apps miss it.** AI-coded apps implement a password policy (length, complexity) but almost never check the password against breach data, so users happily set 'Password123!' which is in every dump. The OWASP Credential Stuffing Cheat Sheet cites Microsoft's analysis that MFA would have stopped 99.9% of account compromises, yet generated auth scaffolds frequently leave MFA as an unimplemented TODO. Complexity rules do nothing against stuffing because the leaked passwords are already 'complex'.

**How to implement.** Call the Have I Been Pwned Pwned Passwords range API with k-anonymity: SHA-1 (or NTLM) the candidate password client-or-server-side, send ONLY the first 5 hex chars of the hash to GET https://api.pwnedpasswords.com/range/{prefix}, then compare the returned suffix list locally — the full password/hash never leaves your boundary. Add the 'Add-Padding: true' request header so the response is padded with decoy '0'-count entries to a random 800-1,000 records, hiding whether there was a real match from a network observer. For high volume or air-gapped use, self-host the downloadable corpus (HaveIBeenPwned/PwnedPasswordsAzureFunction) or a k-anonymity proxy. This satisfies OWASP ASVS v4 requirement 2.1.7 (verify new passwords against breach datasets). Reject or warn on hits; never store the user's plaintext or SHA-1. Pair with TOTP/WebAuthn MFA, and notify users (with prior login date/time/location) when a correct password fails the subsequent MFA step — that signal means their password is compromised.

**When it applies.** Every app — Pwned Passwords range check is a few lines and free; MFA enrollment is once-you-have-users but should be designed in from day one.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Credential_Stuffing_Prevention_Cheat_Sheet.html> · <https://haveibeenpwned.com/api/v3> · <https://www.troyhunt.com/enhancing-pwned-passwords-privacy-with-padding/> · <https://github.com/HaveIBeenPwned/PwnedPasswordsAzureFunction>

### 🔵 Device / connection fingerprinting and adaptive, identifier-aware throttling

**What it is.** Beyond IP rate limits, build risk signals from the request itself (device and connection fingerprints) and throttle/step-up based on the targeted account and the source, so a botnet spraying one password across millions of accounts from rotating residential IPs still gets caught.

**Why it matters / why AI-coded apps miss it.** Naive rate limiting in AI-coded apps is per-IP, which credential-stuffing toolkits trivially defeat with rotating residential/proxy IPs, and 'lock the account after N failures' tied to a username creates a denial-of-service against legitimate users. The actual attack pattern — low-and-slow, one or two attempts per account across huge account sets — is invisible to per-IP counters.

**How to implement.** Layer signals: (1) device fingerprint (OS, browser, language, screen, fonts, plugins) via a library like fingerprintjs — treat as advisory since clients can spoof it; (2) connection fingerprinting (JA3/JA4 TLS, HTTP/2 frame fingerprint, HTTP header order) to flag headless/bot stacks; (3) IP intelligence — correlate against published hosting ranges (AWS/GCP publish theirs), VPN/proxy lists, and residential-vs-datacenter classification. Use graduated friction not hard blocks: require username-then-password sequencing, require a CSRF token fetch and JS execution before accepting login (defeats simple scripted clients), and present CAPTCHA only to suspicious requests. Apply MFA adaptively on new-device/new-IP/impossible-travel. Lockout/throttle should be account-aware AND source-aware, and store per-account authentication IP history so you can scope a breach response. Monitor metrics across each layer rather than trusting any single signal.

**When it applies.** Once you have users/data — meaningful when you are actually a stuffing target; over-engineering fingerprinting on a pre-launch app is wasted effort.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Credential_Stuffing_Prevention_Cheat_Sheet.html>

### 🟢 Account/user enumeration defense: uniform responses AND uniform timing

**What it is.** Login, registration, password reset, and email-change endpoints must reveal nothing about whether a given username/email exists — identical response body, identical HTTP status code, and (critically) constant-time processing regardless of account existence or password correctness.

**Why it matters / why AI-coded apps miss it.** This is one of the most common vibe-coded leaks because the framework's default 'helpful' messages ('No account with that email', 'Email already registered', 'Welcome back!') are an enumeration oracle, and reset/signup forms almost always confirm existence. Even after messages are fixed, the timing side channel remains: a generated handler does an early return when the user is not found (skipping the expensive bcrypt verify), so 'user exists' responses are measurably slower, letting an attacker enumerate by latency before stuffing.

**How to implement.** Login: return one generic message ('Login failed; invalid user ID or password.') and the SAME HTTP status for both bad-user and bad-password. Reset: 'If that email address is in our database, we will send a reset link.' Signup: 'A link to activate your account has been emailed.' — never 'this email is already in use' (handle a duplicate by emailing the existing account instead). To kill timing leaks, run the same code path no matter what: the OWASP Authentication Cheat Sheet warns that 'quick exit' on a missing user creates a time-based oracle, and recommends a constant-time flow — so when the user does not exist, still perform a hash verification against a fixed dummy/decoy hash so password-check time is constant. Keep HTTP status codes identical (do not mix 200/403). Add per-account rate limiting and CAPTCHA to raise enumeration cost; if you do account lockout, tie it carefully and prefer throttling so it can't be weaponized into DoS or itself become an oracle. The same uniform-message + uniform-timing rule applies to email-change and MFA endpoints.

**When it applies.** Every app — pure correctness, no infrastructure cost, and it is a default-on bug in most generated auth code.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html>

### 🟢 OAuth 2.0 / OIDC client hardening: PKCE S256, exact redirect-URI match, state, nonce, no implicit/ROPC

**What it is.** When your app is an OAuth/OIDC client (Login with Google/GitHub, or a first-party SPA/native app), wire the flow to RFC 9700 / OAuth 2.1: Authorization Code grant with PKCE, exact-string redirect-URI matching, CSRF protection, ID-token validation, and never the implicit or password grants.

**Why it matters / why AI-coded apps miss it.** RFC 9700 (BCP 240, published January 2025) makes Authorization Code + PKCE the universal baseline; it says clients SHOULD NOT use the implicit grant, and OAuth 2.1 removes implicit entirely. AI-coded integrations frequently copy stale tutorials: they use response_type=token (implicit), skip PKCE, register a wildcard or substring redirect URI, forget to verify state, or never validate the ID token's nonce/iss/aud — each of which enables token leakage, authorization-code injection, CSRF login, or mix-up attacks across multiple IdPs.

**How to implement.** Use Authorization Code grant with PKCE on ALL clients: generate a high-entropy code_verifier, send code_challenge with code_challenge_method=S256 (never 'plain' — RFC 9700 notes S256 is the only method that does not expose the verifier, and the AS MUST mitigate PKCE downgrade by rejecting a code_verifier at the token endpoint when no code_challenge was present in the authorization request). Register and exact-string match the full redirect_uri (RFC 9700 mandates exact string comparison) — NO wildcards, NO trailing-path tricks; the only allowed exception is the variable port of a localhost loopback in native apps (RFC 8252). CSRF: a one-time, session-bound state value bound to the user session (PKCE or the OIDC nonce can substitute, but state is the safe default for plain OAuth). OIDC: always send and validate nonce in the ID token (replay/injection defense — do NOT consume any issued token until the nonce check passes), and validate iss, aud, exp, and signature. For multiple IdPs, defend against mix-up attacks via the iss authorization-response parameter (RFC 9207) or distinct redirect URIs per AS. Clients SHOULD NOT use the implicit grant (response_type=token), and MUST NOT use the Resource Owner Password Credentials grant. Never put access tokens in URL query parameters (RFC 6750). For SPAs, prefer a Backend-For-Frontend (BFF) that holds tokens server-side and gives the browser only an HttpOnly session cookie, rather than storing tokens in localStorage (per the OAuth 2.0 for Browser-Based Apps BCP). Where tokens must be exposed, consider sender-constraining them with DPoP (RFC 9449) so a stolen token can't be replayed.

**When it applies.** Every app that uses OAuth/OIDC — these are non-negotiable correctness requirements; BFF token storage and DPoP are once-you-have-users / production.

**Sources:** <https://www.rfc-editor.org/rfc/rfc9700.html> · <https://www.rfc-editor.org/info/rfc9700/> · <https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-26.html> · <https://auth0.com/blog/demystifying-oauth-security-state-vs-nonce-vs-pkce/>

### 🔴 SAML service-provider validation: signature scope, XSW defense, audience/recipient/timestamps

**What it is.** If you accept SAML SSO (the common enterprise path), the service-provider code must verify not just that a signature is valid but that it covers the exact assertion you trust, that the assertion was issued FOR you, recently, and in response to your request — defeating XML Signature Wrapping (XSW) and assertion-replay attacks.

**Why it matters / why AI-coded apps miss it.** SAML is XML, and the classic XSW attack injects a second forged assertion (e.g. elevating user to admin or impersonating another user) while leaving the original signature technically valid — the SP validates the signature on element A but reads claims from attacker-controlled element B. AI-coded or hand-rolled SAML handlers (and even popular libraries historically) use getElementsByTagName / relative XPath and skip audience/recipient checks, so a signed assertion minted for app X is replayed to app Y. SAML is rarely covered well by generic security tooling, so these bugs ship silently.

**How to implement.** (1) Schema-validate the XML first, against LOCAL trusted copies of the SAML schemas (never auto-download), with schema hardening to disable wildcard types. (2) Confirm the ds:Reference URI in the signature actually covers the <saml:Assertion> element whose claims you consume; reject unsigned assertions and never trust embedded KeyInfo — pin the IdP's signing cert/key from out-of-band metadata (StaticKeySelector / X509KeySelector). Use absolute XPath, never getElementsByTagName, to select security-relevant nodes. (3) Validate <saml:Audience> equals your SP EntityID; validate the Destination on <samlp:Response> exactly equals your ACS URL; validate Recipient in <SubjectConfirmationData>. (4) Validate NotBefore / NotOnOrAfter on both Conditions and SubjectConfirmationData, plus InResponseTo matching your original AuthnRequest ID, and implement replay detection (OneTimeUse / cache of seen assertion IDs). (5) Require signature algorithm RSA-SHA-256 or stronger — reject rsa-sha1, hmac-sha1, and SHA-1 DigestMethod (NIST SP 800-131A Rev. 2 disallows SHA-1 in digital signatures). (6) Require TLS for all exchanges; treat IdP-initiated/unsolicited responses as higher risk (no login-CSRF protection) and allowlist RelayState redirect targets. Use a maintained library and keep it patched rather than parsing XML yourself.

**When it applies.** Regulated/enterprise — only relevant once you sell to orgs that demand SAML SSO, but then it is high-stakes (admin impersonation / cross-app takeover) and easy to get subtly wrong.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/SAML_Security_Cheat_Sheet.html> · <https://workos.com/blog/fun-with-saml-sso-vulnerabilities-and-footguns>

### 🟢 Password-reset & email-change hardening, account recovery, and step-up auth

**What it is.** The recovery surface — forgot-password, change-email, and recovery flows — is the real account-takeover front door, so reset tokens must be unguessable/single-use/short-lived and stored hashed, email changes must prove control of BOTH addresses, and security-sensitive actions must require step-up (fresh) authentication.

**Why it matters / why AI-coded apps miss it.** Reset and email-change are where AI-coded apps lose accounts wholesale: predictable or long-lived reset tokens, tokens stored in plaintext, auto-login after reset, sessions not invalidated post-reset, and — most damaging — email changes that take effect after verifying only the NEW address (or no verification at all), letting an attacker with a borrowed session or a guessed password silently move the account to their inbox. Many flows also never re-authenticate for sensitive changes, so a hijacked session = total takeover.

**How to implement.** Reset token: cryptographically random, high entropy, SINGLE-use (invalidate on use), short expiry, and stored HASHED (treat it like a password). Send the reset link only to the already-registered email; show the uniform 'if that address exists…' message with constant timing (enumeration defense); rate-limit per account; prefer throttling over lockout so reset can't be turned into DoS. Add Referrer-Policy: no-referrer on reset pages so a token in the URL doesn't leak via Referer to third-party assets. After reset: do NOT auto-login; require re-login; invalidate existing sessions (or prompt to); and email a confirmation that a reset occurred (never email the password). Email change (the OWASP control): require the current password (re-auth), store the new address as PENDING, and send messages to BOTH addresses — to the OLD address a confirm/abort notice and to the NEW address a verification link — using time-limited nonces; only switch the registered email after the change is confirmed, keep the old address active until then, and log the user out to re-authenticate. Step-up: for high-value actions (changing email/password/MFA, payments, deleting account), require a fresh auth event — via OIDC acr_values / max_age, or the OAuth Step-Up Authentication Challenge (RFC 9470), where the resource server returns WWW-Authenticate with error=insufficient_user_authentication and the required acr_values/max_age, and the client re-runs authorization to obtain a token reflecting the stronger/recent auth (acr + auth_time claims).

**When it applies.** Every app for the reset-token and email-change basics; step-up via RFC 9470 / acr is once-you-have-users or regulated/enterprise (sensitive operations, financial-grade).

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html> · <https://owasp.org/www-community/pages/controls/Changing_Registered_Email_Address_For_An_Account> · <https://www.rfc-editor.org/rfc/rfc9470.html> · <https://www.authlete.com/developers/stepup_authn/>

**Key references:** [OWASP Credential Stuffing Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Credential_Stuffing_Prevention_Cheat_Sheet.html) · [OWASP Authentication Cheat Sheet (enumeration & timing)](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html) · [OWASP Forgot Password Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html) · [OWASP SAML Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/SAML_Security_Cheat_Sheet.html) · [OWASP Control: Changing a User's Registered Email Address](https://owasp.org/www-community/pages/controls/Changing_Registered_Email_Address_For_An_Account) · [RFC 9700 / BCP 240 — Best Current Practice for OAuth 2.0 Security (Jan 2025)](https://www.rfc-editor.org/rfc/rfc9700.html) · [RFC 9207 — OAuth 2.0 Authorization Server Issuer Identification (mix-up defense)](https://www.rfc-editor.org/rfc/rfc9207.html) · [RFC 9449 — OAuth 2.0 Demonstrating Proof of Possession (DPoP, sender-constrained tokens)](https://www.rfc-editor.org/rfc/rfc9449.html) · [RFC 9470 — OAuth 2.0 Step Up Authentication Challenge Protocol](https://www.rfc-editor.org/rfc/rfc9470.html) · [Have I Been Pwned — Pwned Passwords API (k-anonymity range)](https://haveibeenpwned.com/api/v3) · [Troy Hunt — Enhancing Pwned Passwords Privacy with Padding (Add-Padding header)](https://www.troyhunt.com/enhancing-pwned-passwords-privacy-with-padding/) · [IETF — OAuth 2.0 for Browser-Based Apps (BFF / token storage), draft -26](https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-26.html) · [Auth0 — State vs. Nonce vs. PKCE](https://auth0.com/blog/demystifying-oauth-security-state-vs-nonce-vs-pkce/)

## API, GraphQL & realtime security — OWASP API Top 10 (2023)

The OWASP API Security Top 10 (2023) is dominated by authorization failures, not injection: the top risk (API1 BOLA), the merged property-level risk (API3), and function-level access (API5) are all "the server trusted an ID/role/field the client supplied." AI-coded apps are especially prone to these because the generated code almost always wires up CRUD and ORM serialization correctly (so it works in the demo) but silently skips per-object ownership checks, returns whole DB rows, and binds whole request bodies onto models. Authorization is the one thing a scaffold cannot infer — it requires knowing your business rules — so it is the single most common and most severe gap. This deep layer goes past "add auth middleware": object- and field-level checks on every record, mass-assignment allowlists, resource/cost ceilings (rate, payload, pagination, GraphQL depth/complexity/batching), anti-automation on sensitive business flows, treating third-party API responses as untrusted input, hardening GraphQL (introspection off, persisted-query/trusted-document allowlists, batching/alias caps) and realtime channels (Origin allowlist to stop CSWSH, handshake + per-message auth, message size/rate caps), and locking down object-storage signed URLs (short TTL, least-privilege signer, content-length/type conditions, server-side signatureAge enforcement). Sources are OWASP API Security Top 10 2023, OWASP WSTG, OWASP Cheat Sheets (GraphQL, WebSocket), Apollo, graphql.org, and AWS (Prescriptive Guidance + S3 user guide) — all verified to resolve and support the claims (graphql.org/learn/security bot-blocks the fetcher but is live and on-topic).

### 🟢 API1 BOLA + API5 BFLA: per-object and per-function authorization

**What it is.** BOLA (API1, the #1 API risk) is when an endpoint the user is allowed to call lets them act on an object they don't own by manipulating an ID (e.g. GET /orders/124 returns someone else's order). BFLA (API5) is when the user reaches a function/endpoint they should never reach at all — an admin route, or the same route via a different HTTP method (GET allowed, but DELETE/PUT/POST not blocked).

**Why it matters / why AI-coded apps miss it.** BOLA is the most exploited API flaw because IDs are trivially enumerable and the bug is invisible in a single-user demo — every test passes when you're logged in as the owner. AI scaffolds generate `db.order.findUnique({where:{id}})` straight from the path param and almost never add `AND ownerId = currentUser.id`, and they protect `/admin/*` paths but miss admin-capable actions sitting inside ordinary controllers or reachable via an unguarded HTTP verb (OWASP's BFLA scenario is exactly a GET→POST method swap to an invite endpoint).

**How to implement.** Enforce authorization at the data layer, not just middleware: every query that takes a client-supplied ID must be scoped to the caller (`WHERE id=? AND owner_id=:me`, or a centralized `can(user, action, object)` policy via Oso/Casbin/Cedar/CASL, or Postgres Row-Level Security). OWASP is explicit that comparing the session user ID to a body field is NOT sufficient — re-check ownership in every function that uses client input to load a record. Deny-by-default for functions (OWASP: deny all access by default, grant explicitly per role): admin controllers inherit from an authz-checking base; enforce allowed roles AND allowed HTTP methods per route (block method-swap). Prefer random UUIDv4/ULID over sequential integer IDs to raise enumeration cost (OWASP recommends unpredictable GUIDs as defense in depth, not a substitute for checks). Write automated authz tests (user A cannot read/modify/delete user B's objects, and cannot reach admin functions) and fail the build on regression.

**When it applies.** Every app (any multi-tenant or multi-user app — this is the most common AND most severe AI-coded gap)

**Sources:** <https://owasp.org/API-Security/editions/2023/en/0xa1-broken-object-level-authorization/> · <https://owasp.org/API-Security/editions/2023/en/0xa5-broken-function-level-authorization/> · <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/12-API_Testing/02-API_Broken_Object_Level_Authorization>

### 🟢 API3 Broken Object Property-Level Authorization (mass assignment + excessive data exposure)

**What it is.** The 2023 merge of two 2019 risks with one root cause: missing authorization at the individual property level. Excessive data exposure = the response includes fields the user shouldn't read (password hashes, internal flags, other users' PII, total_stay_price). Mass assignment = the request body can set fields the user shouldn't write (role=admin, isVerified=true, balance, blocked=false).

**Why it matters / why AI-coded apps miss it.** AI-generated code defaults to whole-object serialization (`return user`, `res.json(record)`, `to_json()`, `model_dump()`) and whole-body binding (`User.update(req.body)`, `Object.assign(entity, body)`, Pydantic/ActiveRecord/Mongoose models that accept any field). Both work perfectly in the demo and leak/accept everything by default. The dangerous fields (role, ownerId, price, internal status) ride along invisibly.

**How to implement.** Outbound: never serialize raw entities — use explicit response DTOs/serializers that cherry-pick fields (FastAPI `response_model`, NestJS `class-transformer` @Expose/@Exclude with excludeExtraneousValues, Rails `ActiveModel::Serializer`/Jbuilder, .NET DTOs, Java MapStruct). OWASP explicitly says avoid generic `to_json()`/`to_string()` and add schema-based response validation as an extra layer, keeping returned data to the bare minimum. Inbound: bind only an explicit allowlist of writable properties — Rails strong params `permit(:title,:body)`, Mongoose strict schemas (never `{strict:false}`), Django serializer `fields`/`read_only_fields`, Spring `@JsonIgnore`/DTOs, never spread `...req.body` onto a model. Treat sensitive fields (role, permissions, price, ownership, verification, balance) as server-controlled only.

**When it applies.** Every app

**Sources:** <https://owasp.org/API-Security/editions/2023/en/0xa3-broken-object-property-level-authorization/> · <https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/12-API_Testing/03-Testing_for_Excessive_Data_Exposure>

### 🔵 API4 Unrestricted Resource Consumption (rate, payload, pagination & cost ceilings)

**What it is.** APIs without limits on request rate, body/array/string size, pagination page size, query cost, or downstream spend let a single client exhaust CPU/memory/storage/bandwidth — or run up real money on metered third parties (SMS, email, cloud egress). OWASP's examples: a money-burning attack on forgot-password triggering tens of thousands of $0.05 SMS in minutes, and an 18GB download (over the 15GB cache limit) turning a ~$13/mo bill into $8,000.

**Why it matters / why AI-coded apps miss it.** Scaffolds ship endpoints with no rate limiter, accept unbounded JSON bodies, allow `?limit=999999` or unbounded GraphQL lists, and call Twilio/SendGrid/OpenAI with no per-user cap or billing alarm. None of this surfaces in normal use; it surfaces on the first abusive script or a bad bill. AI code also rarely sets container memory/CPU limits or upload size caps.

**How to implement.** Layer the controls: (1) Rate limiting per identity AND per IP with stricter buckets on expensive/sensitive ops (login, OTP, password reset, signup, search, AI/LLM calls) — express-rate-limit/rate-limiter-flexible, nginx limit_req, API gateway (AWS API Gateway usage plans/throttling, Cloudflare/Kong), or a token bucket in Redis. (2) Payload caps (OWASP: enforce max string length, max array elements, and max upload file size): max body size (express `express.json({limit:'100kb'})`, nginx `client_max_body_size`), plus magic-number type checks on uploads. (3) Server-side pagination caps: clamp `limit`/`per_page` to a hard max (e.g. 100), reject larger, prefer cursor pagination for big collections. (4) Per-request timeouts and container resource limits (Kubernetes requests/limits, Lambda memory/timeout). (5) Spend guardrails (OWASP: set provider spending limits, or billing alerts when limits aren't possible): AWS Budgets, Cloud Billing alerts, per-user quotas on metered third-party calls. (6) Reject zip-bombs/decompression bombs and bound regex/JSON-parse work.

**When it applies.** Once you have users/data (any internet-exposed endpoint; the money-burning variants matter the moment you wire up SMS/email/LLM/cloud)

**Sources:** <https://owasp.org/API-Security/editions/2023/en/0xa4-unrestricted-resource-consumption/> · <https://cheatsheetseries.owasp.org/cheatsheets/Denial_of_Service_Cheat_Sheet.html>

### 🔵 API6 sensitive business flows + API10 unsafe consumption of third-party APIs

**What it is.** API6: an endpoint exposes a business flow that is harmful when automated even though each individual request is technically authorized — ticket/console scalping, bulk reservation to starve inventory (OWASP's airline-seat example), referral/credit farming, comment/spam flooding. API10: your service trusts data from upstream third-party APIs more than user input — processing their responses without validation, over an insecure channel, and blindly following their redirects.

**Why it matters / why AI-coded apps miss it.** API6 is a logic/anti-automation problem a code scanner can't see and AI scaffolds never address — the flow works as designed; the abuse is doing it 10,000x/sec from headless browsers. API10 is missed because devs (and AI) assume 'it's Stripe/GitHub/a partner, it's safe,' so third-party data flows unsanitized into SQL/templates/redirects — a poisoned or compromised integration becomes injection or an open redirect that exfiltrates tokens.

**How to implement.** API6 (OWASP engineering-layer mitigations): identify abuse-prone flows with the business, then raise attacker cost — device fingerprinting that denies unexpected clients (headless browsers), CAPTCHA/challenge (hCaptcha/Turnstile) or biometric checks on the flow, analyze user flow to detect non-human patterns (add-to-cart→purchase in <1s), block Tor exit nodes / well-known proxies, and add per-account and per-flow quotas plus inventory holds. Secure machine-to-machine/B2B endpoints to the same standard as public ones (OWASP notes these often lack the usual protections). API10: validate and sanitize every field from third-party responses exactly like user input before using it in queries/templates/commands (parameterized queries, output encoding); use TLS for all integration calls; set timeouts; maintain an allowlist of permitted redirect destinations and do NOT blindly follow 3xx redirects; assess the provider's security posture before integrating.

**When it applies.** Once you have users/data (API6 matters for commerce/booking/referral/UGC apps; API10 the moment you call any external API)

**Sources:** <https://owasp.org/API-Security/editions/2023/en/0xa6-unrestricted-access-to-sensitive-business-flows/> · <https://owasp.org/API-Security/editions/2023/en/0xaa-unsafe-consumption-of-apis/>

### 🟠 GraphQL hardening: introspection off, depth/complexity caps, batching/alias limits, persisted/trusted queries

**What it is.** GraphQL concentrates several API risks: introspection/GraphiQL hand attackers the full schema; nested queries and large list args (`first: 999999`) cause DoS; a single HTTP request can carry many operations via query batching or repeated aliases (alias overloading) to bypass naive per-request rate limits and brute-force OTP/login; and the graph's node/nodes pattern enables IDOR.

**Why it matters / why AI-coded apps miss it.** GraphQL servers commonly default to introspection ON and unbounded query depth/cost, and most rate limiters count HTTP requests — useless against alias/batch abuse where one request does thousands of operations (OWASP notes batching enables brute-forcing passwords, OTPs, and session tokens). AI scaffolds rarely add depth/complexity instrumentation or batching caps, and frequently leave verbose stack traces and 'Did you mean' field suggestions on in production.

**How to implement.** Disable introspection and GraphiQL in prod (Apollo: introspection:false; graphql-java NoIntrospectionGraphqlFieldVisibility; express-graphql NoIntrospection validation rule) and disable the 'Did you mean' field-suggestion hint. Enforce a max query depth (graphql-depth-limit; Java MaxQueryDepthInstrumentation) and a query cost/complexity ceiling (graphql-cost-analysis, graphql-validation-complexity; Java MaxQueryComplexityInstrumentation) — OWASP calls cost analysis the most thorough DoS defense; apply a separate smaller limit to nested-list depth (graphql.org). Cap list sizes / require pagination bounds. Defend batching/aliases at the CODE level, not just nginx: limit operations per batch, cap alias/field repetition, and rate-limit by query cost and by sensitive object (OWASP: implement object-level request rate limiting or block batching for sensitive objects like OTP/token/login). For first-party clients, move to a persisted-query / trusted-document allowlist: turn OFF Automatic Persisted Queries (APQ lets clients register arbitrary operations at runtime — it is a cache, not a safelist) and run the router/server in safelist mode so only pre-registered operation IDs execute (Apollo persisted query lists). Enforce authz on nodes AND edges (resolver-level ownership checks to prevent IDOR), validate inputs via scalars/enums, use parameterized DB access, set a server timeout, and suppress stack traces in prod.

**When it applies.** Once you have users/data (any production GraphQL API; persisted-query/trusted-document allowlisting is At scale / first-party hardening)

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/GraphQL_Cheat_Sheet.html> · <https://www.apollographql.com/docs/graphos/platform/security/persisted-queries> · <https://graphql.org/learn/security/>

### 🟠 WebSocket security (Origin allowlist, handshake + per-message auth, message caps) and signed object-storage URLs

**What it is.** WebSockets bypass the same-origin/CORS model: browsers send cookies on the WS handshake but enforce no SOP, enabling Cross-Site WebSocket Hijacking (CSWSH) where a malicious page opens an authenticated socket as the victim. They also need per-message authorization, transport encryption, and message size/rate limits. Separately, object-storage presigned URLs (S3/GCS/Azure SAS) are bearer tokens: anyone holding the URL gets exactly the permissions and lifetime baked in, and an S3 presigned PUT will silently overwrite an existing object with the same key.

**Why it matters / why AI-coded apps miss it.** AI-generated realtime code typically validates nothing on the handshake, treats 'connected = fully authorized' (no per-message authz, no re-validation), runs ws:// in dev and forgets wss:// + Origin checks in prod, and accepts unbounded messages. For uploads, scaffolds mint presigned URLs with multi-day expiry, the app's broad credentials, and no content-type/size restriction — a leaked URL becomes a long-lived read/write grant or an arbitrary-object-overwrite vector.

**How to implement.** WebSocket (per OWASP cheat sheet): validate the Origin header against an explicit allowlist on every handshake (exact-match, no wildcards/substring) — this is the primary CSWSH defense, and malicious JS cannot forge Origin; add a CSRF token in the handshake if you already use CSRF protection; never rely on cookies alone. Authenticate at the handshake (session/JWT) AND authorize every message ('connection != unlimited access'); periodically re-validate long-lived sessions. Use wss:// in prod, never ws://. Cap message size (cheat sheet suggests ~64KB or less) and rate (~100 msgs/min baseline), validate every message against a JSON schema/allowlist, verify binary by magic number not declared type, and add nonce/timestamp checks against replay. Don't put long-lived tokens in the WS URL (they land in access logs and must be redacted) — pass post-connect. Presigned/SAS URLs (AWS): shortest viable expiry (S3 console caps at 12h; CLI/SDK SigV4 max is 7 days with IAM-user creds, and STS/role-signed URLs die when the session ends, typically 1–6h); sign with a least-privileged principal — AWS is explicit that a presigned URL can never grant access the signer lacks; constrain uploads with conditions — content-length-range (size cap) and Content-Type via a POST policy — and enforce server-side with a bucket policy `s3:signatureAge` condition (e.g. deny if signature >10 min old) and optional aws:SourceIp/data-perimeter conditions; prefer temporary STS credentials over long-lived keys; monitor usage via CloudTrail/access logs.

**When it applies.** Once you have users/data (WebSocket items for any realtime app; signed-URL hardening the moment you offer direct-to-storage upload/download)

## Email & domain integrity

Email and domain integrity is where vibe-coded apps silently leak trust: the app sends password-reset and receipt mail from a domain anyone can spoof, ships with p=none "monitoring" DMARC forever, never hardens its parked/marketing subdomains, and has a registrar account protected by a single reused password. This layer goes past "set up SPF/DKIM/DMARC" into the parts practitioners actually get wrong: identifier alignment (the thing that makes DMARC mean anything), reaching and surviving p=reject without dropping legitimate mail, transport security on the wire (MTA-STS + TLS-RPT, with DANE where DNSSEC exists), brand/anti-impersonation signaling (BIMI with VMC/CMC), making transactional links un-phishable (token entropy, host-header injection, enumeration), and locking the domain itself at the registry so a stolen registrar login cannot repoint your MX/NS. Currency note: DMARCbis has now been PUBLISHED as RFC 9989 (Proposed Standard, May 2026), obsoleting RFC 7489 and RFC 9091 — it removes pct, deprecates rf/ri, replaces the Public Suffix List with a DNS Tree Walk, and adds psd/np/t. p=none remains the standards-endorsed STARTING point (RFC 9989 calls it Monitoring Mode); the failure mode is never advancing past it, not using it during rollout. Primary sources: RFC 9989 (DMARCbis) plus the obsoleted RFC 7489, RFC 8461 (MTA-STS), RFC 8460 (TLS-RPT), RFC 8058 (one-click unsubscribe), the BIMI Group / DigiCert VMC-CMC guidance, Google/Yahoo bulk-sender rules (Feb 2024) and Microsoft's 2025 Outlook enforcement, OWASP Forgot Password / Authentication cheat sheets, M3AAWG parked-domain BCP (2022), and ICANN EPP-status / registry-lock and DNSSEC/CT guidance.

### 🟢 DMARC enforcement (p=reject) with correct SPF/DKIM identifier alignment

**What it is.** DMARC is a DNS TXT record at _dmarc.<domain> that tells receivers what to do when a message fails authentication, and crucially requires identifier alignment: SPF or DKIM must not only pass but the authenticated domain must align with the visible From: header domain. p=reject instructs receivers to reject spoofed mail outright. The governing spec is now RFC 9989 (DMARCbis, May 2026), which obsoletes the original RFC 7489.

**Why it matters / why AI-coded apps miss it.** Without DMARC at enforcement, anyone can send mail with your exact From: address (CEO fraud, fake password resets, invoice fraud) and it lands in inboxes. AI-coded apps almost universally stop at p=none ('monitoring') and never advance, or jump to p=reject and break legitimate mail because they never checked alignment — SPF/DKIM can pass while still failing DMARC if the passing domain doesn't align with From:. p=none provides zero spoofing protection; it only collects reports. Note p=none is not a mistake in itself — RFC 9989 explicitly defines it as the Monitoring-Mode starting point — the mistake is treating it as the destination.

**How to implement.** Publish at _dmarc.example.com: v=DMARC1; p=reject; rua=mailto:dmarc@example.com; adkim=s; aspf=s; fo=1. Roll out in stages: start p=none with rua= aggregate reporting, ingest the XML reports with a tool/dashboard (do not eyeball raw XML), confirm every legitimate sender — your app's ESP (SES/SendGrid/Postmark/Mailgun), marketing platform, support desk, calendar — passes a DMARC-aligned SPF or DKIM, then move p=none -> p=quarantine -> p=reject. Prefer DKIM alignment over SPF because SPF breaks on forwarding; alignment defaults to relaxed (organizational-domain match), tighten to strict (adkim=s/aspf=s) only when you control all senders. Isolate third-party senders on dedicated subdomains (mail.example.com, news.example.com) so one vendor's misconfig can't force you off reject. Set sp=reject for subdomains and explicitly protect parked/non-sending domains and subdomains with v=spf1 -all, a null MX (example.com. IN MX 0 . per RFC 7505), and v=DMARC1; p=reject. Currency: RFC 9989 (obsoletes RFC 7489 + RFC 9091) removes the pct tag entirely, deprecates rf/ri, replaces the Public Suffix List with an in-protocol DNS Tree Walk to find the organizational domain, and adds psd (public-suffix-domain flag), np (non-existent-subdomain policy), and t (test mode). Write records that do NOT rely on pct — gate rollout with the np tag and staged policy instead.

**When it applies.** Every app — any domain that sends mail (even just password resets/receipts) needs to reach p=reject; parked/unused domains need the null-MX + SPF -all + p=reject lockdown. Bulk senders (5,000+/day to Gmail/Yahoo) have been required to publish a DMARC record (p=none is sufficient for that bar) since Feb 2024; Microsoft's Outlook began routing non-compliant high-volume mail to Junk on 5 May 2025 and moved to outright rejection (550 5.7.15) for senders lacking SPF/DKIM/DMARC.

**Sources:** <https://www.rfc-editor.org/rfc/rfc9989.html> · <https://datatracker.ietf.org/doc/html/rfc7489> · <https://support.google.com/a/answer/81126> · <https://www.m3aawg.org/sites/default/files/doc_files/m3aawg_parked_domains_bcp-2022-06.pdf> · <https://techcommunity.microsoft.com/blog/microsoftdefenderforoffice365blog/strengthening-email-ecosystem-outlook%E2%80%99s-new-requirements-for-high%E2%80%90volume-senders/4399730>

### 🔵 MTA-STS + TLS-RPT (and DANE) for SMTP transport encryption

**What it is.** MTA-STS (RFC 8461) lets a receiving domain publish a policy declaring that sending MTAs MUST use TLS with a valid, matching certificate to your MX hosts — closing STARTTLS-stripping and MX-spoofing downgrade attacks. TLS-RPT (RFC 8460) publishes a reporting address so senders send you daily I-JSON reports of TLS negotiation / policy failures.

**Why it matters / why AI-coded apps miss it.** SMTP's STARTTLS is opportunistic: a network attacker can strip the STARTTLS advertisement and force plaintext delivery of your password-reset and OTP emails, or present a bogus cert via MX spoofing, with no warning. DMARC stops header spoofing but does nothing for the transport hop. AI-coded apps configure DMARC and assume mail is 'secure,' missing that the wire itself can be downgraded. MTA-STS + TLS-RPT is what makes encryption enforceable and gives you visibility into failures.

**How to implement.** Three pieces: (1) DNS TXT at _mta-sts.example.com: v=STSv1; id=<unique opaque string, e.g. 20260619T0000Z> — bump id on every policy change (id has no implied ordering, it just must change). (2) Serve the policy over HTTPS at https://mta-sts.example.com/.well-known/mta-sts.txt with a cert valid for the mta-sts.example.com DNS-ID, HTTP 200, no redirects, fields: version: STSv1 / mode: enforce / mx: <each MX, *.example.com wildcard allowed for the leftmost label only> / max_age: <seconds, RFC 8461 caps it at 31557600 ≈ 1 year>. (3) DNS TXT at _smtp._tls.example.com: v=TLSRPTv1; rua=mailto:tlsrpt@example.com. Deploy mode: testing first, watch TLS-RPT reports for failures (cert mismatches, MX not listed, expired certs), then switch to mode: enforce; keep max_age long once stable. If your MX provider supports DNSSEC, ALSO publish DANE TLSA records — DANE and MTA-STS are complementary (DANE is anchored by DNSSEC validation, MTA-STS by web-PKI + DNS, so they fail differently and back each other up). Major inbound providers (Google, Microsoft 365/Exchange Online) now support MTA-STS for inbound mail.

**When it applies.** Once you have users/data — any domain receiving sensitive mail (reset links, OTPs, PII) should enforce MTA-STS; it requires you to control your MX and serve an HTTPS host, so it's a deliberate step beyond the DNS-only basics.

**Sources:** <https://datatracker.ietf.org/doc/html/rfc8461> · <https://www.rfc-editor.org/rfc/rfc8460.html> · <https://techcommunity.microsoft.com/blog/exchange/introducing-mta-sts-for-exchange-online/3106386>

### 🔴 BIMI with VMC/CMC for authenticated brand logo display

**What it is.** BIMI (Brand Indicators for Message Identification) is a DNS TXT record at default._bimi.<domain> that points to your logo (l= SVG URL) and authority-evidence certificate (a= VMC or CMC URL). Supporting mailbox providers display your verified logo next to authenticated mail. It is gated on DMARC enforcement.

**Why it matters / why AI-coded apps miss it.** BIMI is an anti-impersonation and trust signal: a displayed logo cues that the message is authenticated and from you, and (with a VMC/CMC) that a CA verified your trademark or logo provenance. It is impossible to obtain without DMARC at quarantine/reject on the organizational domain, so it doubles as a forcing function and a real-world check that your enforcement is genuine. AI-coded apps skip it entirely and miss both the brand-trust and the implicit 'did we actually reach enforcement' verification. It is NOT a security control by itself — it's an authenticated-identity display that only works on top of correct DMARC.

**How to implement.** Prerequisites first: DMARC at p=quarantine or p=reject on the organizational domain, with no sp=none carve-out (and no pct under DMARCbis since pct is gone), sustained — providers/CAs effectively want a stable enforcement history (commonly cited as roughly 6-8 weeks before issuance) before a cert provisions. Produce the logo as SVG Tiny 1.2 Portrait/Secure (SVG Tiny PS), 1:1 square, small (~32 KB or under), no scripts/external refs; host over HTTPS. Publish default._bimi.example.com TXT: v=BIMI1; l=https://example.com/logo.svg; a=https://example.com/vmc.pem. For a= choose: VMC (DigiCert/Entrust) requires a registered trademark plus a rigorous, EV-style identity/trademark verification (in-person/notary/video validation of the organization), and on Gmail unlocks the blue verified checkmark; CMC (Common Mark Certificate) widens eligibility to non-trademarked / older logos — Gmail's CMC option (rolled out early 2025) instead requires proof the logo has been publicly displayed on your domain for ~12 months via archive verification. Gmail requires a certified logo (VMC or CMC) and does not display self-asserted records; Yahoo/AOL accept self-asserted (blank a=) records; Apple Mail also supports BIMI. A self-asserted (no-cert) record gets display only on providers that allow it.

**When it applies.** Regulated/enterprise (and brand-conscious senders) — optional polish, valuable for high-volume or impersonation-targeted brands (finance, SaaS, retail). The VMC trademark + certificate cost (and even CMC's 12-month-display bar) makes it overkill for a side project, but the DMARC-enforcement prerequisite it forces is universally good.

**Sources:** <https://bimigroup.org/verified-mark-certificates-vmc-and-bimi/> · <https://www.digicert.com/blog/bimi-setup-guide-for-vmc-and-cmc> · <https://redsift.com/guides/bimi-in-2026-verified-logos-cmcs-and-the-fastest-path-to-inbox-display>

### 🟢 Transactional-email anti-phishing and link safety

**What it is.** Hardening the app-generated mail itself (password resets, magic links, receipts, OTPs, invites) and its embedded links so the messages train users to recognize legitimate mail, cannot be weaponized, and cannot be confused with phishing. Includes one-click unsubscribe done safely (RFC 8058) for any bulk/marketing stream.

**Why it matters / why AI-coded apps miss it.** Transactional mail is the single highest-value phishing target an app produces, and AI-coded apps generate it most carelessly: reset tokens with weak entropy or long/no expiry, links built from an attacker-controllable Host header (host-header injection -> poisoned reset link to attacker domain), redirect endpoints in the link that enable open-redirect/token-leak, account enumeration via different responses, and links to bare IPs/raw click-trackers that look exactly like phishing. They also ship marketing blasts without compliant unsubscribe, getting throttled or rejected by Gmail/Yahoo/Outlook.

**How to implement.** Tokens: cryptographically-random, >=128 bits of entropy (e.g. 32 hex / URL-safe base64 chars from a CSPRNG), single-use, short TTL (15-60 min — OWASP says expire after an appropriate short period), invalidated on use and on password change, compared in constant time, never logged. Links: build absolute URLs from a server-side configured canonical base URL, never from the Host/X-Forwarded-Host header (OWASP explicitly warns against trusting the Host header — use a hardcoded or allowlisted domain) to prevent host-header injection of reset links; send recipients straight to your apex/known host over HTTPS — no open-redirect ?next=/?url= passthrough, no link-shortener or bare-IP links in security mail. Account-enumeration: return an identical 'if an account exists, we sent a link' response for existent and non-existent addresses (per the OWASP Forgot Password and Authentication cheat sheets), and rate-limit reset requests per-account and per-IP. Anti-phishing hygiene: consistent, plain From: identity and a dedicated sending subdomain; never ask users to click and enter their current password; keep copy free of urgency/credential-harvesting patterns so it stays distinguishable from phishing. One-click unsubscribe (RFC 8058) for marketing streams (required by Gmail/Yahoo for 5,000+/day senders since Feb 2024, and by Outlook in 2025): List-Unsubscribe: <https://example.com/unsub/<opaque>> plus List-Unsubscribe-Post: List-Unsubscribe=One-Click; the message MUST carry a valid DKIM signature whose h= covers both List-Unsubscribe and List-Unsubscribe-Post, the URL MUST be HTTPS with an opaque/hard-to-forge token (so spammers can't forge unsubscribes), the POST body MUST be exactly List-Unsubscribe=One-Click with NO cookies/authorization/context, and you should process the unsubscribe within ~2 days. Transactional mail (receipts, resets) is exempt from one-click unsubscribe.

**When it applies.** Every app — every app that emails users sends reset/verify links; getting token+link safety right is foundational. The one-click unsubscribe / bulk-sender pieces become mandatory Once you have users (bulk/marketing volume to the major mailbox providers).

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html> · <https://www.rfc-editor.org/rfc/rfc8058.html> · <https://support.google.com/a/answer/81126>

### 🟢 Domain hijacking defense: registry lock, DNSSEC, and registrar account hardening

**What it is.** Protecting ownership of the domain itself at the registrar and registry layer so credential theft or social engineering can't transfer the domain or repoint NS/MX. Registrar (transfer) lock is the client-side flag (clientTransferProhibited); Registry Lock is a registry-level service that toggles serverTransferProhibited/serverUpdateProhibited/serverDeleteProhibited and requires out-of-band human verification for any change; DNSSEC cryptographically signs DNS answers.

**Why it matters / why AI-coded apps miss it.** If an attacker takes your domain they get everything downstream — they repoint MX to read your mail, repoint NS/A to serve a clone, and (because they control DNS) obtain valid TLS certs and bypass all your SPF/DKIM/DMARC. This is the highest-blast-radius email/domain risk and the one AI-coded projects never think about: the domain sits in a personal registrar account with a reused password, no MFA, no transfer lock, expiry on a lapsed card. DNSSEC protects DNS answer integrity but does NOT protect the management account — you need both.

**How to implement.** Registrar account: unique strong password + phishing-resistant MFA (FIDO2/passkey or TOTP, not SMS), dedicated role-based admin accounts, and a monitored/non-personal admin contact email (itself on a p=reject domain). Enable the registrar Transfer Lock (clientTransferProhibited) at minimum. For high-value domains add Registry Lock — a premium registry service (offered via registrars like CSC/MarkMonitor/Gandi corporate) that sets the server-side EPP statuses serverUpdateProhibited / serverDeleteProhibited / serverTransferProhibited and requires offline, out-of-band human verification (e.g. a passphrase phone call) before any transfer/NS/DNS change at the registry, so it holds even if registrar creds are stolen. Per ICANN's EPP status-code guidance these serverXxxProhibited codes are set by the registry, not the registrar, which is exactly why registry lock survives a compromised registrar login. Enable DNSSEC (publish the DS record at the parent) to stop DNS spoofing/cache poisoning, and add CAA records to constrain which CAs may issue certs for your domain. Operational: enable auto-renew AND track expiry independently, keep WHOIS/registrant contact current (with privacy), and document a change-control process. Lookalike/typosquatting monitoring: continuously watch Certificate Transparency (CT) logs (crt.sh feeds, certstream/Sunlight tile logs) and newly-registered-domain feeds for edit-distance-1/2 permutations, homoglyph/IDN-homograph variants, and your brand across other TLDs — CT monitoring catches the attacker's cert issuance (often within hours of registration, before the phishing site goes live); alert and act (takedown, defensive registration of the highest-risk permutations).

**When it applies.** Registrar MFA + transfer lock + DNSSEC + CAA + auto-renew: Every app. Registry Lock and active typosquatting/CT monitoring: At scale / Regulated-enterprise — the registry-lock fee and monitoring tooling are justified once a hijack would be materially damaging (real revenue, brand, regulated data).

**Sources:** <https://www.icann.org/resources/pages/epp-status-codes-list-2014-06-18-en> · <https://www.cscdbs.com/blog/registry-lock-and-dns-explained/> · <https://inventivehq.com/blog/detecting-phishing-domains-with-certificate-transparency>

## Software supply chain & DevSecOps

This layer addresses the build-and-delivery pipeline, not the running app: provenance of what you ship, integrity of what you pull in, and security gates inside CI. AI-coded ("vibe-coded") projects are especially exposed here because the generated code is almost always app logic — the model rarely scaffolds an SBOM, pins Actions by SHA, signs artifacts, or wires up SAST/secret-scanning. Meanwhile the 2025 attack wave (tj-actions/changed-files / CVE-2025-30066 compromising 23,000+ repos by overriding its existing tags to point at a CI-memory secret-dumping commit; pnpm CVE-2025-69263 storing HTTP/git tarball deps in the lockfile without integrity hashes; ongoing dependency-confusion and typosquat campaigns) targets exactly these gaps. The items below go beyond "run npm audit": SBOM generation/consumption with VEX, SLSA build provenance, keyless Sigstore signing + branch protection, full dependency/lockfile/Action-SHA/OIDC hardening, IaC + container image scanning with distroless bases, and dynamic-secret management with SAST/DAST/fuzzing in CI. Each is mapped to an honest tier so a solo builder knows what is table-stakes versus what only matters once you have users, scale, or compliance obligations. (Note: app-runtime hardening — CSP strict-dynamic/nonces, SRI+crossorigin, OAuth PKCE, DMARC p=reject, TLS 1.3, __Host- cookie prefixes — lives in a separate runtime/web cluster, not this build-pipeline layer.)

### 🔴 SBOM generation & consumption (CycloneDX / SPDX) with VEX

**What it is.** A Software Bill of Materials is a machine-readable inventory of every direct and transitive component (and its version + hash) in your build. CycloneDX (OWASP/ECMA) and SPDX (ISO/IEC 5962) are the two standard formats; VEX (Vulnerability Exploitability eXchange) is a companion document stating whether a given CVE actually affects you.

**Why it matters / why AI-coded apps miss it.** Without an SBOM you cannot answer 'are we affected by the next Log4Shell?' in minutes — you find out by manual grep across repos, days late. AI-coded apps accumulate large transitive trees the author never reviewed, so the unknown-unknown surface is large, and nobody generates the SBOM because the model never suggests it. Generating an SBOM but never re-scanning it against new advisories (consumption) is the common half-measure.

**How to implement.** Generate at BUILD time from source/lockfiles, not from a published image: use `syft <dir-or-image> -o cyclonedx-json` (also emits SPDX), OWASP `cdxgen` (reads lockfiles/manifests for accurate transitive trees — OpenSSF's 2025 tool comparison rates it most reliable for multi-language repos, and notes syft can miss deps other tools find), or `trivy sbom`. Pick CycloneDX for security/vuln workflows, SPDX for license/compliance. Emit a stable artifact per release and store it (attach to the OCI image or GitHub release). CONSUME it: continuously re-scan stored SBOMs with `grype sbom:./sbom.json` or `trivy sbom ./sbom.json` so newly disclosed CVEs map to already-shipped builds. Suppress non-exploitable findings with a CycloneDX/OpenVEX VEX document rather than ignoring scanner output. Automate via the sbomify or anchore/sbom-action GitHub Action.

**When it applies.** Once you have users/data (table-stakes for anything you ship or sell). Becomes legally mandatory for Regulated/enterprise: US EO 14028 / NTIA minimum elements for federal suppliers, and the EU Cyber Resilience Act, which requires manufacturers to maintain a machine-readable SBOM (SPDX/CycloneDX) in technical docs, available to authorities on request, by the Dec 2027 enforcement deadline.

**Sources:** <https://owasp.org/www-project-cyclonedx/> · <https://openssf.org/blog/2025/06/05/choosing-an-sbom-generation-tool/> · <https://github.com/anchore/syft>

### 🔴 SLSA build provenance & attestation (L1-L3)

**What it is.** SLSA (Supply-chain Levels for Software Artifacts) v1.0 defines a Build track, L1-L3, of increasing trustworthiness in an artifact's provenance — a signed statement of what was built, from which source commit, by which builder, with which inputs.

**Why it matters / why AI-coded apps miss it.** Provenance is what lets a consumer (or your own deploy gate) prove an artifact came from your CI running your reviewed source — not from a developer laptop or a hijacked build. AI-coded pipelines almost never emit provenance, so a tampered or substituted artifact is indistinguishable from a legitimate one. L3 specifically defends against the build-server insider/co-tenant and signing-key-exfiltration threats that plain signing does not.

**How to implement.** Targets (per slsa.dev v1.0): L1 = provenance exists showing how the artifact was built (can be used to prevent mistakes but is trivial to forge/bypass). L2 = build runs on dedicated hosted infra, not a workstation, and provenance is tied to that infra via a digital signature. L3 = hardened builds that prevent runs from influencing one another (run isolation) and prevent the provenance signing-key material from being accessible to user-defined build steps. Easiest path on GitHub: `actions/attest-build-provenance` (a wrapper over `actions/attest` as of v4) generates an in-toto SLSA provenance predicate and signs it keylessly via Sigstore/Fulcio, uploaded to the GitHub attestations API; the `slsa-framework/slsa-github-generator` reusable workflow can reach SLSA L3 on GitHub-hosted runners. Verify before deploy with `slsa-verifier verify-artifact`, `cosign verify-attestation --type slsaprovenance`, or `gh attestation verify`. Enforce a minimum source repo/commit and builder ID in the verification policy.

**When it applies.** At scale (any project distributing artifacts/images others consume; L3 expected for Regulated/enterprise and OSS publishers).

**Sources:** <https://slsa.dev/spec/v1.0/levels> · <https://slsa.dev/spec/v1.0/requirements> · <https://github.com/slsa-framework/slsa-github-generator>

### 🟢 Artifact & commit signing (Sigstore/cosign), signed git commits, branch protection

**What it is.** Cryptographically signing what you ship (container images, binaries, SBOMs, provenance) and what you commit, so tampering and impersonation are detectable. Sigstore/cosign enables keyless signing using short-lived OIDC-bound certs from Fulcio recorded in the Rekor transparency log.

**Why it matters / why AI-coded apps miss it.** An unsigned image in a registry can be silently replaced; an unsigned commit can be forged under any name (git has no built-in author authentication). AI-coded repos typically have neither, and 'pushed to main with admin bypass' is the norm. Keyless signing removes the 'where do I store the private key' problem that historically blocked signing — there is no long-lived key to leak.

**How to implement.** Keyless image signing in CI (needs `id-token: write`): `cosign sign $IMAGE@sha256:...` — authenticates via OIDC to get an ephemeral Fulcio cert, logs to Rekor, no stored key. Attach attestations: `cosign attest --type cyclonedx --predicate sbom.json $IMAGE` and `--type slsaprovenance`. VERIFY at admission/deploy (keyless requires identity flags): `cosign verify $IMAGE --certificate-identity-regexp '^https://github.com/ORG/' --certificate-oidc-issuer https://token.actions.githubusercontent.com`. Enforce in-cluster with a policy controller (Sigstore policy-controller / Kyverno / OPA Gatekeeper) so unsigned images are rejected. Git side: sign commits/tags (`git config commit.gpgsign true`, or SSH signing via `gpg.format ssh`, or `gitsign` for keyless). Branch protection (or a ruleset) on the default branch: require PR review, require status checks (the scans below) to pass, require signed commits, block force-push, and enable 'Do not allow bypassing the above settings' so admins/bypass roles are also bound.

**When it applies.** Signed commits + branch protection: Every app. Artifact signing + verification gate: Once you have users / publish images.

**Sources:** <https://docs.sigstore.dev/cosign/signing/signing_with_containers/> · <https://github.com/sigstore/cosign> · <https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches>

### 🟢 Dependency pinning, lockfile integrity, SHA-pinned Actions, least-privilege OIDC CI tokens

**What it is.** Making the set of code that runs in your build and your app fully deterministic and tamper-evident: pinned dependency versions with integrity hashes, third-party GitHub Actions pinned to a 40-char commit SHA, and CI jobs that hold no long-lived cloud credentials.

**Why it matters / why AI-coded apps miss it.** This is the single most exploited area in 2025. The tj-actions/changed-files attack (CVE-2025-30066) overrode the action's existing version tags to point at a malicious commit that dumped CI-runner memory secrets into workflow logs, hitting 23,000+ repos precisely because everyone pinned to mutable tags like @v4; the chain originated from a stolen SpotBugs maintainer PAT, then reviewdog, then tj-actions, then Coinbase. pnpm CVE-2025-69263 stored HTTP/git tarball deps in the lockfile without integrity hashes, letting a server serve different code per install despite a committed lockfile. Dependency-confusion and typosquatting exploit unpinned/floating ranges. AI-coded repos default to `@v4` tags, broad caret ranges, plain `npm install`, and full-scope `GITHUB_TOKEN` with a long-lived cloud key in Secrets — the worst case on every axis.

**How to implement.** Pin Actions by full SHA: `uses: actions/checkout@<40-hex-sha> # v4.2.2` (use Dependabot/Renovate to bump SHAs and StepSecurity Harden-Runner to detect runtime drift/egress). Lockfiles: commit them; in CI use `npm ci` / `pnpm install --frozen-lockfile` / `yarn install --immutable` — never plain `install`. Python: `pip install --require-hashes -r requirements.txt` generated by `pip-compile --generate-hashes` (or uv/poetry with hashes). Upgrade pnpm to >=10.26.0 (the fixed release for CVE-2025-69263) so HTTP/git tarball deps are recorded and verified with integrity hashes. Set top-level `permissions:` minimal: `permissions: { contents: read }` and elevate per-job only where needed (`id-token: write` for OIDC, `packages: write` to push). Replace stored cloud keys with OIDC: configure cloud trust to `token.actions.githubusercontent.com`, scope the trust to a specific repo/branch via the `sub` claim, and exchange the token at runtime (`aws-actions/configure-aws-credentials` with `role-to-assume`, no `aws-access-key-id`). Block dependency confusion with scoped registries / `.npmrc` lockdown.

**When it applies.** Every app (SHA-pinning, frozen lockfiles, minimal GITHUB_TOKEN are zero-cost and stop real 2025 attacks; OIDC the moment CI touches a cloud).

**Sources:** <https://unit42.paloaltonetworks.com/github-actions-supply-chain-attack/> · <https://docs.github.com/en/actions/security-for-github-actions/security-hardening-your-deployments/about-security-hardening-with-openid-connect> · <https://github.com/advisories/GHSA-7vhp-vf5g-r2fw>

### 🔵 IaC scanning, container image scanning & minimal/distroless base images

**What it is.** Static analysis of infrastructure-as-code (Terraform, Kubernetes, CloudFormation, Helm) for insecure misconfigurations, plus vulnerability scanning of container images against known CVEs, plus shrinking the image attack surface with minimal/distroless bases.

**Why it matters / why AI-coded apps miss it.** AI-generated Terraform and Dockerfiles routinely ship public S3 buckets, `0.0.0.0/0` security groups, privileged containers, missing encryption, and bloated `node:20`/`python` base images carrying hundreds of packages and dozens of OS CVEs the app never uses. These are deploy-time-fatal but invisible without scanning, and the model optimizes for 'it runs', not 'it is locked down'.

**How to implement.** IaC: run `checkov -d .` (deep Terraform cross-resource analysis) and/or `trivy config .` in CI as a required check. Note tfsec is deprecated — Aqua merged it into Trivy in Feb 2023, all tfsec check IDs (AVD-*) carry over to `trivy config` unchanged, so migrate. Be aware these miss git-sourced/remote Terraform modules by default, so scan vendored/expanded modules too. Image scanning: `trivy image --severity HIGH,CRITICAL --exit-code 1 $IMAGE` (fast, OS+lib coverage) and/or `grype $IMAGE` (can scan an SBOM directly); fail the build on fixable HIGH/CRITICAL. Base images: use multi-stage builds — build in a full image, copy only the artifact into `gcr.io/distroless/*`, Chainguard/Wolfi, Alpine, or `scratch`; distroless `nodejs20-debian12` ships ~20 packages instead of hundreds and removes shells/utilities (no `sh`/`bash` to exploit). Run as non-root, read-only rootfs, drop Linux capabilities. Rescan periodically since new CVEs land against already-built images.

**When it applies.** Once you have users/data (any cloud deploy or container in production; the IaC misconfig is usually the breach path).

**Sources:** <https://www.checkov.io/> · <https://github.com/aquasecurity/trivy> · <https://github.com/GoogleContainerTools/distroless>

### 🟢 Secrets management & rotation (Vault / cloud KMS) + SAST / DAST / fuzzing & secret scanning in CI

**What it is.** Two paired disciplines: (a) keeping secrets out of code/images and making them short-lived and rotatable via a vault/KMS; (b) automated security testing gates in CI — static analysis (SAST), running-app testing (DAST), fuzzing, and secret scanning of the repo and history.

**Why it matters / why AI-coded apps miss it.** Hard-coded API keys are the most common AI-coded-app finding — the model inlines `OPENAI_API_KEY='sk-...'` to make the example run, and it lands in git history and the image layer forever. Static long-lived credentials mean one leak = total compromise. And without SAST/DAST/secret-scan gates, injection/XSS/auth-bypass and leaked keys reach production unreviewed because no human read the generated code path.

**How to implement.** Secrets: never commit; load from a manager. Use HashiCorp Vault DYNAMIC secrets — DB/cloud creds minted on demand with a TTL/lease and auto-revoked on lease expiry — or cloud-native (AWS Secrets Manager with rotation, GCP Secret Manager, Azure Key Vault) with managed rotation schedules; encrypt with cloud KMS/HSM and prefer workload identity (IRSA / GKE Workload Identity / managed identity) over static keys. Rotate on a schedule AND immediately on any suspected exposure. CI gates: secret scanning with `gitleaks detect` / `trufflehog` in pre-commit AND CI (scan full history, not just the diff); enable GitHub push protection. SAST with `semgrep ci` (SQLi/XSS/secrets/authz patterns) or CodeQL on PRs, blocking on new HIGH findings. DAST with OWASP ZAP baseline scan against a deployed preview env. Fuzzing for parsers/protocol code via OSS-Fuzz (free for OSS) or `jazzer`/libFuzzer/`cargo fuzz` in CI. Run fast scans on every PR, deeper scans nightly.

**When it applies.** Secret scanning + no hard-coded secrets + SAST: Every app. Dynamic secrets / KMS rotation: Once you have users/data. DAST + fuzzing: At scale / Regulated.

## Security governance, monitoring & process

This is the advanced governance-and-process layer that AI-coded apps almost universally skip: they ship features but never threat-model the data flows, never centralize tamper-evident logs or wire alerts to them, have no incident-response runbook, no way for a researcher to report a bug, no privacy/consent/erasure plumbing, and no business-logic abuse testing (the one bug class scanners cannot find). The six items below are sequenced from "do this in design" (threat modeling) through "do this in production" (logging/alerting, IR, disclosure) to "do this continuously" (privacy rights, business-logic abuse testing). Each cites current 2023-2026 primary guidance: OWASP Top 10:2025 A09 (renamed "Security Logging and Alerting Failures" — Monitoring was deliberately replaced by Alerting), OWASP Top 10 for Business Logic Abuse (first release May 2025), OWASP WSTG and cheat sheets, NIST SP 800-61r3 (April 2025, supersedes r2, a CSF 2.0 Community Profile), RFC 9116, EDPB cookie-banner taskforce report (Jan 2023), the ICO's finalized storage-and-access (cookies) guidance, GDPR Arts. 15/17, and PortSwigger's single-packet race-condition research. All source URLs in this section were fetched and confirmed to resolve and support their claim as of June 2026; the one substantive correction made during fact-check was re-attributing the "manual art" quote to its true origin (the WSTG Introduction to Business Logic) rather than the Business Logic Abuse project, which actually uses a Turing-machine modeling approach.

### 🟢 Threat modeling with STRIDE (security) + LINDDUN (privacy), data-flow diagrams, and attack trees

**What it is.** A structured design-time analysis: draw a data-flow diagram (DFD) of your system with trust boundaries, then walk each element through STRIDE to enumerate security threats (and, because this section also covers privacy, LINDDUN for privacy threats), rank them by likelihood x impact, and assign mitigations. Driven by the four questions: What are we building? What can go wrong? What are we going to do about it? Did we do a good enough job?

**Why it matters / why AI-coded apps miss it.** AI codegen produces components in isolation and never reasons about the system's trust boundaries (browser->API, API->DB, service->third party), so whole threat classes (a tampered JWT role, a missing authZ check across a boundary, a repudiation gap, an over-collection privacy harm) are never even considered. Threat modeling is the only activity that finds 'what's missing' rather than 'what's wrong in code I already wrote' — exactly the blind spot of code-level scanners and AI review.

**How to implement.** Build the DFD with the five element types: external entities (actors), processes, data stores, data flows, and trust boundaries (boundaries are where attacks land). Use OWASP Threat Dragon (free, Apache-2.0, supports STRIDE/LINDDUN/CIA, auto-generates threats) or pytm (threat-model-as-code, diffable in git) or draw.io; the MS Threat Modeling Tool also bakes in STRIDE. Apply STRIDE per-element, each maps to a violated property: Spoofing->Authentication (stolen-token impersonation), Tampering->Integrity (unintended DB update), Repudiation->Non-repudiation (attacker edits logs), Information Disclosure->Confidentiality, Denial of Service->Availability (lock-out via failed-auth flooding), Elevation of Privilege->Authorization (tamper a JWT to change role). For data flows handling PII, also run LINDDUN (Linkability, Identifiability, Non-repudiation, Detectability, Disclosure, Unawareness, Non-compliance) so the privacy item later in this section has a threat model to sit on. Rank by likelihood x impact (avoid DREAD's subjectivity; prefer OWASP Risk Rating or CVSS). Do it in the design phase and re-run when the architecture changes; track each threat-with-mitigation as a tracked issue, not a one-off doc. Anchor the DFD on a C4-model system map (c4model.com).

**When it applies.** Every app (lightweight single-page DFD + STRIDE pass at design time — costs an hour and is the highest-leverage security activity for a small team). Becomes Once-you-have-users/data when handling auth, payments, or PII (full per-boundary STRIDE + LINDDUN, maintained as the system evolves, mitigations tracked to closure).

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Threat_Modeling_Cheat_Sheet.html> · <https://owasp.org/www-project-threat-dragon/> · <https://owasp.org/www-project-pytm/> · <https://c4model.com/>

### 🟢 Centralized, tamper-evident security logging with detection & alerting (OWASP A09:2025)

**What it is.** Emit structured security-relevant events to a centralized, integrity-protected store (SIEM), then alert on them in real/near-real time with defined thresholds and escalation. OWASP renamed the 2025 category from 'Security Logging and Monitoring Failures' to 'Security Logging and Alerting Failures' specifically to stress that logging without acting on it is the failure: 'great logging with no alerting provides minimal value in identifying security incidents.'

**Why it matters / why AI-coded apps miss it.** AI-coded apps log app errors to stdout/console and nothing else: no authentication events, no access-control failures, no audit trail, no alerting, logs stored only locally and never backed up. You can be breached for months with zero signal. A09 stays in the Top 10 precisely because passive, local, un-alerted logging is the norm — and the 2025 rename exists to call out that a dashboard nobody watches is not detection.

**How to implement.** Log the OWASP event set (see the Logging Vocabulary Cheat Sheet's AUTHN/AUTHZ/INPUT/SESSION/MALICIOUS taxonomy): authentication successes AND failures, authorization/access-control failures, input/output validation failures, session-management failures (cookie/JWT tampering), high-value transactions, privileged-function and admin use, and 'suspicious business-logic activity such as actions out of order / bypassing flow control.' Per event capture When/Where/Who/What (UTC ISO-8601 timestamp, app id+version, source IP, user/session id, event type, severity, outcome). NEVER log secrets: passwords, session IDs, access tokens, API keys, DB connection strings, full PAN/card data, or sensitive PII (CWE-532); neutralize log inputs to prevent log injection (CWE-117). Ship in a parseable format (JSON, CEF/syslog) to a centralized SIEM (ELK/Elastic, Loki/Grafana, Datadog, Splunk, or a cloud-native CloudWatch/Cloud Logging sink) over TLS, async, off the app host. Make it tamper-evident: append-only tables, write-once/immutable object storage (e.g. S3 Object Lock / WORM), and hash-chaining or signing of entries so modification/deletion is detectable; restrict and audit log read access. Then ALERT: set thresholds and escalation (e.g. burst of failed logins per account or across IPs, large data exports, spikes in slow/expensive queries), have alerts reviewed within a defined SLA, and add honeytokens as trip-wires. Tie into OWASP Coraza/ModSecurity CRS at the edge to generate attack telemetry.

**When it applies.** Every app needs centralized auth + access-control logging with at least one alert (e.g. credential-stuffing burst). Tamper-evident WORM/hash-chaining and tuned SIEM alerting is Once-you-have-users/data; full SOC-grade correlation, retention SLAs, and 24/7 on-call triage is At-scale / Regulated.

**Sources:** <https://owasp.org/Top10/2025/A09_2025-Security_Logging_and_Alerting_Failures/> · <https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Logging_Vocabulary_Cheat_Sheet.html>

### 🟢 Incident response plan & runbooks (NIST SP 800-61r3, CSF 2.0)

**What it is.** A written, exercised plan defining how you detect, triage, contain, eradicate, recover from, and learn from a security incident, with role-specific runbooks for the common scenarios (account takeover, leaked secret, data breach, ransomware). NIST SP 800-61r3 (April 2025, 'Incident Response Recommendations and Considerations for Cybersecurity Risk Management: A CSF 2.0 Community Profile') reframes the old four-phase lifecycle around the CSF 2.0 functions.

**Why it matters / why AI-coded apps miss it.** Solo/AI-built projects have no plan: when a key leaks or an account-takeover wave hits, they improvise — often destroying evidence, mis-rotating credentials, or missing a legal breach-notification clock. r3 (released April 3 2025, first revision since 2012, supersedes r2) exists because incidents are now more frequent and dynamic, and because 'Improve' must be continuous, not a single post-mortem.

**How to implement.** Map your plan to SP 800-61r3's CSF 2.0 functions. Preparation lives in Govern/Identify/Protect: assign roles and an incident commander, build an asset inventory, document legal/regulatory notification duties (e.g. GDPR Art. 33's 72-hour authority-notification clock), pre-stage tooling and out-of-band comms. Active response is Detect/Respond/Recover: detection feeds from your SIEM (item above); Respond covers triage/severity classification, containment, eradication, evidence/forensics preservation, and stakeholder comms; Recover covers restoration, post-incident review, and updating playbooks. 'Improve' runs continuously across the whole lifecycle, not just at the end. Write concrete runbooks: e.g. leaked-secret runbook = rotate the credential, revoke active sessions/tokens, scan git history (gitleaks/trufflehog), check access logs for use, notify affected parties. Keep a severity matrix (SEV1-3) with response-time targets and an escalation tree. Tabletop-exercise at least the top scenarios. Note A09:2025's prevention guidance explicitly tells adopters to 'adopt an incident response and recovery plan, such as NIST 800-61r2 or later' (use r3, the current edition).

**When it applies.** Every app should have a one-page IR plan + leaked-secret and account-takeover runbooks. Full r3-aligned plan with tabletop exercises and a breach-notification workflow is Once-you-have-users/data; formal IRT, on-call rotation, and IR retainer is At-scale / Regulated.

**Sources:** <https://csrc.nist.gov/pubs/sp/800/61/r3/final> · <https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-61r3.pdf> · <https://www.nist.gov/news-events/news/2025/04/nist-revises-sp-800-61-incident-response-recommendations-and-considerations>

### 🟢 Responsible-disclosure path: security.txt (RFC 9116), VDP, and a vuln-management SLA

**What it is.** A published, machine-readable way for outside researchers to report vulnerabilities (security.txt), a Vulnerability Disclosure Policy (VDP) that scopes testing and grants safe harbor, and an internal SLA that ties severity to acknowledge/triage/fix timelines. Bug bounty is the paid tier on top.

**Why it matters / why AI-coded apps miss it.** AI-coded sites give a finder no contact channel, so good-faith researchers either give up or go public (full disclosure) with no patch ready. No VDP also means a researcher who pokes your app has no legal safe harbor, chilling exactly the reports you want. And without an SLA, valid reports rot in an inbox.

**How to implement.** Publish security.txt at https://example.com/.well-known/security.txt over HTTPS (RFC 9116, which uses the RFC 8615 well-known path). Two fields are MANDATORY: Contact (mailto:/https URL/tel — MUST always be present) and Expires (RFC 3339/ISO-8601 timestamp; MUST be present and MUST NOT appear more than once; RECOMMENDED to be less than a year in the future to avoid staleness). Useful optional fields: Policy (link to your VDP), Encryption (PGP key), Acknowledgments (hall of fame), Preferred-Languages, Canonical, Hiring. It is RECOMMENDED to OpenPGP-clearsign the file and to include a Canonical field. Example: 'Contact: mailto:security@example.com' / 'Expires: 2026-12-31T18:37:07Z' / 'Policy: https://example.com/.well-known/vdp'. Write the VDP per ISO/IEC 29147 (vulnerability disclosure, ed.2 2018, reaffirmed 2024) and handle reports internally per ISO/IEC 30111 (handling); US federal teams can also reference NIST SP 800-216 (Recommendations for Federal Vulnerability Disclosure Guidelines, 2023), though it is government-scoped, so ISO 29147/30111 are the better general fit. State in-scope/out-of-scope assets, prohibited actions, and a safe-harbor clause (take real legal advice on wording). Prefer coordinated disclosure (private report, publish once a fix ships). Set an SLA keyed to CVSS severity, e.g. acknowledge <=2-3 business days, triage <=5 business days, then remediation targets like Critical <=7 days / High <=30 / Medium <=90 / Low <=180, with status updates throughout. When advisories publish, include affected versions, fixed versions, and a CVE. Only add a paid bug bounty (HackerOne/Bugcrowd/Intigriti) AFTER the internal triage/fix pipeline is mature, otherwise you drown in reports.

**When it applies.** security.txt + a basic VDP with safe harbor is Every app (it is nearly free). A severity-based remediation SLA is Once-you-have-users/data. Funded bug bounty is At-scale.

**Sources:** <https://www.rfc-editor.org/rfc/rfc9116.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Vulnerability_Disclosure_Cheat_Sheet.html> · <https://www.iso.org/standard/72311.html> · <https://csrc.nist.gov/pubs/sp/800/216/final>

### 🟢 Privacy & consent engineering: cookie consent, data minimization, and DSAR/erasure

**What it is.** The operational privacy plumbing: a compliant consent banner that blocks non-essential cookies/trackers until opt-in, minimized data collection, and working data-subject-access (GDPR Art. 15) and erasure (Art. 17) request handling end to end.

**Why it matters / why AI-coded apps miss it.** AI scaffolds drop in Google Analytics/Meta Pixel and a cosmetic 'Accept' banner that fires trackers before consent, uses pre-ticked boxes or no equally-easy reject, and has no deletion path — exactly the dark patterns the EDPB cookie-banner taskforce (Jan 2023) flagged as invalid consent. There is also usually no way to actually fulfil an erasure request because data is sprayed across DB + logs + analytics + backups.

**How to implement.** Consent: under the ePrivacy Directive + GDPR you need prior, opt-in, freely given consent for any non-essential cookie/tracker; load NOTHING non-essential before consent. Per the EDPB taskforce report and the ICO's finalized storage-and-access (cookies) guidance: a 'Reject all' option must be as easy and as prominent as 'Accept all' on the same/first layer (burying refusal in a 'More options' menu is non-compliant); no pre-ticked boxes, no consent via continued browsing, no deceptive colours/wording nudging acceptance; withdrawal must be as easy as giving consent (e.g. a persistent, visible icon/link). Note the EDPB also rejected relying on 'legitimate interests' as a basis for advertising cookies in lieu of consent. Implement with a CMP (and, if you run EEA/UK ads, Google Consent Mode v2 with an IAB TCF v2.2-certified CMP, noting TCF does not by itself create valid consent). Minimization: collect only fields you need, set retention limits, and keep a data map so you know every store that holds a given user's data. DSAR/erasure: build an authenticated self-serve flow or intake that, on request, exports (Art. 15) or deletes (Art. 17) the subject's data across ALL live systems plus analytics and documents backup handling; respond without undue delay and within one month (extendable by two further months for complex/numerous requests). Wire 'delete account' to actually cascade-delete or irreversibly anonymize, and propagate deletion to processors. Keep an audit log of DSARs.

**When it applies.** Consent banner done right + a manual erasure process is Every app that has EU/UK/California visitors or sets non-essential cookies. Self-serve DSAR/erasure tooling, data mapping, and processor propagation is Once-you-have-users/data; DPO, RoPA, and DPIAs are Regulated/enterprise.

**Sources:** <https://www.edpb.europa.eu/system/files/2023-01/edpb_20230118_report_cookie_banner_taskforce_en.pdf> · <https://gdpr-info.eu/art-15-gdpr/> · <https://gdpr-info.eu/art-17-gdpr/> · <https://ico.org.uk/for-organisations/direct-marketing-and-privacy-and-electronic-communications/guidance-on-the-use-of-storage-and-access-technologies/>

### 🟢 Business-logic abuse testing: workflow bypass, price/quantity tampering, race conditions

**What it is.** Manual, scenario-driven testing of how your application's rules can be subverted — abuse cases that are valid HTTP but violate intended business flow: skipping a workflow step, tampering price/quantity/IDs, replaying single-use tokens, or racing single-use limits. Codified in the OWASP WSTG Business Logic Testing chapter and the new OWASP Top 10 for Business Logic Abuse (first release May 2025).

**Why it matters / why AI-coded apps miss it.** This is the bug class scanners and AI review CANNOT find, because each request is individually 'valid'. The OWASP WSTG Introduction to Business Logic states it plainly: 'automation of business logic abuse cases is not possible and remains a manual art relying on the skills of the tester and their knowledge of the complete business process and its rules,' and that automated tools 'have no means of detecting if a user is able to circumvent the business process flow.' AI-generated CRUD checks each request in isolation but never enforces sequence, server-side pricing, idempotency, or concurrency — so coupon/refund double-spend, negative quantities, role-field mass assignment, and free upgrades are rampant.

**How to implement.** Test the OWASP Top 10 for Business Logic Abuse classes: BLA1 Action Limit Overrun (concurrent requests beat a single-use check via TOCTOU, e.g. redeem one coupon/gift card N times), BLA2 Concurrent Workflow Order Bypass (race the final step ahead of prerequisites), BLA3 Object State Manipulations (mass-assignment of protected fields like isAdmin/role/price), BLA5 Artifact Lifetime Exploitation (replay one-time tokens/sessions), BLA6 Missing Transition Validation (jump to a later workflow stage by calling its endpoint directly). Concretely: (1) Workflow bypass: try invoking step 3's endpoint without completing steps 1-2; reorder/replay requests (WSTG-BUSL-06, Testing for the Circumvention of Work Flows). (2) Price/quantity tampering: intercept with Burp and send negative or fractional quantities, a client-supplied price, swap product/price IDs, stack coupons — always re-derive price and totals server-side from trusted data, never trust client values. (3) Race conditions: identify single-use or rate-limited endpoints with impact and fire many requests in the race window; use Burp Repeater's 'Send group in parallel' or Turbo Intruder with the single-packet attack (engine=Engine.BURP2, concurrentConnections=1, HTTP/2 only — incompatible with HTTP/1) to land 20-30 requests in one TCP packet and neutralize network jitter. Defend by making state changes atomic with the datastore's concurrency features (SELECT ... FOR UPDATE, transactions, optimistic locking), unique constraints as defense-in-depth, idempotency keys on payment/mutation endpoints, server-side enforcement of the state machine and limits, and not splitting one logical operation across two data stores. Map property-level checks to OWASP API3:2023 (BOPLA, which merged Mass Assignment + Excessive Data Exposure) and resource limits to API4:2023 (Unrestricted Resource Consumption).

**When it applies.** Every app with money, quotas, multi-step flows, or roles must at minimum enforce server-side pricing/limits and test step-skipping. Race-condition/single-packet testing and idempotency keys are Once-you-have-users/data (payments, coupons, inventory); continuous abuse-case test suites and dedicated red-teaming are At-scale.
