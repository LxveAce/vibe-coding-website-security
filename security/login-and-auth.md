# Login & authentication — the mistakes vibe-coded apps ship

> Inspired by the "YOUR LOGIN ISN'T SECURE" breakdown — the auth flaws AI code generators produce by
> default. Each with what it is, why AI gets it wrong, the fix, when it applies, and cited sources.
> See also the broader [security/vulnerabilities.md](vulnerabilities.md) and [`../AGENTS.md`](../AGENTS.md).

AI code generators reliably produce auth that "works" in the happy path but leaks under adversarial pressure: tokens dropped into localStorage, role checks living in React route guards, "2FA" that is really just an emailed code, login endpoints with no throttling, and password fields that accept "Password1!". These are the highest-frequency, highest-impact mistakes in vibe-coded apps because the model optimizes for a working demo, not for an attacker. The fixes are well-documented by OWASP and NIST and are mostly about moving trust to the server, hardening token storage, and using real second factors plus breach checks. Most of this is table-stakes for any app that has real users; the heavier controls (passkeys, distributed rate limiting, local Pwned Passwords mirrors) scale up with sensitivity and traffic.

---

## 1. Auth tokens in localStorage/sessionStorage (XSS can steal them)

**What it is.** Storing a JWT or session token in localStorage or sessionStorage so the frontend can read it and attach it to API calls. Any JavaScript running on the origin can read these stores, so a single XSS bug hands an attacker every token.

**Why vibe-coded apps get it wrong.** It is the path of least resistance the model reaches for: localStorage.setItem('token', jwt) is one line and instantly works with fetch/axios interceptors, whereas httpOnly cookies require coordinated server Set-Cookie headers and CSRF handling. Vibe-coded apps also tend to ship XSS holes (dangerouslySetInnerHTML, unsanitized markdown), which turns the storage choice into a full account-takeover vector.

**Fix.** Store the session/refresh token in a cookie set by the server with HttpOnly; Secure; SameSite=Strict (or Lax if you need top-level cross-site navigation), so JavaScript cannot read it; the __Host- cookie prefix (e.g. __Host-SessionID) is recommended to lock the cookie to the host and Path=/. Keep access tokens short-lived (5-15 min for sensitive apps, up to ~30-60 min otherwise) and pair with refresh-token rotation: issue a new refresh token on every use, make old ones single-use, and add reuse detection (per RFC 9700 §4.14) that revokes the whole token family on replay. If you genuinely need a SPA to hold a token, use the Backend-for-Frontend (BFF) pattern so the browser only ever holds an opaque cookie and the BFF holds the real token. Regenerate the session identifier on login and on any privilege change, and never accept a session ID the server did not generate, to prevent session fixation. None of this removes the need to actually fix XSS (CSP, output encoding) and add CSRF protection (tokens or SameSite) since cookies are auto-sent.

**When it applies.** Every app — anything with a login needs correct token storage; XSS-to-account-takeover applies to the simplest site. Refresh-token rotation/reuse-detection and BFF matter more once you have real users or sensitive data.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html> · <https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies> · <https://datatracker.ietf.org/doc/html/rfc9700>

## 2. Authorization/role checks done on the client instead of the server

**What it is.** Enforcing 'who can do what' in the frontend — hiding an Admin button, gating a route with a client-side guard, or trusting a role claim the browser sends — while the API itself happily serves the request to anyone who calls it directly. This is OWASP's #1 risk, Broken Access Control.

**Why vibe-coded apps get it wrong.** The model builds UI and API as if the UI is the only caller, so it 'protects' a feature by not rendering it. That is UX, not security: anyone can open DevTools, call the endpoint with curl/Postman, or tamper with an ID in the URL (IDOR). OWASP's Top 10:2025 data found that 100% of tested applications had some form of broken access control, and it is the single most common class generated by AI scaffolding.

**Fix.** Enforce every authorization decision in trusted server-side code (controller, middleware, gateway, or serverless function) on every request, regardless of how it was initiated. Adopt deny-by-default: a request is rejected unless an explicit rule permits it. Centralize the check (one reusable middleware/policy layer, e.g. Express middleware, Spring Security, ASP.NET policies, Django permissions, or a policy engine like OPA/Cedar) rather than scattering ad hoc if-statements. Enforce record ownership / object-level authorization (don't just check 'is logged in', check 'owns this resource') to kill IDOR — OWASP API Top 10 lists Broken Object Level Authorization (BOLA, API1:2023) as the top API risk. Never trust a role/claim the client supplies as the source of truth; derive it server-side from the session. Treat client-side guards purely as UX. Log access-control failures and alert on repeated denials.

**When it applies.** Every app — this is the most damaging and most common vibe-coded mistake; even a tiny multi-user app must enforce authz server-side. Centralized policy engines and audit logging skew toward scale/regulated.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html> · <https://owasp.org/Top10/2025/A01_2025-Broken_Access_Control/> · <https://owasp.org/API-Security/editions/2023/en/0xa1-broken-object-level-authorization/>

## 3. Treating email links/codes as strong MFA

**What it is.** Calling an emailed magic link or one-time code (or an SMS code) '2FA' and treating it as a strong second factor. Email and SMS inherit the security of an account/phone number you don't control and are defeated by SIM-swap, SS7 interception, number-porting, and email-account takeover.

**Why vibe-coded apps get it wrong.** Email/SMS codes are easy to bolt on with an existing transactional-email/SMS provider and feel like 'real 2FA' to both the developer and the model, so AI scaffolds them by default. The subtlety — that email is often not an independent factor (the email account frequently shares the same password and lacks its own MFA, and neither email nor SMS is phishing-resistant) — is exactly the kind of nuance a happy-path generator skips.

**Fix.** Prefer phishing-resistant MFA: WebAuthn/FIDO2 passkeys (security keys or platform authenticators). Passkeys are bound to your domain (Relying Party ID) so a credential for example.com cannot be triggered by a phishing page on evil.com — the signature changes with the origin; use a library like SimpleWebAuthn to handle the registration/authentication ceremonies rather than hand-rolling the crypto. The next tier down is TOTP authenticator apps (RFC 6238; e.g. otplib/speakeasy server-side, Google Authenticator/1Password/Authy client-side) — not phishing-resistant but free of SIM-swap/SS7 risk. Note NIST SP 800-63B-4 (final): SMS/PSTN out-of-band OTP is a 'restricted' authenticator (allowed only with extra controls and risk monitoring), and email SHALL NOT be used for out-of-band authentication at all — so don't count an emailed code as a real factor. OWASP likewise says SMS/phone should not protect apps holding PII, and that email 'relies entirely on the security of the email account, which often lacks MFA.' Per OWASP, any MFA beats none, but the factor choice materially changes the security you actually get.

**When it applies.** Once you have users/data — every app benefits from MFA (Microsoft data: MFA blocks ~99.9% of automated account-compromise), but passkeys/TOTP become important as accounts hold anything worth stealing. Phishing-resistant passkeys are effectively mandatory for admin/privileged and regulated/enterprise accounts.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Multifactor_Authentication_Cheat_Sheet.html> · <https://pages.nist.gov/800-63-4/sp800-63b.html> · <https://developer.mozilla.org/en-US/docs/Web/API/Web_Authentication_API>

## 4. No rate limiting / lockout / bot protection on auth endpoints

**What it is.** Login, signup, and password-reset endpoints that accept unlimited attempts with no throttling, lockout, or bot mitigation. This invites credential stuffing (replaying leaked username/password pairs), brute force, account enumeration, and reset-email flooding.

**Why vibe-coded apps get it wrong.** Rate limiting is invisible in a demo — the app works fine for one well-behaved user, so the model never adds it. There is no failing test to prompt it, and proper throttling requires shared state (Redis, a gateway) the scaffold doesn't set up. Reset endpoints are especially neglected because they 'just send an email'.

**Fix.** Tie the failed-attempt counter to the account, not just the source IP (attackers rotate proxies, so per-IP alone is insufficient), and define a lockout threshold, observation window, and lockout duration. Use exponential/progressive lockout (OWASP example: start at ~1s and double per failure, or lock for ~15-20 min after N failures). Put a rate limiter in front of every auth endpoint — login, signup, reset request, reset confirm, MFA verify — using express-rate-limit + a shared store (Redis), an API gateway, or a WAF/edge service (Cloudflare Turnstile/Rate Limiting, AWS WAF). Add bot defenses: CAPTCHA/Turnstile after a few failures, plus credential-stuffing-specific controls (device fingerprinting, breached-password screening) from the OWASP Credential Stuffing cheat sheet. On password reset specifically, rate-limit per account and return a consistent 'if an account exists, we sent an email' response with uniform response timing to prevent user enumeration. Log auth failures and alert on spikes. MFA and breached-password checks are complementary defenses since stuffing uses valid leaked passwords.

**When it applies.** Every app for basic login throttling and reset rate-limiting (trivial to add, blocks the most common automated attacks). Distributed/per-account limiting, WAF/bot management, and stuffing-specific tooling scale up with traffic and value.

**Sources:** <https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Credential_Stuffing_Prevention_Cheat_Sheet.html> · <https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html>

## 5. Allowing weak or known-breached passwords

**What it is.** Accepting weak passwords with no real strength policy and, critically, no check against breached-password corpuses. A password can satisfy 'one uppercase + one number + 8 chars' and still be in every leaked-credential list (e.g. Password1!), making it useless against credential stuffing.

**Why vibe-coded apps get it wrong.** Generators reproduce the legacy myth of composition rules (forced symbols, periodic rotation) because that pattern dominates older training data, and they almost never wire up a breach check because it requires an external API call and an extra step. The result looks 'secure' but blocks none of the passwords attackers actually try.

**Fix.** Follow NIST SP 800-63B-4: require a minimum length (NIST: 8+ generally, and 15+ when a password is used as the only factor; OWASP frames <8 with MFA or <15 without MFA as weak), allow at least 64 characters and all Unicode/whitespace to support passphrases, and drop composition rules and mandatory periodic rotation. Screen every new/changed password against a breached-password blocklist — use Have I Been Pwned's Pwned Passwords API, which uses k-anonymity (your client SHA-1-hashes the password, sends only the first 5 hex chars, gets back the matching suffixes, and compares locally so the full password/hash never leaves the client); the service is free and offers a downloadable corpus you can mirror locally for privacy/latency at scale. Add a client- and server-side strength estimator like zxcvbn-ts to nudge users. Store passwords with a memory-hard hash — Argon2id first choice (OWASP baseline: 19 MiB memory, 2 iterations, 1 parallelism), or scrypt/bcrypt where Argon2id is unavailable — per the OWASP Password Storage cheat sheet; never plain or fast-hashed (SHA-256 is unsuitable for passwords). Reject the breached/common passwords with a clear message instead of silently allowing them.

**When it applies.** Every app — a length policy and a Pwned Passwords lookup are cheap and stop the exact passwords stuffing attacks rely on. Self-hosting the Pwned Passwords corpus and stricter AAL2/AAL3 alignment lean toward scale/regulated.
